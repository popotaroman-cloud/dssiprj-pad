# Functional Requirements — ระบบฟอร์มเสนอโครงการ

**เอกสาร:** FRD-PMS-001  
**เวอร์ชัน:** 1.2  
**วันที่:** 2026-06-09  
**สถานะ:** Draft

---

## 1. ภาพรวมระบบ

ระบบรับเสนอโครงการออนไลน์ (Project Proposal Form System) เป็นส่วนหนึ่งของ Project Management System ทำหน้าที่รองรับการกรอกข้อมูลโครงการ การส่งคำขออนุมัติ และการติดตามสถานะในระดับ Pre-project โดยฟอร์มแบ่งออกเป็น 7 ขั้นตอน

---

## 2. ขอบเขต (Scope)

ครอบคลุมตั้งแต่การสร้างร่างโครงการ (Draft) จนถึงการส่งเข้าสู่กระบวนการอนุมัติ (Submitted) ไม่ครอบคลุมขั้นตอนการอนุมัติ การวางแผน หรือการดำเนินโครงการ

---

## 3. ผู้ใช้งาน (Actors)

| Actor | บทบาท |
|---|---|
| Proposer | ผู้กรอกและส่งฟอร์มเสนอโครงการ |
| Admin | จัดการข้อมูลอ้างอิง (Master Data) ของระบบ เช่น รายการ KPI |
| System | ตรวจสอบข้อมูล, สร้างรหัสอ้างอิง, ส่งแจ้งเตือน |
| Approver | รับแจ้งเตือนและดำเนินการอนุมัติ (อยู่นอกขอบเขต FRD นี้) |

## 5. ความต้องการเชิงฟังก์ชัน

### FR-01 — ข้อมูลทั่วไปโครงการ (Step 1)

**FR-01-01** ระบบต้องให้ Proposer กรอกชื่อโครงการ ความยาวไม่เกิน 120 ตัวอักษร และแสดงตัวนับอักขระแบบ real-time

**FR-01-02** ระบบต้องให้เลือกแผนก/หน่วยงาน จากรายการ dropdown ที่ดึงข้อมูลจากฐานข้อมูลองค์กร

**FR-01-03** ระบบต้องให้เลือกประเภทโครงการ 1 ประเภท ได้แก่ ระบบสารสนเทศ (IT), ก่อสร้าง/ปรับปรุง, วิจัยและพัฒนา, การตลาด, ฝึกอบรม, อื่น ๆ

**FR-01-04** ระบบต้องให้เลือกลำดับความสำคัญ 1 ระดับ ได้แก่ ต่ำ, ปานกลาง, สูง, วิกฤต โดยแต่ละระดับแสดง visual indicator ที่ต่างกัน

**FR-01-05** ระบบต้องให้กรอกชื่อผู้รับผิดชอบโครงการ (บังคับ) และชื่อผู้สนับสนุนโครงการ Sponsor (ไม่บังคับ)

**FR-01-06** ระบบต้องให้เลือก Tag ที่เกี่ยวข้องได้หลายรายการ (multi-select) จากรายการที่กำหนดไว้ล่วงหน้า

---

### FR-02 — วัตถุประสงค์และขอบเขต (Step 2)

**FR-02-01** ระบบต้องให้กรอกวัตถุประสงค์โครงการ (textarea) ความยาวไม่เกิน 500 ตัวอักษร พร้อมตัวนับอักขระ

**FR-02-02** ระบบต้องให้กรอกขอบเขตงาน (Scope) ความยาวไม่เกิน 500 ตัวอักษร โดยระบุทั้งสิ่งที่ครอบคลุมและไม่ครอบคลุม

**FR-02-03** ระบบต้องให้กรอกผลลัพธ์ที่คาดหวัง (Expected Outcome) เป็น free text

**FR-02-04** ระบบต้องให้กรอกกลุ่ม Stakeholder ที่ได้รับผลกระทบ (บังคับ) และผู้ใช้งานหลัก End Users (ไม่บังคับ)

---

### FR-03 — การเชื่อมโยงนโยบาย / แผนแม่บท (Step 3)

**FR-03-01** ระบบต้องให้เลือกหรือพิมพ์ชื่อแผนยุทธศาสตร์ที่เชื่อมโยง เช่น แผนพัฒนาองค์กร 5 ปี หรือแผน Digital Transformation โดยรองรับ typeahead search จากรายการที่กำหนดในระบบ

**FR-03-02** ระบบต้องให้ระบุนโยบายองค์กรหรือนโยบายภาครัฐที่โครงการสอดคล้อง เช่น ยุทธศาสตร์ชาติ 20 ปี, นโยบาย ESG, มาตรฐาน ISO (multi-select หรือ free text)

**FR-03-03** ระบบต้องให้เลือกระดับความสอดคล้อง 1 ระดับ ได้แก่ สูง (โครงการตรงตามแผนโดยตรง), ปานกลาง (สนับสนุนบางส่วน), สนับสนุน (เป็นองค์ประกอบเสริม)

**FR-03-04** ระบบต้องให้กรอกคำอธิบายความสอดคล้อง (textarea) อธิบายว่าโครงการช่วยขับเคลื่อนแผนหรือนโยบายนั้นอย่างไร

**FR-03-05** ระบบต้องแสดงคำเตือนหากไม่ได้ระบุการเชื่อมโยงใด ๆ แต่ไม่บังคับให้กรอก เพื่อให้ Proposer ตัดสินใจเอง

---

### FR-04 — ตัวชี้วัด KPI (Step 4)

**FR-04-01** ระบบต้องให้เพิ่ม KPI ได้ไม่จำกัดจำนวน โดยฟิลด์ชื่อตัวชี้วัดต้องรองรับการเลือกจาก dropdown KPI Master (typeahead search) หรือพิมพ์ชื่อใหม่แบบ free text แต่ละรายการประกอบด้วยข้อมูล ดังนี้

| ฟิลด์ | ประเภท | บังคับ |
|---|---|---|
| ชื่อตัวชี้วัด | Text (150 ตัวอักษร) | ใช่ |
| ค่าปัจจุบัน (Baseline) | Number + หน่วย | ใช่ |
| เป้าหมาย (Target) | Number + หน่วย | ใช่ |
| วันที่บรรลุเป้าหมาย | Date | ใช่ |
| วิธีวัดผล | Dropdown: Survey / Report / ระบบ IT / การสังเกต / อื่น ๆ | ใช่ |
| ความถี่รายงาน | Dropdown: รายสัปดาห์ / รายเดือน / รายไตรมาส / รายปี | ใช่ |

**FR-04-02** ระบบต้องบังคับให้มี KPI อย่างน้อย 1 รายการก่อนดำเนินการขั้นตอนถัดไป

**FR-04-03** ระบบต้องให้ลบ KPI แต่ละรายการได้ โดยแสดง confirmation dialog ก่อนลบ

**FR-04-04** ระบบต้องแสดงคำแนะนำ inline ว่าควรมี KPI ครอบคลุมทั้ง Output (สิ่งที่โครงการส่งมอบ) และ Outcome (ผลกระทบต่อองค์กร)

---

### FR-05 — แผนและงบประมาณ (Step 5)

**FR-05-01** ระบบต้องให้กรอกวันที่เสนอ (auto-fill วันปัจจุบัน แก้ไขได้) วันเริ่มต้น และวันสิ้นสุดโครงการ (คาดการณ์) โดยระบบต้องตรวจสอบว่าวันสิ้นสุดต้องไม่น้อยกว่าวันเริ่มต้น

**FR-05-02** ระบบต้องคำนวณระยะเวลาโครงการ (จำนวนเดือน) แบบ auto-calculate เมื่อกรอกวันเริ่มต้นและวันสิ้นสุด

**FR-05-03** ระบบต้องให้กรอกงบประมาณที่ขอ (บาท) และงบสำรอง (บาท) เป็น number input และคำนวณงบรวมอัตโนมัติ

**FR-05-04** ระบบต้องให้เลือกหมวดงบประมาณหลักและแหล่งงบประมาณ จาก dropdown

**FR-05-05** ระบบต้องให้ระบุจำนวนทีมงาน และความต้องการบุคลากรภายนอก

**FR-05-06** ระบบต้องแสดงคำเตือนหากงบประมาณที่กรอกเกินเกณฑ์ที่กำหนด (configurable threshold) โดยแจ้งว่าจะต้องผ่านการอนุมัติเพิ่มเติม

---

### FR-06 — การประเมินความเสี่ยง (Step 6)

**FR-06-01** ระบบต้องให้ประเมินระดับความเสี่ยง 5 ด้าน ได้แก่ เทคโนโลยี, งบประมาณ, ระยะเวลา, บุคลากร, กฎหมาย/ข้อบังคับ โดยแต่ละด้านเลือกได้จาก ไม่มี / ต่ำ / ปานกลาง / สูง / วิกฤต

**FR-06-02** ระบบต้องคำนวณและแสดง Risk Score โดยรวม (Overall Risk Level) จากการประเมินทั้ง 5 ด้าน

**FR-06-03** ระบบต้องให้กรอกมาตรการป้องกันและแก้ไขความเสี่ยง (Mitigation Plan) เป็น free text

**FR-06-04** ระบบต้องให้กรอกเงื่อนไขที่อาจทำให้โครงการล้มเหลว (Critical Failure Factors) เป็น free text

**FR-06-05** หาก Overall Risk Level อยู่ในระดับ สูง หรือ วิกฤต ระบบต้องแสดงคำเตือนและบังคับให้กรอก Mitigation Plan ก่อนดำเนินการต่อ

---

### FR-07 — เอกสารแนบและยืนยันการส่ง (Step 7)

**FR-07-01** ระบบต้องรองรับการแนบไฟล์ประเภท PDF, DOCX, XLSX, PNG, JPG ขนาดไม่เกิน 10 MB ต่อไฟล์ และรวมทุกไฟล์ไม่เกิน 50 MB

**FR-07-02** ระบบต้องแสดงรายการไฟล์ที่แนบแล้ว พร้อมปุ่มลบแต่ละไฟล์ได้

**FR-07-03** ระบบต้องบังคับให้ Proposer ติ๊ก checkbox ยืนยัน 2 ข้อก่อนส่ง ได้แก่ รับรองความถูกต้องของข้อมูล และรับทราบนโยบายการเสนอโครงการ

**FR-07-04** เมื่อ Proposer กดส่ง ระบบต้องตรวจสอบ validation ทุก field บังคับในทุก step ก่อนส่ง หากไม่ผ่านให้ redirect กลับไป step ที่มีปัญหาและแสดง error message

**FR-07-05** เมื่อส่งสำเร็จ ระบบต้องสร้างรหัสอ้างอิงโครงการ (Project Reference ID) รูปแบบ `PRJ-YYYY-NNNN` โดยอัตโนมัติ

**FR-07-06** ระบบต้องส่ง email แจ้งเตือนไปยัง Proposer และ Approver ที่เกี่ยวข้องทันทีหลังส่งสำเร็จ

---

### FR-08 — การบันทึกร่างและการกู้คืน (Auto-save)

**FR-08-01** ระบบต้องบันทึกร่างอัตโนมัติทุก 30 วินาที เมื่อมีการเปลี่ยนแปลงข้อมูล

**FR-08-02** ระบบต้องมีปุ่ม "บันทึกร่าง" ใน footer ของทุก step ให้ Proposer กดบันทึกได้ตลอดเวลา

**FR-08-03** ระบบต้องแสดงสถานะการบันทึก เช่น "บันทึกเมื่อ 10:23 น." แบบ real-time

**FR-08-04** ร่างที่บันทึกต้องคงอยู่อย่างน้อย 90 วัน หากยังไม่ถูกส่งหรือลบ

**FR-08-05** Proposer ต้องสามารถกลับมาแก้ไขร่างต่อจากที่ค้างไว้ได้ และระบบต้องนำผู้ใช้ไปยัง step สุดท้ายที่กรอกค้างไว้

---

### FR-09 — การนำทางระหว่าง Step

**FR-09-01** ระบบต้องแสดง step indicator ที่บอกสถานะของแต่ละ step ได้แก่ ยังไม่กรอก, กำลังกรอก, กรอกครบแล้ว, มีข้อผิดพลาด

**FR-09-02** Proposer ต้องสามารถย้อนกลับ step ก่อนหน้าได้ตลอดเวลาโดยไม่สูญเสียข้อมูลที่กรอกไปแล้ว

**FR-09-03** การข้ามไปยัง step ถัดไปต้องผ่าน validation ของ step ปัจจุบันก่อน

**FR-09-04** ระบบต้องแสดง progress bar แสดงเปอร์เซ็นต์ความสมบูรณ์ของฟอร์มโดยรวม

---

### FR-10 — การจัดการข้อมูลโครงการ (Data Management)

#### UC-07 — ค้นหาและกรองโครงการ

**FR-10-01** ระบบต้องให้ Proposer ค้นหาโครงการด้วยคีย์เวิร์ด (ชื่อโครงการ หรือ Reference ID) โดยแสดงผลลัพธ์แบบ real-time ขณะพิมพ์ (debounce ≥ 300 ms)

**FR-10-02** ระบบต้องรองรับการกรองโครงการแบบ multi-condition ได้แก่

| เงื่อนไขกรอง | ประเภท | หมายเหตุ |
|---|---|---|
| แผนก/หน่วยงาน | Dropdown (multi-select) | จากฐานข้อมูลองค์กร |
| ประเภทโครงการ | Checkbox | เลือกได้หลายประเภท |
| สถานะโครงการ | Checkbox | Draft / Proposed / Under Review / Approved / Rejected / Returned |
| ช่วงวันที่เสนอ | Date Range Picker | วันเริ่มต้น – วันสิ้นสุด |
| ลำดับความสำคัญ | Dropdown | ต่ำ / ปานกลาง / สูง / วิกฤต |

**FR-10-03** ระบบต้องให้เรียงลำดับผลลัพธ์ (Sort) ตามคอลัมน์ที่คลิกได้ ได้แก่ วันที่เสนอ, ชื่อโครงการ, สถานะ, งบประมาณ (ascending/descending)

**FR-10-04** ระบบต้องแสดงผลการค้นหาพร้อม Pagination โดยค่าเริ่มต้น 20 รายการต่อหน้า และให้ผู้ใช้เลือกได้ (20 / 50 / 100)

---

#### UC-08 — ดูรายการโครงการทั้งหมด

**FR-10-05** ระบบต้องแสดงรายการโครงการของ Proposer ในรูปแบบตาราง (List View) โดยแต่ละแถวแสดงข้อมูล ได้แก่ Reference ID, ชื่อโครงการ, ประเภท, สถานะ (พร้อม color badge), งบประมาณ, วันที่เสนอ และผู้รับผิดชอบ

**FR-10-06** ระบบต้องรองรับการดูข้อมูลสรุปโครงการแบบ Quick Preview (hover หรือ expand inline) โดยไม่ต้องเปิดหน้าใหม่

**FR-10-07** Proposer ต้องเห็นเฉพาะโครงการที่ตนเองเป็น Proposer หรือผู้รับผิดชอบ เว้นแต่ได้รับสิทธิ์ Admin

---

#### UC-09 — ลบร่างโครงการ

**FR-10-08** ระบบต้องอนุญาตให้ลบได้เฉพาะโครงการที่มีสถานะ **Draft** เท่านั้น โครงการที่ส่งแล้ว (Proposed ขึ้นไป) ต้องไม่สามารถลบได้

**FR-10-09** ก่อนลบ ระบบต้องแสดง confirmation dialog ที่แจ้งว่า "การลบไม่สามารถกู้คืนได้" และบังคับให้พิมพ์ชื่อโครงการยืนยันก่อนกดยืนยัน

**FR-10-10** เมื่อลบสำเร็จ ระบบต้องบันทึก Audit Log และนำ Proposer กลับสู่หน้ารายการโครงการพร้อมแสดง toast notification ยืนยัน

---

#### UC-10 — คัดลอกโครงการ (Duplicate)

**FR-10-11** ระบบต้องให้ Proposer คัดลอกโครงการที่มีอยู่ทุกสถานะเป็น Draft ใหม่ โดยคัดลอกข้อมูลทุก field ยกเว้น Reference ID, วันที่เสนอ, สถานะ และไฟล์แนบ

**FR-10-12** ชื่อโครงการที่คัดลอกต้องเพิ่มคำว่า `(สำเนา)` ต่อท้ายโดยอัตโนมัติ และ Proposer แก้ไขชื่อใหม่ได้ทันทีใน Step 1

**FR-10-13** ระบบต้องบันทึก Audit Log ระบุที่มาของ Draft ใหม่ว่าคัดลอกมาจาก Reference ID ใด

---

#### UC-11 — Export ข้อมูลโครงการ

**FR-10-14** ระบบต้องรองรับ Export โครงการเป็น 2 รูปแบบ

| รูปแบบ | เนื้อหา | หมายเหตุ |
|---|---|---|
| PDF | ฟอร์มครบทุก Step ใน layout พิมพ์ได้ รวม KPI และ Risk Table | รองรับ Unicode / ภาษาไทย |
| Excel (.xlsx) | ข้อมูลแบบ structured ทุก field แยก sheet ตาม Step | รองรับ Filter ใน Excel |

**FR-10-15** Proposer ต้องสามารถ Export ได้เฉพาะโครงการที่ตนเองมีสิทธิ์เข้าถึง และระบบต้องบันทึก Audit Log ทุกครั้งที่มีการ Export

**FR-10-16** ไฟล์ Export ต้องสร้างเสร็จภายใน 10 วินาที และระบบต้องแสดง loading indicator ระหว่างรอ

#### UC-12 — ดูประวัติการแก้ไข (Audit Log)

**FR-10-17** ระบบต้องบันทึก Audit Log ทุกการเปลี่ยนแปลงข้อมูล โดยแต่ละรายการประกอบด้วย

| ฟิลด์ | รายละเอียด |
|---|---|
| Timestamp | วันที่และเวลา (ระดับวินาที) |
| ผู้ดำเนินการ | ชื่อ-นามสกุล + รหัสพนักงาน |
| การกระทำ | Create / Edit / Delete / Submit / Export / Duplicate |
| Field ที่เปลี่ยน | ชื่อ field |
| ค่าก่อนเปลี่ยน | Old Value |
| ค่าหลังเปลี่ยน | New Value |

**FR-10-18** ระบบต้องแสดง Audit Log ของโครงการในรูปแบบ Timeline เรียงตาม timestamp ล่าสุดก่อน

**FR-10-19** Audit Log ต้องเก็บรักษาตลอดอายุโครงการ และต้องไม่สามารถแก้ไขหรือลบได้โดยผู้ใช้ทั่วไป

### FR-11 — การจัดการข้อมูลหลัก KPI (KPI Master Data Management)

> ฟังก์ชันนี้ใช้งานโดย **Admin** เพื่อบริหารรายการ KPI ที่ปรากฏใน dropdown ของฟอร์มเสนอโครงการ (Step 4)

#### UC-13 — เพิ่มรายการ KPI Master

**FR-11-01** ระบบต้องให้ Admin เพิ่มรายการ KPI ใหม่เข้าสู่ Master Data โดยกรอกข้อมูลดังนี้

| ฟิลด์ | ประเภท | บังคับ | เงื่อนไข |
|---|---|---|---|
| ชื่อตัวชี้วัด | Text (150 ตัวอักษร) | ใช่ | ต้องไม่ซ้ำในระบบ (case-insensitive) |
| คำอธิบาย | Textarea (300 ตัวอักษร) | ไม่ | อธิบายความหมายและวิธีวัด |
| หน่วยวัดเริ่มต้น | Text (50 ตัวอักษร) | ไม่ | เช่น %, บาท, ครั้ง, คน |
| ประเภท KPI | Dropdown | ใช่ | Output / Outcome / Process |
| สถานะ | Toggle | ใช่ | Active / Inactive |

**FR-11-02** ระบบต้องตรวจสอบชื่อตัวชี้วัดซ้ำก่อนบันทึก หากซ้ำต้องแสดง error message และไม่บันทึก

**FR-11-03** เมื่อเพิ่มสำเร็จ รายการใหม่ต้องปรากฏใน dropdown ของฟอร์ม Step 4 ทันที (ไม่ต้อง reload หน้า)

---

#### UC-14 — แก้ไขรายการ KPI Master

**FR-11-04** ระบบต้องให้ Admin แก้ไขข้อมูลทุกฟิลด์ของ KPI Master ที่มีอยู่ได้ ยกเว้น ID ภายใน

**FR-11-05** การแก้ไขชื่อ KPI ต้องแสดงคำเตือนหากมีโครงการที่กำลัง Draft อยู่ใช้ KPI นี้ เพื่อให้ Admin รับทราบผลกระทบ แต่ไม่บล็อกการแก้ไข

**FR-11-06** โครงการที่ **ส่งแล้ว (Proposed ขึ้นไป)** ต้องไม่ได้รับผลกระทบจากการแก้ไข KPI Master — ระบบต้องเก็บชื่อ KPI ณ เวลาที่ส่งฟอร์มไว้แยกต่างหาก (snapshot)

**FR-11-07** ระบบต้องบันทึก Audit Log ทุกครั้งที่มีการแก้ไข KPI Master พร้อมระบุ Admin ผู้แก้ไข, field ที่เปลี่ยน, ค่าก่อน/หลัง และ timestamp

---

#### UC-15 — ลบรายการ KPI Master

**FR-11-08** ระบบต้องตรวจสอบก่อนลบว่า KPI นั้นถูกใช้งานในโครงการใดบ้าง

| กรณี | การกระทำของระบบ |
|---|---|
| ไม่มีโครงการใดใช้ | ลบได้ทันทีหลัง confirmation |
| มีโครงการสถานะ Draft ใช้อยู่ | แสดงคำเตือนพร้อมรายชื่อโครงการ บังคับให้ Admin ยืนยันก่อนลบ |
| มีโครงการสถานะ Proposed ขึ้นไปใช้อยู่ | **ไม่อนุญาตให้ลบ** แสดง error พร้อมรายชื่อโครงการที่ใช้งาน |

**FR-11-09** แทนการลบถาวร ระบบต้องรองรับการ **ปิดใช้งาน (Inactive)** KPI เพื่อซ่อนออกจาก dropdown โดยไม่กระทบข้อมูลเดิม

**FR-11-10** เมื่อลบหรือปิดใช้งานสำเร็จ ระบบต้องบันทึก Audit Log และ KPI ต้องหายออกจาก dropdown ของฟอร์มทันที

#### การแสดงผลใน Dropdown (Step 4 ฝั่ง Proposer)

**FR-11-11** dropdown KPI ใน Step 4 ต้องรองรับ typeahead search กรองรายการขณะพิมพ์ และแสดงเฉพาะ KPI ที่มีสถานะ Active

**FR-11-12** เมื่อ Proposer เลือก KPI จาก Master ระบบต้องดึงค่า **หน่วยวัดเริ่มต้น** มาแสดงใน field หน่วยของ Baseline และ Target โดยอัตโนมัติ (แก้ไขได้)

**FR-11-13** Proposer ยังคงสามารถพิมพ์ชื่อ KPI ใหม่ที่ไม่มีใน Master ได้ (free text) ระบบต้องแสดง indicator แยกแยะให้ชัดเจนว่าเป็น KPI จาก Master หรือ KPI ที่กรอกใหม่

### FR-13 — การจัดการ System Metadata / Configuration (UC-21)

> ฟังก์ชันนี้ใช้งานโดย **Admin** เพื่อกำหนดค่าพารามิเตอร์ของระบบที่อ้างอิงใน Business Rules และ Non-functional Requirements

#### FR-13-01 — การตั้งค่าเกณฑ์งบประมาณ (Budget Threshold)

**FR-13-01-01** Admin ต้องสามารถกำหนดเกณฑ์งบประมาณสำหรับการอนุมัติแต่ละระดับได้

| พารามิเตอร์ | ค่าเริ่มต้น | หมายเหตุ |
|---|---|---|
| เกณฑ์อนุมัติระดับ 1 | 1,000,000 บาท | ต่ำกว่านี้ผ่านได้ 1 ระดับ |
| เกณฑ์อนุมัติระดับ 2 | 5,000,000 บาท | ต่ำกว่านี้ผ่านได้ 2 ระดับ (อ้างอิง BR-01) |
| เกณฑ์อนุมัติระดับ 3 | > 5,000,000 บาท | ต้องผ่านการอนุมัติ 3 ระดับ |

**FR-13-01-02** การเปลี่ยนค่าเกณฑ์ต้องมีผลกับโครงการ **ใหม่** เท่านั้น ไม่ส่งผลย้อนหลัง

---

#### FR-13-02 — การตั้งค่า Draft และ Auto-save

**FR-13-02-01** Admin ต้องสามารถกำหนดค่าพารามิเตอร์ต่อไปนี้ได้

| พารามิเตอร์ | ค่าเริ่มต้น | ช่วงที่อนุญาต |
|---|---|---|
| Draft Expiry (วัน) | 90 วัน | 30–365 วัน |
| Draft Expiry Warning (วันก่อนหมดอายุ) | 15 วัน | 7–30 วัน |
| Auto-save Interval (วินาที) | 30 วินาที | 10–300 วินาที |

---

#### FR-13-03 — การตั้งค่าไฟล์แนบ

**FR-13-03-01** Admin ต้องสามารถกำหนดค่าพารามิเตอร์ไฟล์แนบได้

| พารามิเตอร์ | ค่าเริ่มต้น | ช่วงที่อนุญาต |
|---|---|---|
| ขนาดไฟล์สูงสุดต่อไฟล์ (MB) | 10 MB | 1–50 MB |
| ขนาดรวมสูงสุดต่อโครงการ (MB) | 50 MB | 10–500 MB |
| ประเภทไฟล์ที่อนุญาต | PDF, DOCX, XLSX, PNG, JPG | เลือก multi-checkbox |

---

#### FR-13-04 — การตั้งค่า Approval Routing

**FR-13-04-01** Admin ต้องสามารถกำหนด Approver แต่ละระดับโดย map กับตำแหน่งหรือหน่วยงานได้

**FR-13-04-02** ระบบต้องรองรับ fallback Approver กรณี Approver หลักไม่อยู่เกินจำนวนวันทำการที่กำหนด (ค่าเริ่มต้น 3 วัน อ้างอิง Notification table)

---

#### FR-13-05 — การตั้งค่า Email Notification

**FR-13-05-01** Admin ต้องสามารถเปิด/ปิด และแก้ไข template ของ Email แต่ละเหตุการณ์ได้

| เหตุการณ์ | เปิดใช้งานเริ่มต้น |
|---|---|
| ส่งฟอร์มสำเร็จ | ✅ |
| ร่างใกล้หมดอายุ | ✅ |
| Approver เตือน 3 วันทำการ | ✅ |
| Export ไฟล์สำเร็จ | ❌ |

**FR-13-05-02** Template รองรับ placeholder เช่น `{{project_name}}`, `{{ref_id}}`, `{{proposer_name}}`, `{{due_date}}`

---

#### FR-13-06 — Audit Log ของ Configuration

**FR-13-06-01** ทุกการเปลี่ยนแปลง Configuration ต้องบันทึก Audit Log แยกต่างหากจาก Audit Log ของโครงการ โดยระบุ Admin ผู้เปลี่ยน, ค่าก่อน/หลัง และ timestamp

**FR-13-06-02** Admin ต้องสามารถดู Configuration History ย้อนหลังได้ไม่น้อยกว่า 1 ปี

## 8. Non-functional Requirements (เฉพาะที่เกี่ยวข้อง)

**NFR-01** Auto-save ต้องไม่ทำให้หน้าแอปหยุดทำงาน (non-blocking)

**NFR-02** ฟอร์มต้องใช้งานได้บน browser หลัก (Chrome, Firefox, Edge, Safari) และ responsive บนหน้าจอตั้งแต่ 768px ขึ้นไป

**NFR-03** การส่งฟอร์มต้องแล้วเสร็จภายใน 5 วินาที (รวม file upload ไม่เกิน 50 MB)

## 9. สถานะโครงการ (Proposal Status Flow)

```
Draft → Proposed → Under Review → Approved / Rejected / Returned
```

| สถานะ | ความหมาย | ผู้เปลี่ยน |
|---|---|---|
| Draft | กำลังร่าง ยังไม่ส่ง | Proposer |
| Proposed | ส่งแล้ว รอการพิจารณา | System (อัตโนมัติ) |
| Under Review | อยู่ระหว่างพิจารณา | Approver |
| Approved | อนุมัติแล้ว | Approver |
| Rejected | ปฏิเสธ | Approver |
| Returned | ส่งกลับแก้ไข | Approver |
