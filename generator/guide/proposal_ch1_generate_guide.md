# คู่มือการเขียนบทที่ 1 (บทนำ) ของ Proposal (Proposal Chapter 1 Generation Guide)

> ใช้คู่กับ [`proposal_template_ch1.md`](../template/proposal_template_ch1.md) (โครงเปล่าแบบเติมช่องว่าง) และ [`proposal_ch1_example.md`](../example/proposal_ch1_example.md) (ตัวอย่างที่กรอกครบจริง) — ไฟล์นี้บอก **กฎและวิธีเขียน** บทที่ 1 ของ proposal โครงงาน/ปริญญานิพนธ์ภาษาไทยตามรูปแบบมาตรฐานมหาวิทยาลัยไทย ให้ครบ 6 หัวข้อ (1.1–1.6) ด้วยภาษาวิชาการที่ถูกต้อง อ่านลื่น และสอดคล้องกันทั้งบท

เป้าหมายไม่ใช่แค่ "เติมหัวข้อให้ครบ" แต่ทำให้ทุกหัวข้อ **ร้อยเรียงเป็นเรื่องเดียวกัน** — ปัญหาใน 1.1 ต้องนำไปสู่วัตถุประสงค์ 1.2 ซึ่งสะท้อนกลับมาเป็นประโยชน์ 1.4 และขอบเขต 1.3 ต้องเป็นความสามารถที่ทำให้วัตถุประสงค์เป็นจริง ถ้าหัวข้อเหล่านี้ไม่เชื่อมกัน อาจารย์ที่ปรึกษาจะจับได้ทันที

## 1. ขั้นตอนการทำงาน

1. **รวบรวมข้อมูลที่จำเป็น** (ดู checklist ข้อ 2) — ถ้าข้อมูลที่มีไม่ครบ ให้ถามเฉพาะส่วนที่ขาดและจำเป็นจริง ๆ อย่าถามทีละข้อจนน่ารำคาญ
2. **ขุดข้อมูลจากบริบทก่อนถาม** — ถ้ามี functional requirements, รายการ endpoint, รายชื่อ role, หรือ tech stack อยู่แล้ว (เช่นไฟล์ `fn req/`, use case spec, README) ให้ดึงมาใช้แทนการถามซ้ำ
3. **เขียนทั้ง 6 หัวข้อเป็น Markdown** ตามโครงสร้างและวิธีเขียนข้อ 4 โดยยึด `proposal_template_ch1.md` เป็นโครง
4. **ตรวจความสอดคล้อง** ระหว่าง 1.1 → 1.2 → 1.3 → 1.4 ก่อนส่ง (ดูข้อ 6)

## 2. ข้อมูลที่ต้องมีก่อนเขียน (Checklist)

| ต้องรู้ | ใช้ในหัวข้อ |
|---------|-------------|
| ชื่อระบบ/โครงงาน + ชนิด (เว็บแอป/แอปมือถือ/ฯลฯ) | 1.1, 1.2, ทั้งบท |
| บริบท + ปัญหาเดิม (ทำด้วยมือ? กระจัดกระจาย? ตรวจสอบยาก?) | 1.1 |
| กลุ่มผู้ใช้ (roles) ทั้งหมด | 1.2, 1.3 |
| ความสามารถของแต่ละ role | 1.3 |
| เทคโนโลยี/เครื่องมือที่ใช้ (frontend, backend, DB, ภาษา) | 1.5 |
| ช่วงเวลาดำเนินงาน (เดือน/ปีเริ่ม–จบ) | 1.6 |

ถ้าขาด **ปัญหาเดิม** หรือ **รายการ role + ความสามารถ** ให้ถามก่อนเสมอ เพราะเป็นแกนของบท เขียนเดาเองไม่ได้ ส่วน tech stack / timeline ถ้าไม่มีให้ใส่ค่าที่สมเหตุสมผลแล้วทำเครื่องหมายให้แก้ทีหลัง

## 3. โครงสร้างบทที่ 1 (ใช้ลำดับนี้เสมอ)

```
# บทที่ 1
บทนำ
## 1.1 ที่มาและความสำคัญ
## 1.2 วัตถุประสงค์              (ข้อ 1.2.1, 1.2.2, ...)
## 1.3 ขอบเขตของโครงงาน        (### 1.3.x ต่อ role → รายการความสามารถ แบบเลข 1, 2, ... หรือ bullet)
## 1.4 ประโยชน์ที่คาดว่าจะได้รับ   (ข้อ 1.4.1, 1.4.2, ...)
## 1.5 เครื่องมือที่ใช้ในการพัฒนา   (### 1.5.x ฮาร์ดแวร์/ซอฟต์แวร์/ภาษา/ฐานข้อมูล → รายการ แบบเลข 1, 2, ... หรือ bullet)
## 1.6 แผนการดำเนินการ           (ย่อหน้านำ + ตาราง Gantt)
```

> **กฎเลขหัวข้อ: ลึกได้ไม่เกิน 3 ระดับ** (เช่น `1.3.1` — ห้าม `1.3.1.1`) รายการที่อยู่ลึกกว่านั้นให้เริ่มด้วยเลขลำดับ `1. 2. 3.` หรือ bullet (`- ...`) ตามความเหมาะสมของบริบท

> รูปแบบ heading anchor `{#...}` ท้ายหัวข้อ (เช่น `## 1.1 ที่มาและความสำคัญ {#1.1-ที่มาและความสำคัญ}`) เป็นของ Google Docs ตอน export — ใส่ให้ถ้าทำงานบน Google Docs อยู่แล้ว ถ้าไม่แน่ใจ ใส่ได้ ไม่เสียหาย

## 4. วิธีเขียนแต่ละหัวข้อ

### 1.1 ที่มาและความสำคัญ — เขียน 3 ย่อหน้า (สูตร "บริบท → ปัญหา → ทางแก้")

หัวใจของบททั้งหมด ต้องวางเป็น 3 จังหวะ เพราะมันคือการ "ขายปัญหา" ก่อนเสนอทางแก้:
1. **บริบท/สภาพปัจจุบัน** — ใคร ทำอะไร ภายใต้เงื่อนไขอะไร (ขึ้นต้นด้วย "ในปัจจุบัน...") แล้วชี้ช่องว่างของวิธีเดิม ("อย่างไรก็ตาม...")
2. **ปัญหาเชิงลึก + ผลกระทบรูปธรรม** — ยกตัวอย่างให้เห็นภาพ (ความล่าช้า ข้อมูลไม่ตรงกัน เอกสารหาย ตรวจย้อนหลังยาก) แล้วสรุปผลกระทบรวม ("ส่งผลให้...ไม่มีประสิทธิภาพ")
3. **ตระหนัก + เสนอสิ่งที่พัฒนา** — ขึ้นต้น "จากปัญหาดังกล่าว ผู้พัฒนาจึงตระหนักถึง..." แล้วบอกว่าพัฒนาอะไร เพื่ออะไร และช่วยใครได้อย่างไร

ปัญหาที่ยกใน 1.1 ต้องเป็นปัญหาที่ระบบ **แก้ได้จริง** เท่านั้น อย่ายกปัญหาที่ระบบไม่เกี่ยว เพราะมันจะลอยไม่เชื่อมกับ 1.2/1.4

### 1.2 วัตถุประสงค์ — เขียนเป็นข้อ (ไม่ใช่ย่อหน้ายาว)

แตกเป็นข้อย่อย `1.2.1`, `1.2.2`, ... แต่ละข้อเป็นวัตถุประสงค์ที่ชัดและวัดผลได้ 1 เรื่อง ขึ้นต้นด้วยคำกริยา ("เพื่อพัฒนา...", "เพื่อเพิ่ม...", "เพื่อลด..."):
- **ข้อแรกเป็นวัตถุประสงค์หลัก** — "เพื่อพัฒนา {ชื่อระบบ}" พร้อมระบุ role ทั้งหมดที่จะปรากฏใน 1.3
- **ข้อถัด ๆ ไปเป็นวัตถุประสงค์รอง** — เพิ่มประสิทธิภาพ / ลดปัญหา ที่ล้อกับปัญหาใน 1.1

โดยทั่วไป 2–4 ข้อ ข้อละ 1–2 บรรทัด สั้นกระชับ — ห้ามยกเป็นย่อหน้ายาวก้อนเดียว เพราะวัตถุประสงค์ที่แยกเป็นข้อจะตรวจสอบและอ้างอิงกลับมาที่ 1.4 ได้ง่ายกว่า

### 1.3 ขอบเขตของโครงงาน — แตกตาม role

- 1 `### 1.3.x` ต่อ 1 role (เรียงตามที่ระบุใน 1.2) — หัวข้อ role เป็นเลข 3 ระดับ (1.3.1, 1.3.2, ...)
- ใต้แต่ละ role ลิสต์ความสามารถ **เป็นเลขลำดับ `1. 2. 3.` หรือ bullet** (เลือกให้เข้ากับบริบท) — ไม่ใช้เลข 4 ระดับแบบ `1.3.1.1` เพราะเกินกฎ 3 ระดับ
- **ทุกข้อลงท้ายด้วย "...ได้"** (สมัครสมาชิกได้, เข้าสู่ระบบได้, แก้ไขข้อมูลส่วนตัวได้) เพราะขอบเขตคือ "สิ่งที่ผู้ใช้ทำได้ในระบบ"
- เรียงจากฟังก์ชันพื้นฐาน (สมัคร/เข้าสู่ระบบ/แก้โปรไฟล์) ไปฟังก์ชันเฉพาะทางของ role นั้น
- role ที่สิทธิ์สูงกว่ามักครอบฟังก์ชันของ role ล่าง + เพิ่มสิทธิ์อนุมัติ/จัดการ — สะท้อนให้ตรงความจริง

### 1.4 ประโยชน์ที่คาดว่าจะได้รับ — 3–5 ข้อ

แต่ละข้อ **map กับวัตถุประสงค์** และมองจากมุม "ใครได้อะไร": (ก) ผู้ใช้กลุ่มหลัก (ข) ผู้ปฏิบัติงาน/เจ้าหน้าที่ (ค) เชิงระบบ/องค์กร (ลดความผิดพลาด เพิ่มประสิทธิภาพ มีข้อมูลสรุป) เลขข้อ `1.4.x`

### 1.5 เครื่องมือที่ใช้ในการพัฒนา (Development Tools)

แยก 4 หัวข้อย่อยเสมอ: `### 1.5.1 ฮาร์ดแวร์` / `### 1.5.2 ซอฟต์แวร์` / `### 1.5.3 ภาษาโปรแกรมที่ใช้` / `### 1.5.4 ฐานข้อมูลที่ใช้` — ใต้แต่ละหัวข้อลิสต์รายการ **เป็นเลขลำดับ `1. 2. 3.` หรือ bullet** (ไม่ใช้เลข 4 ระดับ `1.5.x.y`) แต่ละรายการเขียน "ชื่อเครื่องมือ + วลีว่าใช้ทำอะไร" (เช่น "Django Framework เป็นเฟรมเวิร์กโอเพนซอร์สสำหรับพัฒนาเว็บแอปพลิเคชันด้วยภาษา Python")

### 1.6 แผนการดำเนินการ — ย่อหน้านำ + ตาราง Gantt

1 บรรทัดนำ ("...ผู้พัฒนาได้แบ่งขั้นตอนการดำเนินงานออกเป็น N ขั้นตอน ดังตารางที่ 1.1") + คำบรรยายตาราง "ตารางที่ 1.1: ขั้นตอนการดำเนินงาน" + ตาราง Markdown ที่หัวคอลัมน์เป็นเดือน/ปี และแถวเป็น 8 ขั้นตอนมาตรฐาน (ศึกษาความเป็นไปได้ → นำเสนอหัวข้อ → ศึกษาค้นคว้าข้อมูล → ศึกษาการใช้เครื่องมือ → วิเคราะห์และออกแบบ → เขียนโปรแกรม → ทดสอบและแก้ปัญหา → จัดทำเอกสาร) ระบายช่องเดือนที่ทำจริงด้วย `▓` ถ้ารู้ timeline

## 5. หลักการเขียนภาษาไทยวิชาการ (ใช้ทั้งบท)

- ใช้สรรพนาม **"ผู้พัฒนา"** แทนตัวเอง ห้ามใช้ "ผม/ฉัน/เรา" หรือภาษาพูด
- น้ำเสียงทางการ ประโยคสมบูรณ์ หลีกเลี่ยงคำฟุ่มเฟือยและคำกำกวม
- เว้นวรรคแบบไทยให้ถูก หลีกเลี่ยงประโยคยาวเกินจนจับใจความยาก
- เลขหัวข้อต้องต่อเนื่องถูกลำดับ และ **ลึกไม่เกิน 3 ระดับ** (เช่น 1.3.1) — รายการที่ลึกกว่านั้นใช้เลขลำดับ `1. 2. 3.` หรือ bullet

### ศัพท์วิชาการและศัพท์เทคนิค (สำคัญ — อาจารย์จับเรื่องนี้เป็นอันดับต้น ๆ)

ภาษาวิชาการที่ "ดูน่าเชื่อถือ" อยู่ที่การเลือกคำให้ถูกทะเบียน (register) และจัดการศัพท์ภาษาอังกฤษให้เป็นระบบ ไม่ใช่แค่เขียนยาว ๆ ให้ยึดกฎเหล่านี้:

- **ใช้ศัพท์ทางการ ไม่ใช่คำพูด** — เลือกคำระดับเอกสารราชการ/วิชาการ:
  - ใช้ "เว็บแอปพลิเคชัน / ระบบ / แอปพลิเคชัน" — ไม่ใช้ "เว็บ / แอป / โปรแกรม" แบบลำลอง
  - ใช้ "ผู้ใช้งาน, ดำเนินการ, จัดเก็บ, ตรวจสอบ, ประยุกต์ใช้, ทั่วถึง, มีประสิทธิภาพ" — เลี่ยง "คนใช้, ทำ, เก็บ, เช็ก, เอามาใช้, ครบ ๆ, เวิร์ก"
- **ทับศัพท์ตามหลักราชบัณฑิตยสภา** ให้สะกดถูกและคงเส้นคงวา เช่น แอปพลิเคชัน, เทคโนโลยี, ฐานข้อมูล, เฟรมเวิร์ก, ออนไลน์, ดาวน์โหลด, อัปโหลด (ไม่ใช่ "อัพโหลด"), แดชบอร์ด — เลือกรูปไหนแล้วใช้รูปนั้นทั้งบท
- **ศัพท์เทคนิคครั้งแรก: ไทย (English)** — พบครั้งแรกให้วงเล็บภาษาอังกฤษกำกับ เช่น "หน่วยกิตกิจกรรม", "การจัดการเอกสารด้วยตนเอง (Manual)", "แดชบอร์ด (Dashboard)" ครั้งถัดไปใช้คำไทยล้วน ไม่ต้องวงเล็บซ้ำ
- **ชื่อเฉพาะ/ผลิตภัณฑ์/ภาษาโปรแกรม คงรูปอังกฤษ** — Django Framework, Python, MySQL, SQLite3, Tailwind CSS, Visual Studio Code, HTML, Facebook (ไม่ทับศัพท์)
- **คงคำเรียกระบบให้เดียวกันทั้งบท** — ถ้าเริ่มด้วย "เว็บแอปพลิเคชันระบบ..." อย่าสลับไปเป็น "โปรแกรม" หรือ "แอป" กลางคัน
- **เลี่ยงศัพท์เทคนิคเกินจำเป็นใน 1.1–1.2** — บทนำเขียนให้กรรมการที่อาจไม่ใช่สายเทคนิคอ่านรู้เรื่อง เก็บศัพท์ลึก (API, endpoint, framework) ไว้ที่ 1.5 หรือบทถัด ๆ ไป

## 6. ตรวจก่อนส่ง (Consistency check)

- 1.2 วัตถุประสงค์เขียนเป็นข้อ (1.2.1, 1.2.2, ...) ไม่ใช่ย่อหน้ายาว
- ทุก role ที่พูดถึงใน 1.2 มีหัวข้อใน 1.3 ครบ
- ปัญหาทุกข้อใน 1.1 มีวัตถุประสงค์/ขอบเขตรองรับ และสะท้อนเป็นประโยชน์ใน 1.4
- จำนวนขั้นตอนที่บอกใน 1.6 ตรงกับจำนวนแถวในตาราง
- ทุกข้อใน 1.3 ลงท้าย "...ได้"
- ไม่มีเลขหัวข้อใดลึกเกิน 3 ระดับ (`1.3.1.1` = ผิด → ใช้ `1.` หรือ bullet)
