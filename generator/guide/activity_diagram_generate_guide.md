# คู่มือการวาด Activity Diagram แบบมี Swimlane (Activity Diagram Generation Guide) — สถาปัตยกรรม Monolith

> ใช้คู่กับ [`activity_diagram_template.md`](../template/activity_diagram_template.md) เป็นจุดเริ่มก๊อปปี้ และดู [`activity_diagram_example.md`](../example/activity_diagram_example.md) เป็นตัวอย่างที่ทำตามกฎครบทุกข้อ
>
> **หมายเหตุ:** คู่มือนี้เขียนสำหรับระบบที่พัฒนาเป็น **monolith** (แอปพลิเคชันเดียว ฐานข้อมูลเดียว) — ไม่มี API Gateway, ไม่มีการเรียกข้าม service, ไม่มี Audit Log service แยก

---

## 0. นิยาม: Activity Diagram ต่างจาก Use Case Diagram อย่างไร (อ่านก่อนเริ่ม)

**Use case diagram** ([`usecase_generate_guide.md`](usecase_generate_guide.md)) ตอบคำถาม "ผู้ใช้ทำอะไรได้บ้าง" ในระดับเป้าหมาย (goal-level) — ห้ามลงรายละเอียดขั้นตอนภายใน

**Activity diagram** (ไฟล์นี้) ตอบคำถามตรงข้าม — "1 use case นั้นทำงานเป็นขั้นตอนอย่างไรจริง ๆ" ระหว่างผู้ใช้กับระบบ จึงเป็นที่ที่ business rule, decision, และ loop ที่ถูกห้ามไม่ให้ใส่ใน use case diagram ควรมาอยู่แทน

ใช้ activity diagram แบบมี swimlane เมื่อ:
- ต้องการอธิบาย 1 use case ที่มีขั้นตอนโต้ตอบไปมาระหว่างผู้ใช้กับระบบหลายจังหวะ
- ต้องการโชว์กรรมการสอบว่า "ใครทำอะไร ตอนไหน" ในกระบวนการเดียว
- ต้องการเห็น decision/loop ที่ use case diagram บอกไม่ได้ (เช่น เงื่อนไขแตกกิ่ง, การวนซ้ำจนกว่าจะผ่าน)

---

## 1. กฎเรื่อง Partition (Swimlane)

**1.1 ระบบ monolith ทั้งก้อนเป็น partition เดียวชื่อ "ระบบ" — ห้ามแยกย่อย**

เพราะทั้งแอปคือหน่วยเดียวกัน (โค้ดเดียว ฐานข้อมูลเดียว) จึงมี swimlane ฝั่งซอฟต์แวร์แค่เลนเดียว:

- **ห้าม**แยก partition ตาม layer ภายใน (Controller / Service / Repository / Database) — เป็นโครงสร้างภายในของแอปเดียวกัน ไม่ใช่คนละหน่วย
- **ห้าม**แยก partition ตาม module/ฟีเจอร์ (เช่น "ส่วนจัดการเซสชัน", "ส่วนอวตาร") — module เป็นเพียงการจัดกลุ่มโค้ดใน monolith ไม่ใช่ระบบแยก
- **ห้าม**สร้าง partition ประเภท Gateway, Audit Log, Notification Service, Data Center หรือ infrastructure ใด ๆ — สิ่งเหล่านี้เป็นแนวคิดของสถาปัตยกรรม microservice ไม่มีในระบบนี้

```plantuml
' ถูกต้อง — ฝั่งซอฟต์แวร์มีเลนเดียว
|สมาชิก|
:กดปุ่มเริ่ม;
|ระบบ|
:ตรวจสอบเงื่อนไข;
:บันทึกข้อมูล;
```

```plantuml
' ผิด — ห้ามแยกระบบเป็นหลายเลน
|Controller|
:รับ request;
|Service Layer|
:ประมวลผล;
|Database|
:บันทึก;
```

**1.2 Actor ที่เป็นมนุษย์ (สมาชิก, ผู้ดูแลระบบ ฯลฯ) แต่ละคนเป็น partition ของตัวเอง** — ใช้ชื่อเดียวกับ actor ใน use case diagram ถ้ากระบวนการเกี่ยวข้องกับ actor คนเดียว ก็มีแค่ 2 เลน (actor + ระบบ) ซึ่งเป็นกรณีปกติที่สุด

> ข้อแตกต่างจาก use case diagram: ใน use case diagram ห้ามมี actor ชื่อ "ระบบ" (เพราะระบบคือ boundary) แต่ใน activity diagram แบบ swimlane **เลน "ระบบ" เป็นเรื่องถูกต้อง** — เพราะ swimlane ตอบว่า "ขั้นตอนนี้ใครเป็นคนทำ" ซึ่งฝั่งหนึ่งคือผู้ใช้ อีกฝั่งคือซอฟต์แวร์

**1.3 ระบบภายนอกจริง ๆ เท่านั้นที่แยกเลนได้** — ถ้ากระบวนการต้องเรียกบริการภายนอกที่ไม่ใช่ของเรา (เช่น payment gateway ของธนาคาร, บริการส่งอีเมล) จึงจะเพิ่ม partition แยกได้ และต้องตั้งชื่อเฉพาะเจาะจง (เช่น "Payment Gateway (ภายนอก)") — ถ้าไม่มีการเรียกภายนอก ห้ามเพิ่มเลนเกิน actor + ระบบ

**1.4 ถ้าเลนเดียวกันถูกใช้หลายช่วง ให้ใช้ชื่อเดิมซ้ำทุกครั้ง** — PlantUML จะรวมเป็นคอลัมน์เดียวกันเองตราบใดที่ชื่อตรงกันทุกตัวอักษร

## 2. สิ่งที่ไม่ต้องวาด (เพราะเป็น monolith)

**2.1 ไม่ต้องมีขั้นตอน "ตรวจสิทธิ์/JWT/Gateway" คั่นหน้า activity** — การล็อกอินและสิทธิ์การเข้าถึงถือเป็น **pre-condition** ที่เขียนไว้ในตาราง use case description อยู่แล้ว การวาดซ้ำในทุกไดอะแกรมทำให้รกโดยไม่เพิ่มข้อมูล (ยกเว้น use case นั้นคือการล็อกอินเอง)

**2.2 ไม่ต้องมี Audit Log / fork เขียน log** — ระบบ monolith บันทึกข้อมูลลงฐานข้อมูลเดียวกันในขั้นตอนปกติ ไม่มี audit service แยกที่ต้องวาดเป็นกิ่งขนาน ถ้าจำเป็นต้องสื่อว่ามีการบันทึกประวัติ ให้เขียนเป็นกิจกรรมปกติในเลนระบบ เช่น `:บันทึกเซสชันลงประวัติ;`

**2.3 การแจ้งเตือนเป็นกิจกรรมปกติในเลนระบบ** — เช่น `:แสดงการแจ้งเตือนให้เริ่มเซสชันถัดไป;` ไม่ต้องแยกเป็น Notification Service

**2.4 ใช้ `fork` เฉพาะเมื่อระบบทำสองสิ่งพร้อมกันจริงตาม business logic** — เช่น จับเวลาถอยหลังไปพร้อมกับอัพเดตแถบเลือดบอส ไม่ใช่เพื่อแสดง cross-cutting concern

## 3. Loop / Retry Pattern

ใช้ `repeat` / `repeat while` แทนการวาดลูกศรย้อนกลับเอง เพื่อสื่อว่าต้องวนจนกว่าเงื่อนไขจะผ่าน ไม่ใช่ทำครั้งเดียวจบ

```plantuml
|สมาชิก|
repeat
  :กรอก/แก้ไขข้อมูล;
  |ระบบ|
  :ตรวจสอบความถูกต้อง;
repeat while (ข้อมูลผ่านการตรวจสอบหรือไม่?) is (ไม่ผ่าน) not (ผ่าน)
```

`is (...)` คือ label บนเส้นที่วนกลับไปทำซ้ำ ส่วน `not (...)` คือ label บนเส้นที่ออกจาก loop — ระวังอย่าสลับสองอันนี้กัน

## 4. Style มาตรฐาน — Monochrome เท่านั้น

**กฎสำคัญ: ห้ามให้แต่ละ partition มีสีพื้นหลังต่างกันแบบสีรุ้ง (ห้าม 1 partition = 1 สี)** — ใช้โทนเดียว (ขาว/เทา) ทั้งไดอะแกรม:

| บทบาท | พื้นหลัง | เส้นขอบ |
|---|---|---|
| Actor ภายนอก (human) | ขาว (`white`) | ทึบ (`#424242`) |
| ระบบ | เทาอ่อน (`#F5F5F5`) | ทึบ (`#424242`) |
| ระบบภายนอก (ถ้ามี) | เทาอ่อน (`#F5F5F5`) | สื่อด้วยข้อความ "(ภายนอก)" ต่อท้ายชื่อเลน ไม่ใช่สี |

```plantuml
skinparam activity {
  BackgroundColor #F5F5F5
  BorderColor #424242
  FontName "TH Sarabun New"
  FontSize 13
  DiamondBackgroundColor #FAFAFA
  DiamondBorderColor #616161
}
```

**ห้ามใช้สีตระกูลต่างกัน (ส้ม/ฟ้า/เขียว/ม่วง) แยกแต่ละ partition เด็ดขาด** — ทุกอย่างอยู่ในโทนเทา-ขาวเดียวกัน แยกด้วย label เท่านั้น

## 5. Checklist ก่อนส่งไดอะแกรม

- [ ] ฝั่งซอฟต์แวร์มี partition เดียวชื่อ "ระบบ" — ไม่แยกตาม layer, module, หรือ infrastructure (ข้อ 1.1)
- [ ] ไม่มี partition ประเภท Gateway, Audit Log, Notification Service, Data Center (ข้อ 1.1, 2.2)
- [ ] ไม่มีขั้นตอนตรวจสิทธิ์/JWT คั่น — สิทธิ์เป็น pre-condition ใน use case description (ข้อ 2.1)
- [ ] เลนเดียวกันที่ถูกใช้ซ้ำใช้ชื่อเดิมทุกครั้ง ไม่สร้างซ้ำ (ข้อ 1.4)
- [ ] Loop ใช้ `repeat`/`repeat while` ไม่ใช่ลูกศรย้อนกลับเอง และ label `is`/`not` ไม่สลับกัน
- [ ] **ทั้งไดอะแกรมใช้โทนเดียว (ขาว/เทา) ไม่มี partition ไหนมีสีต่างจากที่อื่น**
- [ ] ทุก decision/exception ในไดอะแกรมตรงกับ Basic/Alternative Flow ในตาราง use case description ของ use case นั้น
