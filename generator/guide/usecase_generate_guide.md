# คู่มือการวาด Use Case Diagram (Usecase Diagram Generation Guide)

> ใช้คู่กับ [`template_usecase_description.md`](../template/template_usecase_description.md) (use case แบบข้อความ) — ไฟล์นี้บอกกฎเฉพาะสำหรับการวาด **use case diagram** (`.puml`) ให้ถูกต้องตามหลัก UML และสอดคล้องกับ use case description ที่เขียนคู่กัน
>
> ใช้ [`usecase_template.md`](../template/usecase_template.md) เป็นจุดเริ่มก๊อปปี้ และดู [`usecase_example.md`](../example/usecase_example.md) เป็นตัวอย่างที่ทำตามกฎครบทุกข้อ

## 0. นิยาม: Use Case ต่างจาก Process/Flow การทำงานอย่างไร (สำคัญที่สุด — อ่านก่อนเริ่ม)

**Use case ต้องอยู่ระดับ "เป้าหมายของผู้ใช้" (user-goal level) เท่านั้น** — คือสิ่งที่ actor ภายนอกต้องการบรรลุแล้วได้ผลลัพธ์ที่มีคุณค่าสังเกตเห็นได้ ("ผู้ใช้อยากทำอะไร" ไม่ใช่ "ระบบทำอะไรข้างในเพื่อให้สิ่งนั้นสำเร็จ")

**วิธีทดสอบ:** ลองพูดประโยค **"ในฐานะ {actor} ฉันต้องการ {ทำสิ่งนี้} เพื่อ {ได้ผลลัพธ์}"** — ถ้าพูดแล้วเป็นธรรมชาติ = use case ถ้าพูดไม่ได้เพราะมันเป็นแค่การคำนวณ/เปลี่ยนสถานะเบื้องหลังที่ไม่มีใคร "อยากทำ" มันโดยตรง = ไม่ใช่ use case

**Process/Flow** คือขั้นตอนย่อยหรือกฎทางธุรกิจ (business rule) ที่ทำงานเป็นผลสืบเนื่องของ use case ใดหนึ่ง ไม่มีคุณค่าที่ actor ต้องการเป็นเป้าหมายของตัวมันเอง — ให้เขียนเป็นขั้นตอนใน **Basic Flow / Alternate Flow / Business Rules** ของ use case ที่มันสังกัดอยู่ (ในเอกสาร use case description) ไม่ใช่แยกเป็น use case ใหม่ในไดอะแกรม

### สัญญาณเตือนว่ากำลังสร้าง "process" ปลอมเป็น "use case" (functional decomposition — ข้อผิดพลาดที่พบบ่อยที่สุด)

- ไม่มี actor ที่ริเริ่มด้วยความสมัครใจ (เกิดขึ้นเองอัตโนมัติเสมอ เป็นผลสืบเนื่องของอย่างอื่น)
- ชื่อ use case เป็นคำกริยาระดับ "คำนวณ", "ตรวจสอบ", "อัพเดต", "ประมวลผล" มากกว่าคำว่า "เป้าหมายที่ผู้ใช้อยากทำ"
- ถูกเรียกจาก use case อื่นเพียงจุดเดียวเสมอ ไม่เคยถูกเรียกอิสระหรือใช้ซ้ำจากหลายจุด (สัญญาณว่ามันควรเป็นแค่ step ของ use case นั้น ไม่ใช่แยก use case แล้วโยง `<<include>>`)
- อธิบายเป็นประโยคแล้วประธานต้องเป็น "ระบบ..." ไม่ใช่ "{actor}..."

### เมื่อไหร่ยังยอมให้แยกเป็น use case ได้แม้ actor ไม่ได้ริเริ่มตรง ๆ

มีข้อยกเว้นน้อยกรณี: เมื่อพฤติกรรมนั้นซับซ้อนพอจะมี pre/post-condition และ alternate flow ของตัวเองจริงจัง **และ** ถูกเรียกใช้ซ้ำจาก use case มากกว่า 1 จุด (สมควรแยกเป็น `<<include>>` เพื่อลดการเขียนซ้ำเอกสาร) — ถ้าเข้าเงื่อนไขนี้ไม่ครบทั้งสองข้อ ให้ยุบรวมเป็น step เดียวในเอกสาร use case description ของ use case หลักแทน อย่าสร้าง bubble แยก

### วิธีแก้เมื่อพบ process ปลอมเป็น use case ในไดอะแกรมที่มีอยู่แล้ว

1. เอา use case bubble นั้นออกจากไดอะแกรม
2. ย้ายเนื้อหาของมัน (pre/post-condition, basic flow) ไปเป็นขั้นตอน/business rule ใน use case ที่มันเคยเชื่อมด้วย `<<include>>`/`<<extend>>`
3. ถ้ามันถูกเรียกจากหลายจุดจริงและซับซ้อนพอ ให้พิจารณาคงเป็น use case แยกได้ตามเงื่อนไขข้างต้น แต่ต้องมั่นใจว่ามันมีคุณค่าที่ actor ต้องการจริง ไม่ใช่แค่กลไกภายใน

## 1. กฎเรื่อง Actor

**1.1 Actor ต้องเป็นสิ่งภายนอกขอบเขตระบบเท่านั้น**

ห้ามสร้าง actor ชื่อ "ระบบ" / "System" เด็ดขาด เพราะตัวระบบถูกแทนด้วยกรอบ (system boundary/rectangle) อยู่แล้ว การให้ระบบเป็น actor ของตัวเองขัดกับนิยาม actor ใน UML (actor คือสิ่งภายนอกที่มีปฏิสัมพันธ์กับระบบเพื่อบรรลุเป้าหมาย)

ถ้ามี use case ที่ดูเหมือนไม่มี actor ริเริ่มชัดเจน (เช่น การคำนวณอัตโนมัติ, ประมวลผลเบื้องหลัง, business rule ที่ทำงานเองตามเงื่อนไข) — นั่นแปลว่า use case นั้น**ไม่ใช่ use case อิสระ** แต่เป็นพฤติกรรมภายในที่ควรผูกเข้ากับ use case ของ actor จริงด้วยความสัมพันธ์ `<<include>>` หรือ `<<extend>>` แทน (ดูข้อ 2)

ข้อยกเว้นเดียว: ถ้าสิ่งนั้นเป็น**ระบบภายนอกจริง ๆ** ที่แยกขอบเขตออกจากระบบที่นำเสนอ (เช่น payment gateway, third-party API, cron service ที่ดูแลแยกจากกัน) จะนับเป็น secondary actor ได้ แต่ต้องตั้งชื่อเฉพาะเจาะจง (เช่น "Payment Gateway") ไม่ใช่คำกว้าง ๆ ว่า "ระบบ"

**1.2 เส้นเชื่อม actor–use case ต้องเป็นเส้นธรรมดา ไม่ใช้ลูกศร**

```plantuml
' ถูกต้อง
Actor -- UseCase

' ผิด — ห้ามใช้ลูกศรกับ actor
Actor --> UseCase
```

ลูกศรสงวนไว้สำหรับความสัมพันธ์ระหว่าง use case กับ use case เท่านั้น (`<<include>>` / `<<extend>>`) และเส้นสามเหลี่ยมกลวงสงวนไว้สำหรับ generalization (ดูข้อ 1.3 และ 2.1)

**1.3 Generalization ระหว่าง Actor กับ Actor**

นิยาม: actor ลูก (child actor) เป็น**กรณีเฉพาะ**ของ actor แม่ (parent actor) — actor ลูกทำทุกอย่างที่ actor แม่ทำได้ (สืบทอด use case ทั้งหมดที่ actor แม่เชื่อมอยู่) **บวก**ความสามารถเพิ่มเติมเฉพาะของตัวเอง ใช้เมื่อมี actor สองตัวที่มีความสัมพันธ์แบบ "เป็นชนิดหนึ่งของ" (is-a) ไม่ใช่แค่คล้ายกัน

วิธีวาด: เส้นทึบ + หัวลูกศรเป็น**สามเหลี่ยมกลวง** ชี้จาก actor ลูกไปหา actor แม่ (ทิศทางเดียวกับ inheritance ใน class diagram)

```plantuml
' Actor ลูก --|> Actor แม่
RoomMember --|> Member
```

ตัวอย่างจริง: "สมาชิกในห้อง (Room Member)" คือสมาชิก (Member) ที่กำลังอยู่ในห้องโฟกัส จึงทำทุกอย่างที่ Member ทำได้ (ยืม/คืน, ดูประวัติ ฯลฯ) บวกความสามารถเฉพาะในห้อง (ดูอันดับ, ปิดเป้าหมายห้อง) — เขียนเป็น `RoomMember --|> Member` แล้วเชื่อม use case เฉพาะของห้องเข้ากับ `RoomMember` เท่านั้น ไม่ต้องเชื่อมซ้ำกับ use case ที่ `Member` เชื่อมอยู่แล้ว

ข้อควรระวัง: อย่าใช้ generalization แทนการเชื่อม actor ตรง ๆ กับ use case พร่ำเพรื่อ — ใช้เฉพาะเมื่อมีความสัมพันธ์ is-a จริง ถ้าแค่ actor สองตัวเรียก use case เดียวกันบางส่วน ให้เชื่อมเส้นธรรมดา (`--`) แยกกันตามข้อ 1.2 แทน

## 2. กฎเรื่องความสัมพันธ์ระหว่าง Use Case

มี 3 ความสัมพันธ์ที่ใช้ระหว่าง use case กับ use case — **generalization, `<<include>>`, `<<extend>>`** แต่ละแบบมีความหมายและวิธีวาดต่างกัน ห้ามใช้ปนกัน

| ความสัมพันธ์ | เส้น/หัวลูกศร | ทิศทาง | ใช้เมื่อไหร่ |
|---|---|---|---|
| Generalization | เส้นทึบ + สามเหลี่ยมกลวง | `Specific --|> General` | use case ลูกเป็น**กรณีเฉพาะ**ของ use case แม่ ทำ flow เดียวกันแต่ต่างรายละเอียดบางจุด |
| `<<include>>` | เส้นประ + หัวลูกศรธรรมดา | `Base ..> Included` | ขั้นตอนนั้นเกิดขึ้น**เสมอ**เป็นส่วนหนึ่งของ use case หลัก (mandatory step) |
| `<<extend>>` | เส้นประ + หัวลูกศรธรรมดา | `Extension ..> Base` | ขั้นตอนนั้นเกิดขึ้น**บางครั้ง**ตามเงื่อนไข/ทางเลือก (optional/conditional branch) |

### 2.1 Generalization (use case ต่อ use case)

นิยาม: use case ลูกทำ**เป้าหมายเดียวกัน**กับ use case แม่ แต่มีวิธีการ/เงื่อนไขเฉพาะที่ต่างออกไปบางส่วน (ไม่ใช่แค่ "เกิดขึ้นบางครั้ง" แบบ extend แต่เป็น "อีกรูปแบบหนึ่ง" ของสิ่งเดียวกัน)

วิธีวาด: เส้นทึบ + หัวลูกศรสามเหลี่ยมกลวง ชี้จาก use case ลูกไปหา use case แม่ (ทิศทางเดียวกับ actor generalization ในข้อ 1.3)

```plantuml
' Use case ลูก --|> Use case แม่
usecase "ยืมหนังสือด้วยบัตรสมาชิก" as UC02a
usecase "ยืมหนังสือด้วยบัตรประชาชน\n(ไม่มีบัตรสมาชิก)" as UC02b
usecase "ยืมหนังสือ" as UC02

UC02a --|> UC02
UC02b --|> UC02
```

ใช้น้อยกว่า include/extend มาก — ส่วนใหญ่ใช้เมื่อมี use case หลักที่ทำได้หลายวิธี (เช่น ยืนยันตัวตนได้ทั้งด้วยบัตรสมาชิกหรือบัตรประชาชน) ถ้าไม่แน่ใจว่าจะใช้ generalization หรือ extend ให้ถามตัวเองว่า "มันคือ**อีกวิธีทำสิ่งเดียวกัน**" (generalization) หรือ "มันคือ**ขั้นตอนเสริมที่เกิดตามเงื่อนไข**" (extend)

### 2.2 `<<include>>`

วิธีวาด: เส้นประ + หัวลูกศรธรรมดา ชี้จาก use case หลัก (base) ไปหา use case ที่ถูกรวม (included) พร้อม stereotype `<<include>>` กำกับที่เส้น

```plantuml
UC01 ..> UC02 : <<include>>
```

ใช้เมื่อขั้นตอนนั้นเป็นส่วนบังคับที่เกิดขึ้นทุกครั้งที่ทำ use case หลัก เช่น "สมัครสมาชิก" ต้อง include "สร้างอวตาร" เสมอเพราะบัญชีใหม่ทุกบัญชีต้องมีอวตาร

### 2.3 `<<extend>>`

วิธีวาด: เส้นประ + หัวลูกศรธรรมดา ชี้จาก use case ส่วนขยาย (extension) ไปหา use case หลัก (base) พร้อม stereotype `<<extend>>` กำกับที่เส้น — **ทิศทางตรงข้ามกับ include** (ชี้เข้าหาตัวหลัก ไม่ใช่ออกจากตัวหลัก)

```plantuml
UC05 ..> UC04 : <<extend>>
```

ใช้เมื่อขั้นตอนนั้นเป็นทางเลือก/กรณีพิเศษที่เกิดขึ้นเฉพาะบางเงื่อนไข เช่น "ยกเลิกเซสชันกลางทาง" (UC05) ขยายจาก "เริ่มจับเวลาโฟกัส" (UC04) เพราะเกิดเฉพาะเมื่อผู้ใช้กดยกเลิกเท่านั้น ไม่ใช่ทุกครั้ง

ข้อผิดพลาดที่พบบ่อย: กลับทิศทางลูกศรของ `<<extend>>` (เขียน Base ..> Extension) — ทิศทางที่ถูกต้องคือ **use case ส่วนขยายชี้เข้าหา use case หลักเสมอ** ไม่ใช่กลับกัน

ใช้ `<<include>>`/`<<extend>>` แทนการสร้าง actor "System" เพื่อโยง use case ที่เป็นพฤติกรรมอัตโนมัติเข้ากับ use case ที่ actor จริงเริ่มต้น เช่น การคำนวณ/อัพเดตสถานะเบื้องหลังที่ผูกกับ use case หลักของผู้ใช้

## 3. การจัดกลุ่ม (Grouping / Packages)

จัดกลุ่ม use case ที่เกี่ยวข้องกันด้วย `package "ชื่อกลุ่ม" { ... }` ภายใน system boundary (`rectangle`) เดียว เพื่อให้อ่านง่ายเมื่อจำนวน use case มาก (เกิน ~10 ตัว)

**แนะนำให้ทำ 2 เวอร์ชันคู่กัน:**
1. **ฉบับละเอียด** — 1 use case ต่อ 1 รหัสอ้างอิงใน use case specification (UC-XX) ครบทุกตัว ใช้เป็นเอกสารอ้างอิง/ตอบคำถามเจาะลึก
2. **ฉบับภาพรวม (Top View)** — ยุบแต่ละ package ในฉบับละเอียดให้เหลือ use case ตัวแทนกลุ่มละ 1 ตัว เชื่อม actor เข้ากับ use case ตัวแทนแทนที่จะเชื่อมทีละตัว ใช้เปิดพรีเซนต์/สไลด์แรกให้กรรมการเห็นภาพรวมเร็ว ก่อนค่อยลงรายละเอียดถ้าถูกถาม

ตั้งชื่อไฟล์แยกกันชัดเจน (เช่น `<module>_usecase.puml` สำหรับฉบับละเอียด, `<module>_usecase_overview.puml` สำหรับฉบับภาพรวม) อย่ารวมสองเวอร์ชันในไฟล์เดียว

## 4. การตั้งชื่อและรหัสอ้างอิง

- รหัส use case ในไดอะแกรม (`UC01`, `UC02`, ...) ต้องตรงกับรหัสใน use case description ทุกตัวอักษร ห้ามเลขขาดหรือเกินจากเอกสารข้อความ
- ถ้ามี use case ย่อยที่แตกจาก use case หลัก (เช่น UC-17.1) ให้ตั้งชื่อ alias ในไดอะแกรมแบบตัดจุด (`UC171`) เพราะ PlantUML ไม่รับจุดในชื่อ identifier

## 5. Note และคำอธิบายเสริม

ค่าเริ่มต้น: **ไม่ใส่ note** ในไดอะแกรม เพื่อให้ภาพสะอาดอ่านง่าย โดยเฉพาะฉบับที่จะใช้ฉายในห้องสอบ — คำอธิบายเชิงเหตุผล (ทำไมไม่มี actor System, ทำไมกลุ่มนี้ครอบคลุม UC ไหนบ้าง) ให้พูดด้วยปากเปล่าหรือเก็บไว้ในเอกสารข้อความแทน

ใส่ note เฉพาะเมื่อผู้ใช้ขอ หรือมี business rule ที่ diagram เพียงอย่างเดียวสื่อสารไม่ครบและจำเป็นต้องกำกับไว้จริง ๆ

## 6. Style มาตรฐาน (Skinparam)

ใช้ชุดสีนี้เป็นค่าเริ่มต้นเพื่อความสม่ำเสมอข้ามไดอะแกรมในโครงงานเดียวกัน:

```plantuml
!theme plain
skinparam defaultFontName TH Sarabun New
skinparam defaultFontSize 14
skinparam roundcorner 10

skinparam usecase {
  BackgroundColor #FFFDE7
  BorderColor #795548
}
skinparam actor {
  BackgroundColor #E3F2FD
  BorderColor #1565C0
}
skinparam package {
  BackgroundColor #F5F5F5
  BorderColor #757575
  FontStyle bold
}
skinparam rectangle {
  BackgroundColor #FCFCFC
  BorderColor #9E9E9E
}
skinparam ArrowColor #424242
```

`left to right direction` แนะนำให้ใช้เสมอเมื่อมี actor หลายตัว เพื่อให้จัดวาง actor ทางซ้าย/ขวาของกรอบระบบเป็นธรรมชาติ

## 7. Checklist ก่อนส่งไดอะแกรม

- [ ] ทุก use case ผ่านการทดสอบ "ในฐานะ {actor} ฉันต้องการ..." ตามข้อ 0 ไม่ใช่แค่ process/business rule ที่ควรยุบรวมเข้า use case หลัก
- [ ] ไม่มี actor ชื่อ "ระบบ"/"System" (เว้นแต่เป็นระบบภายนอกจริงและตั้งชื่อเฉพาะ)
- [ ] เส้น actor–use case ทุกเส้นเป็น `--` ไม่ใช่ `-->`
- [ ] actor generalization (ถ้ามี) ใช้ `--|>` ชี้จาก actor ลูกไปหา actor แม่ ไม่ใช้ `--`/`-->`
- [ ] `<<include>>`/`<<extend>>`/generalization ทิศทางถูกต้องตามข้อ 2
- [ ] รหัส UC ในไดอะแกรมตรงกับ use case description ครบ 1:1
- [ ] จัดกลุ่มด้วย package แล้ว (ถ้า use case มากกว่า ~10 ตัว)
- [ ] ไม่มี note ค้างอยู่ (เว้นแต่ตั้งใจใส่)
