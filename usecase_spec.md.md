# Use Case Specification: แอป Focus

## Actor ทั้งหมดในระบบ
* **Guest** — ผู้ใช้ที่ยังไม่ได้สมัครสมาชิก ทำได้เพียงอย่างเดียวคือสมัครสมาชิก (UC-01)
* **สมาชิก (Member)** — ผู้ใช้ที่สมัครและล็อกอินแล้ว เข้าถึงฟีเจอร์ทั้งหมดของแอป
* **สมาชิกในห้อง (Room Member)** — สมาชิกที่เข้าร่วมห้องโฟกัสของผู้อื่น
* **ผู้ดูแลระบบ (Admin)** — ผู้ดูแลข้อมูลหลักของระบบ (ไอเทม อัตราดรอป รูปแบบอวตาร) และดูแลบัญชีผู้ใช้
* **ระบบ (System)** — กระบวนการอัตโนมัติ เช่น คำนวณ HP, ตรวจจับเบรกเกิน, สุ่มไอเทม

# UC-01 การสมัครสมาชิก

## 📄 ข้อมูลทั่วไป (General Information)
* **รหัส Use Case (ID):** UC-01
* **ชื่อ Use Case (Name):** การสมัครสมาชิก
* **ผู้กระทำหลัก (Primary Actor):** Guest
* **ผู้กระทำรอง (Secondary Actor):** ระบบฐานข้อมูลผู้ใช้
* **คำอธิบายสั้นๆ (Description):**
  Guest สร้างบัญชีผู้ใช้ใหม่เพื่อเปลี่ยนสถานะเป็นสมาชิก ก่อนจะสามารถเข้าถึงฟีเจอร์อื่นใดของแอปได้ นี่คือ use case เดียวที่ Guest ทำได้

## 🔒 เงื่อนไขของระบบ (Conditions)
* **เงื่อนไขก่อนหน้า (Pre-conditions):**
  1. ผู้ใช้ยังไม่มีบัญชีในระบบ (สถานะเป็น Guest)
  2. ผู้ใช้ติดตั้งแอปเรียบร้อยแล้ว
* **เงื่อนไขเมื่อสำเร็จ (Post-conditions):**
  1. ระบบสร้างบัญชีผู้ใช้ใหม่และบันทึกลงฐานข้อมูล
  2. สถานะผู้ใช้เปลี่ยนจาก Guest เป็นสมาชิก (Member)
  3. ระบบพาไปยัง UC-02 (สร้างอวตาร) เนื่องจากบัญชีใหม่ยังไม่มีอวตาร

## 🟢 ขั้นตอนการทำงานหลัก (Main Success Scenario / Basic Flow)
1. **Guest:** กดปุ่ม "สมัครสมาชิก" จากหน้าแรกของแอป
2. **ระบบ:** แสดงฟอร์มกรอกข้อมูล (อีเมล/เบอร์โทร, รหัสผ่าน, ชื่อผู้ใช้)
3. **Guest:** กรอกข้อมูลครบถ้วนและกดยืนยัน
4. **ระบบ:** ตรวจสอบความถูกต้องของข้อมูล (รูปแบบอีเมล, ความยาวรหัสผ่าน, ชื่อผู้ใช้ไม่ซ้ำ)
5. **ระบบ:** สร้างบัญชีใหม่ เปลี่ยนสถานะผู้ใช้เป็นสมาชิก และพาไปยังหน้าสร้างอวตาร (UC-02)

## 🔴 ขั้นตอนทางเลือกและข้อยกเว้น (Alternate Flows / Exceptions)
* **[ขั้นตอนที่ 4]a. ข้อมูลไม่ครบหรือไม่ถูกรูปแบบ:**
  * **ระบบ:** แสดงข้อความ "กรุณากรอกข้อมูลให้ถูกต้องและครบถ้วน" พร้อมระบุช่องที่ผิด
  * **ระบบ:** กลับไปที่ขั้นตอนที่ 3 ให้ Guest แก้ไข
* **[ขั้นตอนที่ 4]b. อีเมลหรือชื่อผู้ใช้ถูกใช้ไปแล้ว:**
  * **ระบบ:** แสดงข้อความ "อีเมล/ชื่อผู้ใช้นี้มีคนใช้แล้ว"
  * **ระบบ:** กลับไปที่ขั้นตอนที่ 3 ให้ Guest แก้ไข
* **[ขั้นตอนที่ 3]a. Guest กดยกเลิกกลางคัน:**
  * **ระบบ:** ล้างข้อมูลที่กรอกไว้ พากลับไปหน้าแรก
  * **ระบบ:** จบการทำงาน ผู้ใช้ยังคงสถานะ Guest

## 📝 หมายเหตุ (Notes / Special Requirements)
* **ข้อกำหนดพิเศษ (Business Rules):** รหัสผ่านต้องมีความยาวอย่างน้อย 8 ตัวอักษร; Guest ไม่สามารถเข้าถึง use case อื่นใดในระบบได้จนกว่าจะสมัครสมาชิกสำเร็จ
* **ข้อกำหนดด้าน UI/UX:** ปุ่ม "สมัครสมาชิก" ต้องเห็นได้ชัดเจนตั้งแต่หน้าแรกที่ Guest เปิดแอปครั้งแรก
* **ข้อกำหนดด้านประสิทธิภาพ (Performance):** ระบบต้องตรวจสอบและสร้างบัญชีเสร็จภายใน 2 วินาที

# UC-02 การสร้างอวตาร

## 📄 ข้อมูลทั่วไป (General Information)
* **รหัส Use Case (ID):** UC-02
* **ชื่อ Use Case (Name):** การสร้างอวตาร
* **ผู้กระทำหลัก (Primary Actor):** สมาชิก
* **ผู้กระทำรอง (Secondary Actor):** ระบบฐานข้อมูลอวตาร
* **คำอธิบายสั้นๆ (Description):**
  สมาชิกที่สมัครใหม่เลือกและสร้างอวตารเริ่มต้นของตัวเอง ก่อนเข้าใช้งานฟีเจอร์หลักของแอป

## 🔒 เงื่อนไขของระบบ (Conditions)
* **เงื่อนไขก่อนหน้า (Pre-conditions):**
  1. ผู้ใช้สมัครสมาชิกสำเร็จแล้ว (UC-01)
  2. บัญชียังไม่มีอวตาร
* **เงื่อนไขเมื่อสำเร็จ (Post-conditions):**
  1. อวตารถูกสร้างที่ HP เต็ม, เลเวล 1, ไม่มีไอเทมแต่งตัว
  2. สมาชิกเข้าสู่หน้าหลักของแอปได้

## 🟢 ขั้นตอนการทำงานหลัก (Main Success Scenario / Basic Flow)
1. **ระบบ:** แสดงหน้าเลือกรูปแบบอวตารเริ่มต้น
2. **สมาชิก:** เลือกรูปแบบอวตารที่ต้องการ
3. **สมาชิก:** กดยืนยันการสร้าง
4. **ระบบ:** บันทึกอวตารลงบัญชีของสมาชิก (HP เต็ม, เลเวล 1)
5. **ระบบ:** พาสมาชิกเข้าสู่หน้าหลักของแอป

## 🔴 ขั้นตอนทางเลือกและข้อยกเว้น (Alternate Flows / Exceptions)
* **[ขั้นตอนที่ 2]a. สมาชิกยังไม่แน่ใจ ต้องการดูตัวเลือกเพิ่มเติม:**
  * **ระบบ:** อนุญาตให้เลื่อนดูรูปแบบอวตารอื่นได้ก่อนกดยืนยัน
  * **ระบบ:** กลับไปที่ขั้นตอนที่ 2

## 📝 หมายเหตุ (Notes / Special Requirements)
* **ข้อกำหนดพิเศษ (Business Rules):** สมาชิก 1 บัญชีมีอวตารได้เพียง 1 ตัว
* **ข้อกำหนดด้าน UI/UX:** ควรมีตัวอย่างพรีวิวอวตารแบบเห็นภาพก่อนยืนยัน
* **ข้อกำหนดด้านประสิทธิภาพ (Performance):** ไม่ระบุ

# UC-03 การเพิ่มรายการสิ่งที่โฟกัส

## 📄 ข้อมูลทั่วไป (General Information)
* **รหัส Use Case (ID):** UC-03
* **ชื่อ Use Case (Name):** การเพิ่มรายการสิ่งที่โฟกัส
* **ผู้กระทำหลัก (Primary Actor):** สมาชิก
* **ผู้กระทำรอง (Secondary Actor):** ไม่มี
* **คำอธิบายสั้นๆ (Description):**
  สมาชิกเพิ่มรายการงานหรือเป้าหมายที่ต้องการโฟกัส เพื่อใช้เลือกตอนเริ่มจับเวลา

## 🔒 เงื่อนไขของระบบ (Conditions)
* **เงื่อนไขก่อนหน้า (Pre-conditions):**
  1. สมาชิกล็อกอินแล้วและมีอวตาร (UC-02)
* **เงื่อนไขเมื่อสำเร็จ (Post-conditions):**
  1. รายการใหม่ถูกบันทึกและปรากฏในลิสต์สิ่งที่โฟกัส

## 🟢 ขั้นตอนการทำงานหลัก (Main Success Scenario / Basic Flow)
1. **สมาชิก:** กดปุ่มเพิ่มรายการสิ่งที่โฟกัส
2. **ระบบ:** แสดงฟอร์มกรอกชื่อรายการ
3. **สมาชิก:** กรอกชื่อรายการและกดบันทึก
4. **ระบบ:** บันทึกรายการลงลิสต์ของสมาชิก
5. **ระบบ:** แสดงรายการที่เพิ่มใหม่ในลิสต์

## 🔴 ขั้นตอนทางเลือกและข้อยกเว้น (Alternate Flows / Exceptions)
* **[ขั้นตอนที่ 3]a. สมาชิกไม่กรอกชื่อรายการ:**
  * **ระบบ:** แสดงข้อความ "กรุณากรอกชื่อรายการ"
  * **ระบบ:** กลับไปที่ขั้นตอนที่ 3
* **[ขั้นตอนที่ 3]b. สมาชิกกดยกเลิกกลางคัน:**
  * **ระบบ:** ปิดฟอร์ม ไม่บันทึกข้อมูล กลับสู่ลิสต์เดิม

## 📝 หมายเหตุ (Notes / Special Requirements)
* **ข้อกำหนดพิเศษ (Business Rules):** ไม่จำกัดจำนวนรายการที่เพิ่มได้
* **ข้อกำหนดด้าน UI/UX:** ควรลบหรือแก้ไขรายการที่เพิ่มไปแล้วได้
* **ข้อกำหนดด้านประสิทธิภาพ (Performance):** ไม่ระบุ

# UC-04 เริ่มจับเวลาโฟกัส (ตีบอส)

## 📄 ข้อมูลทั่วไป (General Information)
* **รหัส Use Case (ID):** UC-04
* **ชื่อ Use Case (Name):** เริ่มจับเวลาโฟกัส (ตีบอส)
* **ผู้กระทำหลัก (Primary Actor):** สมาชิก
* **ผู้กระทำรอง (Secondary Actor):** ระบบจับเวลา, ระบบบอส
* **คำอธิบายสั้นๆ (Description):**
  สมาชิกเลือกรายการที่จะโฟกัสและตั้งเวลา ระบบจำลองเป็นการต่อสู้กับบอสที่มี HP ลดลงตามเวลาที่ผ่านไป

## 🔒 เงื่อนไขของระบบ (Conditions)
* **เงื่อนไขก่อนหน้า (Pre-conditions):**
  1. สมาชิกมีรายการสิ่งที่โฟกัสอย่างน้อย 1 รายการ (UC-03)
  2. ไม่มีเซสชันโฟกัสอื่นกำลังทำงานอยู่
* **เงื่อนไขเมื่อสำเร็จ (Post-conditions):**
  1. เซสชันโฟกัสอยู่ในสถานะกำลังดำเนินการ
  2. บอสถูกสร้างขึ้นด้วย HP เท่ากับเวลาที่ตั้ง (1 นาที = 100 HP)

## 🟢 ขั้นตอนการทำงานหลัก (Main Success Scenario / Basic Flow)
1. **สมาชิก:** เลือกรายการที่จะโฟกัสและตั้งระยะเวลา (หรือใช้ค่าเริ่มต้นที่ตั้งไว้ล่วงหน้าจาก UC-24)
2. **สมาชิก:** กดปุ่มเริ่มจับเวลา
3. **ระบบ:** สร้างบอสที่มี HP ตามเวลาที่ตั้ง
4. **ระบบ:** แสดงตัวเลขนาฬิกานับถอยหลัง (mm:ss) คู่กับแถบเลือดบอสตลอดเซสชัน พร้อมจุด checkpoint บนแถบเลือดทุก 5 นาที
5. **ระบบ:** ลดแถบเลือดบอสและตัวเลขนาฬิกาแบบเรียลไทม์ตามเวลาที่ผ่านไป
6. **ระบบ:** เมื่อผ่านแต่ละ checkpoint ส่งสัญญาณเบาๆ (สั่น/เสียงติ๊ก) เพื่อเสริมการรับรู้เวลา
7. **ระบบ:** ช่วง 1 นาทีสุดท้ายก่อนครบเวลา แถบเลือดเปลี่ยนเป็นจังหวะ pulse เพื่อเตือนล่วงหน้าว่าใกล้จบ
8. **ระบบ:** เมื่อเวลาครบ (บอส HP = 0) ส่งต่อไปยัง UC-06 (จบเซสชันสำเร็จ)

## 🔴 ขั้นตอนทางเลือกและข้อยกเว้น (Alternate Flows / Exceptions)
* **[ขั้นตอนที่ 5]a. สมาชิกกดค้างเพื่อยกเลิกกลางทาง:**
  * **ระบบ:** ส่งต่อไปยัง UC-05 (ยกเลิกเซสชันโฟกัสกลางทาง)
* **[ขั้นตอนที่ 1]a. สมาชิกไม่ได้เลือกรายการโฟกัส:**
  * **ระบบ:** แสดงข้อความ "กรุณาเลือกรายการที่จะโฟกัสก่อนเริ่ม"
  * **ระบบ:** กลับไปที่ขั้นตอนที่ 1
* **[ขั้นตอนที่ 6]a. สมาชิกปิดสัญญาณ checkpoint ไว้ในตั้งค่า:**
  * **ระบบ:** ข้ามการสั่น/เสียงติ๊ก แต่ยังคงแสดงจุด checkpoint บนแถบเลือดตามปกติ

## 📝 หมายเหตุ (Notes / Special Requirements)
* **ข้อกำหนดพิเศษ (Business Rules):** อัตราแปลงเวลาเป็น HP คือ 1 นาที = 100 HP; ตัวเลขนาฬิกาและแถบเลือดต้องซิงก์กันตลอดเวลา ห้ามคลาดเคลื่อน
* **ข้อกำหนดด้าน UI/UX:** ตัวเลขนาฬิกาต้องใช้ font แบบ tabular numbers เพื่อไม่ให้ตัวเลขสั่น/ขยับตอนเปลี่ยนหลัก; แถบเลือดบอสต้องมองเห็นชัดเจนตลอดเซสชัน เพื่อให้รู้สึกถึง "การต่อสู้"; checkpoint ทุก 5 นาทีต้องมองเห็นเป็นเส้นแบ่งบนแถบเลือด
* **ข้อกำหนดด้านประสิทธิภาพ (Performance):** การอัพเดตแถบเลือดและตัวเลขนาฬิกาต้องลื่นไหล ไม่กระตุก และไม่คลาดเคลื่อนจากเวลาจริงเกิน 1 วินาที

# UC-05 ยกเลิกเซสชันโฟกัสกลางทาง

## 📄 ข้อมูลทั่วไป (General Information)
* **รหัส Use Case (ID):** UC-05
* **ชื่อ Use Case (Name):** ยกเลิกเซสชันโฟกัสกลางทาง
* **ผู้กระทำหลัก (Primary Actor):** สมาชิก
* **ผู้กระทำรอง (Secondary Actor):** ระบบคำนวณรางวัลบางส่วน
* **คำอธิบายสั้นๆ (Description):**
  สมาชิกยกเลิกเซสชันโฟกัสก่อนเวลาครบกำหนด โดยกดค้างเพื่อยืนยันการยกเลิก

## 🔒 เงื่อนไขของระบบ (Conditions)
* **เงื่อนไขก่อนหน้า (Pre-conditions):**
  1. มีเซสชันโฟกัสกำลังทำงานอยู่ (UC-04)
* **เงื่อนไขเมื่อสำเร็จ (Post-conditions):**
  1. เซสชันสิ้นสุดลง ไม่นับเป็นเซสชันสำเร็จเต็มรูปแบบ
  2. บันทึกลงประวัติว่ายกเลิกกลางทาง พร้อมรางวัลบางส่วนตามสัดส่วนเวลาที่โฟกัสจริง

## 🟢 ขั้นตอนการทำงานหลัก (Main Success Scenario / Basic Flow)
1. **สมาชิก:** กดค้างที่ปุ่มยกเลิก
2. **ระบบ:** แสดงแถบยืนยันการกดค้าง
3. **สมาชิก:** กดค้างจนครบเวลายืนยัน
4. **ระบบ:** คำนวณ EXP และ HP คืนตามสัดส่วนเวลาที่โฟกัสจริงก่อนยกเลิก
5. **ระบบ:** บันทึกเซสชันลงประวัติว่ายกเลิกกลางทาง และพากลับสู่หน้าหลัก

## 🔴 ขั้นตอนทางเลือกและข้อยกเว้น (Alternate Flows / Exceptions)
* **[ขั้นตอนที่ 3]a. สมาชิกปล่อยนิ้วก่อนกดค้างครบเวลา:**
  * **ระบบ:** ยกเลิกการยืนยัน กลับไปที่เซสชันโฟกัสเดิม (UC-04) ต่อไปตามปกติ

# UC-06 จบเซสชันโฟกัสสำเร็จ

## 📄 ข้อมูลทั่วไป (General Information)

* **ชื่อ Use Case (Name):** จบเซสชันโฟกัสสำเร็จ

## 🔒 เงื่อนไขของระบบ (Conditions)
* **เงื่อนไขก่อนหน้า (Pre-conditions):**
  1. เวลาที่ตั้งไว้ครบกำหนด (บอส HP = 0) จาก UC-04
* **เงื่อนไขเมื่อสำเร็จ (Post-conditions):**
  1. หากเซสชันยาวอย่างน้อย 20 นาที สมาชิกได้รับ EXP และกล่องสุ่ม
  2. HP อวตารถูกเติมคืน และ streak อัพเดต ไม่ว่าเซสชันจะยาวเท่าใด
  3. เซสชันถูกบันทึกในประวัติว่าสำเร็จ

## 🟢 ขั้นตอนการทำงานหลัก (Main Success Scenario / Basic Flow)
1. **ระบบ:** ตรวจพบว่าบอส HP = 0 (เวลาโฟกัสครบกำหนด)

3. **ระบบ:** เพิ่ม EXP ให้อวตาร (อาจ trigger UC-14 เลเวลอัพ)
4. **ระบบ:** เติม HP อวตารคืนตามสัดส่วน และอัพเดต streak (อ้างอิง UC-16)
5. **ระบบ:** บันทึกเซสชันลงประวัติว่าสำเร็จ และเข้าสู่ช่วงพัก (UC-07)

## 🔴 ขั้นตอนทางเลือกและข้อยกเว้น (Alternate Flows / Exceptions)

  * **ระบบ:** ส่งต่อไปยัง UC-14 (อวตารเลเวลอัพ) ก่อนดำเนินการขั้นตอนถัดไป
* **[ขั้นตอนที่ 2]a. เซสชันที่จบมีระยะเวลาต่ำกว่า 20 นาที:**
  * **ระบบ:** ไม่มอบ EXP และกล่องสุ่ม พร้อมแสดงข้อความ "โฟกัสอย่างน้อย 20 นาทีเพื่อรับรางวัล"
  * **ระบบ:** ยังคงเติม HP อวตารและอัพเดต streak ตามปกติ (ขั้นตอนที่ 4)
  * **ระบบ:** บันทึกเซสชันลงประวัติว่าสำเร็จ แต่ไม่ได้รับรางวัล

## 📝 หมายเหตุ (Notes / Special Requirements)
* **ข้อกำหนดพิเศษ (Business Rules):** ต้องโฟกัสจริงอย่างน้อย 20 นาทีจึงจะได้ EXP และกล่องสุ่ม; อัตราดรอปรางวัลอิงตามความยาวเซสชัน ตามตารางใน UC-10 (ปรับลดลงจากเดิมเพราะขั้นต่ำใหม่คือ 20 นาที)

# UC-15 คำนวณพลังชีวิตอวตาร (HP Decay)

## 📄 ข้อมูลทั่วไป (General Information)
* **รหัส Use Case (ID):** UC-15
* **ชื่อ Use Case (Name):** คำนวณพลังชีวิตอวตาร (HP Decay)
* **ผู้กระทำหลัก (Primary Actor):** ระบบ
* **ผู้กระทำรอง (Secondary Actor):** สมาชิก
* **คำอธิบายสั้นๆ (Description):**
  ระบบคำนวณ HP ของอวตารตามระยะเวลาที่สมาชิกไม่ได้เปิดแอปหรือโฟกัส

## 🔒 เงื่อนไขของระบบ (Conditions)
* **เงื่อนไขก่อนหน้า (Pre-conditions):**
  1. สมาชิกเปิดแอปหลังจากไม่ได้ใช้งานช่วงหนึ่ง
  2. ไม่มีไอเทมโล่เวลาที่ยังมีผลอยู่ (UC-13)
* **เงื่อนไขเมื่อสำเร็จ (Post-conditions):**
  1. HP ของอวตารอัพเดตตามสถานะล่าสุด

## 🟢 ขั้นตอนการทำงานหลัก (Main Success Scenario / Basic Flow)
1. **ระบบ:** คำนวณเวลาที่ผ่านไปนับจาก `last_focus_completed_at`
2. **ระบบ:** เทียบเวลาที่ผ่านไปกับตารางช่วง HP ดังนี้

   | ช่วงเวลาที่ไม่เปิดแอป | สถานะ HP |
   |---|---|
   | 0-24 ชม. | Grace period, HP เต็ม 100% |
   | 1-2 วัน | โซนเตือน, HP ~70-85% |
   | 2-3 วัน | โซนวิกฤต, HP ~15-35% |
   | 3.5-4 วัน | HP = 0 |

3. **ระบบ:** อัพเดต HP ปัจจุบันของอวตารตามตาราง
4. **ระบบ:** หากอยู่ในโซนเตือนหรือวิกฤต ส่ง notification แจ้งสมาชิก

## 🔴 ขั้นตอนทางเลือกและข้อยกเว้น (Alternate Flows / Exceptions)
* **[ขั้นตอนที่ 1]a. มีไอเทมโล่เวลาที่ยังไม่หมดอายุ:**
  * **ระบบ:** ข้ามการคำนวณลด HP ทั้งหมดในช่วงที่โล่มีผล
* **[ขั้นตอนที่ 3]a. HP คำนวณได้เท่ากับ 0:**
  * **ระบบ:** ส่งต่อไปยัง UC-17 (แมวตาย/อวตารรีเซ็ต)

## 📝 หมายเหตุ (Notes / Special Requirements)
* **ข้อกำหนดพิเศษ (Business Rules):** การคำนวณ HP เป็นแบบ lazy evaluation ตอนเปิดแอป ไม่ใช้ cron job รันตลอดเวลา
* **ข้อกำหนดด้าน UI/UX:** แถบ HP ต้องแสดงสีตามโซน (เขียว/เหลือง/แดง) ให้เห็นชัดเจน
* **ข้อกำหนดด้านประสิทธิภาพ (Performance):** การคำนวณต้องเสร็จทันทีตอนเปิดแอป ไม่หน่วงการโหลดหน้าหลัก

# UC-25 ทำเครื่องหมายเป้าหมายห้องสำเร็จ

## 📄 ข้อมูลทั่วไป (General Information)
* **รหัส Use Case (ID):** UC-25
* **ชื่อ Use Case (Name):** ทำเครื่องหมายเป้าหมายห้องสำเร็จ

* **ผู้กระทำรอง (Secondary Actor):** ระบบสรุปสถิติ
* **คำอธิบายสั้นๆ (Description):**
  เจ้าของห้องปิดเป้าหมายของห้องเมื่อบรรลุผลหรือต้องการจบกิจกรรม ระบบสรุปสถิติของสมาชิกทุกคนในห้อง แล้วบันทึกลงประวัติของแต่ละคน เพื่อให้กลับมาดูย้อนหลังและรู้สึกภูมิใจในความพยายามของตัวเอง แม้จะไม่ได้อยู่อันดับ 1

## 🔒 เงื่อนไขของระบบ (Conditions)
* **เงื่อนไขก่อนหน้า (Pre-conditions):**
  1. ห้องถูกสร้างและมีสมาชิกเข้าร่วมอย่างน้อย 1 คน (UC-18, UC-19)
  2. ห้องยังไม่เคยถูกปิดมาก่อน
* **เงื่อนไขเมื่อสำเร็จ (Post-conditions):**
  1. ห้องถูกปิด ไม่สามารถเพิ่มเวลาโฟกัสในห้องนี้ต่อได้อีก
  2. สถิติของสมาชิกแต่ละคน (เวลาโฟกัสรวม, เวลาพักรวม, เวลาเบรกเกินรวม) ถูกบันทึกลงในประวัติของสมาชิกทุกคน (UC-22)

## 🟢 ขั้นตอนการทำงานหลัก (Main Success Scenario / Basic Flow)
1. **เจ้าของห้อง:** กดปุ่ม "สำเร็จเป้าหมาย" เมื่อห้องบรรลุเป้าหมายที่ตั้งไว้ หรือต้องการจบกิจกรรม
2. **ระบบ:** ยืนยันการปิดห้องกับเจ้าของห้อง
3. **ระบบ:** คำนวณสรุปสถิติของสมาชิกแต่ละคนในห้อง แยกเวลาโฟกัสและเวลาพัก/เบรกเกิน
4. **ระบบ:** บันทึกสรุปห้อง (ชื่อเป้าหมาย, สถิติสมาชิกแต่ละคน, วันที่ปิดห้อง) ลงในประวัติของสมาชิกทุกคนที่เคยเข้าร่วม (เชื่อมกับ UC-22)
5. **ระบบ:** แสดงหน้าสรุปผลให้สมาชิกทุกคนเห็นทันที ทั้งอันดับในห้องและพัฒนาการส่วนตัวของแต่ละคน
6. **ระบบ:** ปิดห้อง

## 🔴 ขั้นตอนทางเลือกและข้อยกเว้น (Alternate Flows / Exceptions)
* **[ขั้นตอนที่ 1]a. สมาชิกทั่วไป (ไม่ใช่เจ้าของห้อง) กดปุ่ม "สำเร็จเป้าหมาย":**
  * **ระบบ:** แสดงข้อความ "เฉพาะเจ้าของห้องเท่านั้นที่ปิดเป้าหมายได้"
* **[ขั้นตอนที่ 2]a. เจ้าของห้องยกเลิกการยืนยัน:**

## 📝 หมายเหตุ (Notes / Special Requirements)
* **ข้อกำหนดพิเศษ (Business Rules):** ห้องที่ปิดแล้วไม่สามารถกลับมาเปิดต่อได้ ต้องสร้างห้องใหม่หากต้องการโฟกัสร่วมกันอีก; ข้อมูลสรุปห้องต้องคงอยู่ถาวรในประวัติของสมาชิกทุกคน แม้ภายหลังจะออกจากห้องหรือเลิกเป็นเพื่อนกันแล้ว
* **ข้อกำหนดด้าน UI/UX:** หน้าสรุปผลควรเน้นพัฒนาการส่วนตัว (เช่น เทียบกับสถิติที่ผ่านมาของตัวเอง หรือป้าย "โฟกัสได้นานที่สุดของตัวเองในรอบนี้") ควบคู่กับอันดับ ไม่ใช่โชว์แค่อันดับอย่างเดียว เพื่อสร้างแรงจูงใจให้คนที่ไม่ได้อันดับ 1 ด้วย
* **ข้อกำหนดด้านประสิทธิภาพ (Performance):** การคำนวณสรุปสถิติของทุกคนในห้องต้องเสร็จภายใน 2 วินาทีก่อนแสดงหน้าสรุปผล

# UC-26 จัดการข้อมูลไอเทมและอัตราดรอป

* **รหัส Use Case (ID):** UC-26

## 🔒 เงื่อนไขของระบบ (Conditions)

## 🟢 ขั้นตอนการทำงานหลัก (Main Success Scenario / Basic Flow)

3. **ผู้ดูแลระบบ:** เพิ่มไอเทมใหม่ แก้ไขข้อมูลไอเทม ปิดใช้งานไอเทม หรือปรับค่าอัตราดรอป/สูตรผสม
4. **ระบบ:** ตรวจสอบความถูกต้อง (เช่น ผลรวมอัตราดรอปของแต่ละช่วงเซสชันต้องเท่ากับ 100%)
5. **ระบบ:** บันทึกการเปลี่ยนแปลงพร้อมประวัติการแก้ไข (ผู้แก้ไข ค่าก่อน/หลัง เวลา)

  * **ระบบ:** กลับไปที่ขั้นตอนที่ 3 ให้ผู้ดูแลระบบแก้ไข
* **[ขั้นตอนที่ 3]a. ปิดใช้งานไอเทมที่ถูกอ้างอิงในสูตรผสม:**
  * **ระบบ:** แสดงคำเตือนพร้อมรายการสูตรผสมที่ได้รับผลกระทบ ให้ผู้ดูแลระบบยืนยันก่อนดำเนินการ

## 📝 หมายเหตุ (Notes / Special Requirements)

  ผู้ดูแลระบบค้นหาบัญชีผู้ใช้ ตรวจสอบข้อมูล และระงับหรือปลดระงับบัญชีที่ใช้งานผิดเงื่อนไข เช่น ตั้งชื่อผู้ใช้หรือชื่อห้องไม่เหมาะสม

## 🔒 เงื่อนไขของระบบ (Conditions)
* **เงื่อนไขก่อนหน้า (Pre-conditions):**
  1. ผู้ดูแลระบบเข้าสู่ระบบด้วยบัญชีที่มีสิทธิ์ผู้ดูแล
* **เงื่อนไขเมื่อสำเร็จ (Post-conditions):**
  1. สถานะบัญชีเป้าหมายเปลี่ยนเป็นระงับหรือใช้งานได้ตามการดำเนินการ
  2. บัญชีที่ถูกระงับไม่สามารถเข้าสู่ระบบได้ และไม่ปรากฏในการค้นหาเพื่อน (UC-21)

## 🟢 ขั้นตอนการทำงานหลัก (Main Success Scenario / Basic Flow)
1. **ผู้ดูแลระบบ:** ค้นหาบัญชีผู้ใช้ด้วยชื่อผู้ใช้หรืออีเมล
2. **ระบบ:** แสดงข้อมูลบัญชี สถานะปัจจุบัน และประวัติการถูกระงับ (ถ้ามี)
3. **ผู้ดูแลระบบ:** เลือกระงับหรือปลดระงับบัญชี พร้อมกรอกเหตุผล
4. **ระบบ:** ขอคำยืนยันก่อนดำเนินการ
5. **ระบบ:** เปลี่ยนสถานะบัญชี บันทึกเหตุผลและเวลาดำเนินการ และแจ้งเตือนเจ้าของบัญชีทางอีเมล

## 🔴 ขั้นตอนทางเลือกและข้อยกเว้น (Alternate Flows / Exceptions)
* **[ขั้นตอนที่ 1]a. ไม่พบบัญชีที่ค้นหา:**
  * **ระบบ:** แสดงข้อความ "ไม่พบผู้ใช้นี้"
* **[ขั้นตอนที่ 3]a. ผู้ดูแลระบบไม่กรอกเหตุผล:**
  * **ระบบ:** แสดงข้อความ "กรุณาระบุเหตุผลในการดำเนินการ" และไม่ดำเนินการต่อ

## 📝 หมายเหตุ (Notes / Special Requirements)
* **ข้อกำหนดพิเศษ (Business Rules):** การระงับบัญชีต้องระบุเหตุผลเสมอ; ข้อมูลโฟกัส สถิติ และไอเทมของบัญชีที่ถูกระงับต้องไม่ถูกลบ เพื่อให้กลับมาใช้งานต่อได้เมื่อปลดระงับ
* **ข้อกำหนดด้าน UI/UX:** ควรแยกสีสถานะบัญชี (ใช้งาน/ระงับ) ให้เห็นชัดเจนในผลการค้นหา
* **ข้อกำหนดด้านประสิทธิภาพ (Performance):** ไม่ระบุ

# UC-29 เรียกดูแดชบอร์ดสถิติการใช้งานระบบ

## 📄 ข้อมูลทั่วไป (General Information)
* **รหัส Use Case (ID):** UC-29
* **ชื่อ Use Case (Name):** เรียกดูแดชบอร์ดสถิติการใช้งานระบบ
* **ผู้กระทำหลัก (Primary Actor):** ผู้ดูแลระบบ
* **ผู้กระทำรอง (Secondary Actor):** ระบบสรุปสถิติ
* **คำอธิบายสั้นๆ (Description):**
  ผู้ดูแลระบบดูภาพรวมการใช้งานของทั้งระบบ เช่น จำนวนสมาชิก จำนวนเซสชันโฟกัสต่อวัน และสถิติการดรอปไอเทมจริงเทียบกับอัตราที่ตั้งไว้ เพื่อใช้ประกอบการปรับสมดุลระบบ

## 🔒 เงื่อนไขของระบบ (Conditions)
* **เงื่อนไขก่อนหน้า (Pre-conditions):**
  1. ผู้ดูแลระบบเข้าสู่ระบบด้วยบัญชีที่มีสิทธิ์ผู้ดูแล
* **เงื่อนไขเมื่อสำเร็จ (Post-conditions):**
  1. ผู้ดูแลระบบเห็นภาพรวมการใช้งานระบบตามช่วงเวลาที่เลือก

## 🟢 ขั้นตอนการทำงานหลัก (Main Success Scenario / Basic Flow)
1. **ผู้ดูแลระบบ:** เปิดหน้าแดชบอร์ดสถิติระบบ
2. **ระบบ:** แสดงตัวกรองช่วงเวลา (วันนี้ / สัปดาห์นี้ / เดือนนี้ / กำหนดเอง)
3. **ผู้ดูแลระบบ:** เลือกช่วงเวลาที่ต้องการ
4. **ระบบ:** แสดงสถิติ ได้แก่ จำนวนสมาชิกใหม่/สมาชิกที่ยังใช้งาน, จำนวนเซสชันโฟกัสและเวลาโฟกัสรวม, อัตราการจบเซสชันสำเร็จเทียบกับยกเลิกกลางทาง, จำนวนอวตารที่หมดสติและชุบชีวิตสำเร็จ, และสถิติการดรอปไอเทมจริงเทียบกับอัตราที่กำหนดใน UC-26

## 🔴 ขั้นตอนทางเลือกและข้อยกเว้น (Alternate Flows / Exceptions)
* **[ขั้นตอนที่ 4]a. ยังไม่มีข้อมูลในช่วงเวลาที่เลือก:**
  * **ระบบ:** แสดงข้อความ "ยังไม่มีข้อมูลในช่วงเวลานี้"

## 📝 หมายเหตุ (Notes / Special Requirements)
* **ข้อกำหนดพิเศษ (Business Rules):** แดชบอร์ดเป็นข้อมูลภาพรวมเชิงสถิติ ไม่แสดงเนื้อหาส่วนตัวของสมาชิก (เช่น ชื่อรายการที่โฟกัส) เพื่อคุ้มครองความเป็นส่วนตัว
* **ข้อกำหนดด้าน UI/UX:** ควรมีกราฟแนวโน้มรายวัน/รายสัปดาห์ และไฮไลต์ค่าที่ผิดปกติ (เช่น อัตราดรอปจริงเบี่ยงเบนจากที่ตั้งไว้เกินเกณฑ์)
* **ข้อกำหนดด้านประสิทธิภาพ (Performance):** การโหลดแดชบอร์ดต้องเสร็จภายใน 3 วินาที
