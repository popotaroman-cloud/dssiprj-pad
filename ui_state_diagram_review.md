# รายงานตรวจสอบความสอดคล้อง — ui_state_diagram.puml

**ขอบเขต:** เทียบ `proposal/usecase/ui_state_diagram.puml` กับ (1) `usecase_description.md` (2) `usecase_module1-7.puml` + `usecase_overview.puml` (Use Case Diagram) (3) `activity_module*_uc*.puml` ทั้ง 27 ไฟล์ — ตรวจความครบถ้วนของ UC ทั้ง 27 ตัว และตรวจข้อความ business rule ที่ปรากฏบน transition เทียบกับเอกสารฉบับล่าสุด

**สรุปผล:** โครงสร้างหลัก (state, การไหลของ core loop, การแบ่งโซน Guest/Member/Admin) ถูกต้องและตรงกับเอกสารเกือบทั้งหมด แต่พบ **3 use case ที่หายไปทั้งหมด** (ไม่มี state/transition ใดๆ อ้างอิงถึงเลย) และ **1 จุดที่ข้อความบน transition ยังอ้างอิงกลไกเก่าที่ถูกยกเลิกไปแล้ว**

---

## ปัญหาที่พบ

### 1. UC-07 (ส่ง Feedback) หายไปทั้งหมด
ไม่มี state หรือ transition ใดในไฟล์กล่าวถึง UC-07 เลย (ตรวจด้วย grep ไม่พบ "UC-07" ในไฟล์) ทั้งที่เป็น use case จริงในโมดูล 1 (`usecase_module1.puml` บรรทัด 51) และ `activity_module1_uc07.puml` ระบุจุดเข้าใช้งานชัดเจนว่ามาจากหน้า "ตั้งค่า" (บรรทัด 28: "เปิดหน้าตั้งค่า เลือกเมนู 'ส่ง Feedback'") state `Settings` (บรรทัด 67) ของ ui_state_diagram ควรมี transition ไปยัง state ใหม่ (เช่น `Feedback`) แต่ปัจจุบันไม่มี

### 2. UC-23 (ดูโปรไฟล์และสถิติเพื่อน) หายไปทั้งหมด
ไม่มี state หรือ transition ใดอ้างถึง UC-23 เลย state `Friends` (บรรทัด 82, 137-138) มีป้ายกำกับเฉพาะ "(UC-21, UC-22)" เท่านั้น ทั้งที่ `activity_module5_uc23.puml` ระบุชัดว่าเป็นหน้าจอแยกต่างหาก (เปิดลิสต์เพื่อน → กดเลือกเพื่อน → หน้าโปรไฟล์เพื่อน) ควรเพิ่ม state เช่น `FriendProfile` พร้อม transition `Friends --> FriendProfile`

### 3. UC-27 (ผู้ดูแลดูและตอบกลับ Feedback) หายไปทั้งหมด + หมายเหตุคลาดเคลื่อน
โซน Admin (บรรทัด 144-153) มีเพียง `ItemMgmt` (UC-25) และ `AvatarMgmt` (UC-26) ไม่มี state สำหรับ UC-27 เลย และ note ท้ายไฟล์ (บรรทัด 168-172) เขียนว่า "...และเป็นไปได้ที่จะมี UC-27+ (Feedback/อื่น ๆ) ขึ้นอยู่กับการนำไปใช้จริง" ซึ่งเขียนราวกับ UC-27 เป็นเพียงความเป็นไปได้ในอนาคต — แต่ความจริง UC-27 ถูกกำหนดไว้แน่นอนแล้วในระดับเดียวกับ UC-25/26 ทั้งใน `usecase_module7.puml` (บรรทัด "UC-27\nดูและตอบกลับ Feedback") และมี activity diagram ครบ (`activity_module7_uc27.puml`) ควรเพิ่ม state `FeedbackMgmt` และแก้ note ให้ตรงกับความเป็นจริงว่า UC-27 มีอยู่แน่นอน ไม่ใช่ตัวเลือกที่ "เป็นไปได้"

### 4. Transition `RoomView → Session` (บรรทัด 130) ใช้ข้อความกลไกเก่าที่ถูกยกเลิกแล้ว
ข้อความบน transition คือ "เริ่มโฟกัสในห้อง\n(**นับรวม HP บอสกลุ่ม**)" แต่ตาม Business Rules ของ UC-17 ในเอกสารฉบับล่าสุด (usecase_description.md บรรทัด 334) ระบุชัดว่า:
> "**ไม่มีบอสกลุ่ม/HP ส่วนกลางอีกต่อไป:** ตัดระบบ Dynamic Boss HP และสูตรดาเมจส่วนกลางแบบเดิมออกทั้งหมด แต่ละเซสชันของแต่ละคนยังคงจำลองเป็นการตีบอสส่วนตัวตามกลไกเดิมของ UC-08 ... ไม่มีการรวม HP หรือดาเมจข้ามคนอีกต่อไป"

กลไกถูกเปลี่ยนจาก "Cooperative Boss HP" เป็น "Time-boxed Competition" (แข่งสะสมเวลาโฟกัสของแต่ละคนแยกกัน) ไปแล้ว ข้อความ "นับรวม HP บอสกลุ่ม" บนไดอะแกรมจึงขัดแย้งกับกฎปัจจุบันโดยตรง ควรแก้เป็นข้อความทำนอง "เริ่มโฟกัสในห้อง (ตีบอสส่วนตัว UC-08 — ไม่มี HP รวมกลุ่ม, แข่งสะสมเวลาแทน)"

## จุดสังเกตเล็กน้อย (ไม่ใช่ข้อผิดพลาด)

- **CreateAvatar อยู่ใน composite state "ยังไม่เข้าสู่ระบบ" (PreAuth)** (บรรทัด 46-61): ตาม Post-condition ของ UC-01 ผู้ใช้เปลี่ยนสถานะเป็น "สมาชิก" ทันทีหลังสมัครสำเร็จ (ก่อนสร้างอวตารเสร็จด้วยซ้ำ) การจัดกลุ่ม `CreateAvatar` ไว้ในโซน "ยังไม่เข้าสู่ระบบ" จึงเป็นมุมมองเชิง UI-flow (ยังไม่เข้าสู่แอปหลัก) มากกว่าเชิงสถานะบัญชี ไม่ใช่ข้อผิดพลาด แต่ควรระวังไม่ให้สับสนกับคำว่า "Guest" ในคอมเมนต์หัวไฟล์ (บรรทัด 45)
- คอมเมนต์หัวโซน PreAuth (บรรทัด 45: "Guest — M1/E1, M2/E3") ไม่ได้กล่าวถึง UC-05 (M1/E3 — ลืมรหัสผ่าน) ทั้งที่ state `ForgotPassword` ใช้งานอยู่จริงในโซนนี้ (บรรทัด 50, 59-60) เป็นความไม่ครบของคอมเมนต์อธิบาย ไม่กระทบตัวไดอะแกรม

## จุดที่ตรวจแล้วตรงกันดี

UC-01–06, UC-08–22, UC-24–26 มี state/transition ครบและคำอธิบาย (ตัวเลข, เงื่อนไข, ชื่อ UC) ตรงกับ usecase_description.md, usecase diagram, และ activity diagram — รวมถึงจุดที่เคยแก้ไขในรอบตรวจรีวิวก่อนหน้า เช่น นิยาม "เวลาโฟกัสสะสม" ของ UC-18 (นับเฉพาะเซสชันจบสำเร็จ), การโอนสิทธิ์เจ้าของห้องอัตโนมัติของ UC-19, HP decay 2 หน่วย/ชม. และสูตร EXP ของ UC-08
