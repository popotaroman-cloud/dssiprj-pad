## M1 · บัญชีและการเริ่มต้นใช้งาน

### E1 · การสมัครสมาชิกและสร้างอวตาร

#### UC-01 · การสมัครสมาชิก

| หัวข้อ | รายละเอียด |
|-------|-----------|
| **รหัสอ้างอิง** | FN-ACC-001 |
| **Actor หลัก** | Guest |
| **Actor รอง** | ระบบฐานข้อมูลผู้ใช้ |
| **คำอธิบายโดยย่อ** | Guest สร้างบัญชีผู้ใช้ใหม่เพื่อเปลี่ยนสถานะเป็นสมาชิก ก่อนจะสามารถเข้าถึงฟีเจอร์อื่นใดของแอปได้ นี่คือ use case เดียวที่ Guest ทำได้ |
| **เงื่อนไขก่อนเริ่มทำงาน (Pre-condition)** | ผู้ใช้ยังไม่มีบัญชีในระบบ (สถานะเป็น Guest) และติดตั้งแอปเรียบร้อยแล้ว |
| **ลำดับเหตุการณ์หลัก (Basic Flow)** | 1. Guest กดปุ่ม "สมัครสมาชิก"<br>2. ระบบแสดงฟอร์มกรอกข้อมูล<br>3. Guest กรอกข้อมูลครบถ้วนและกดยืนยัน<br>4. ระบบตรวจสอบความถูกต้องของข้อมูล<br>5. ระบบสร้างบัญชีใหม่ เปลี่ยนสถานะเป็นสมาชิก และพาไปหน้าสร้างอวตาร |
| **ลำดับเหตุการณ์ทางเลือก (Alternative/Exception Flow)** | ข้อมูลไม่ครบ/ไม่ถูกรูปแบบ → แจ้งเตือนให้แก้ไข; อีเมล/ชื่อผู้ใช้ถูกใช้แล้ว → แจ้งเตือนให้แก้ไข; กดยกเลิกกลางคัน → ล้างข้อมูล กลับหน้าแรก ยังเป็น Guest |
| **เงื่อนไขหลังทำงานสำเร็จ (Post-condition)** | สร้างบัญชีใหม่และบันทึกลงฐานข้อมูล สถานะเปลี่ยนจาก Guest เป็นสมาชิก และพาไปยัง UC-02 |
| **ความสัมพันธ์กับ Use Case อื่น** | `<<include>>` UC-02 (บัญชีใหม่ทุกบัญชีต้องสร้างอวตารเสมอ) |
| **เกี่ยวข้องกับ Service/Endpoint อื่น** | - |

| หัวข้อ | รายละเอียด |
|-------|-----------|
| **รหัสอ้างอิง** | FN-ACC-002 |

| **Actor รอง** | ระบบฐานข้อมูลอวตาร |
| **คำอธิบายโดยย่อ** | สมาชิกที่สมัครใหม่เลือกและสร้างอวตารเริ่มต้นของตัวเอง ก่อนเข้าใช้งานฟีเจอร์หลักของแอป |
| **เงื่อนไขก่อนเริ่มทำงาน (Pre-condition)** | สมัครสมาชิกสำเร็จแล้ว (UC-01) และบัญชียังไม่มีอวตาร |

### E3 · การจัดการบัญชีและการตั้งค่า

#### UC-33 · แก้ไขโปรไฟล์และเปลี่ยนรหัสผ่าน

| หัวข้อ | รายละเอียด |
|-------|-----------|
| **รหัสอ้างอิง** | FN-ACC-004 |
| **Actor หลัก** | สมาชิก |
| **Actor รอง** | ระบบยืนยันตัวตน (Auth Service) |
| **คำอธิบายโดยย่อ** | สมาชิกแก้ไขข้อมูลโปรไฟล์ (ชื่อผู้ใช้แสดงผล, รูปประจำตัว) และเปลี่ยนรหัสผ่านขณะยังล็อกอินอยู่ (คนละกรณีกับการลืมรหัสผ่านตอนเข้าระบบไม่ได้ ดู UC-38) |
| **เงื่อนไขก่อนเริ่มทำงาน (Pre-condition)** | สมาชิกล็อกอินอยู่ (UC-31) |

| **ลำดับเหตุการณ์ทางเลือก (Alternative/Exception Flow)** | ชื่อผู้ใช้ถูกใช้แล้ว → แจ้งเตือนให้แก้ไข; รหัสผ่านเดิมไม่ถูกต้อง → แจ้งเตือน ไม่บันทึก; รหัสผ่านใหม่ไม่ตรงกับช่องยืนยัน → แจ้งเตือน |
| **เงื่อนไขหลังทำงานสำเร็จ (Post-condition)** | ข้อมูลโปรไฟล์/รหัสผ่านอัพเดต หากเปลี่ยนรหัสผ่าน อุปกรณ์อื่นที่เคยล็อกอินไว้จะถูกบังคับออกจากระบบ (ต้องล็อกอินใหม่ตาม UC-31) |
| **ความสัมพันธ์กับ Use Case อื่น** | ไม่มี |

#### UC-38 · ลืมรหัสผ่าน (Forgot Password / Reset Password)

| หัวข้อ | รายละเอียด |
|-------|-----------|
| **รหัสอ้างอิง** | FN-ACC-005 |
| **Actor หลัก** | Guest (สมาชิกที่เข้าสู่ระบบไม่ได้) |
| **Actor รอง** | ระบบยืนยันตัวตน (Auth Service), ระบบส่งอีเมล |
| **คำอธิบายโดยย่อ** | ผู้ใช้ที่มีบัญชีอยู่แล้วแต่จำรหัสผ่านไม่ได้ ขอรีเซ็ตรหัสผ่านผ่านอีเมลที่ลงทะเบียนไว้ เพื่อกลับเข้าสู่ระบบได้อีกครั้ง คนละกรณีกับ UC-33 ที่เป็นการเปลี่ยนรหัสผ่านขณะยังล็อกอินอยู่ |
| **เงื่อนไขก่อนเริ่มทำงาน (Pre-condition)** | มีบัญชีในระบบแล้ว (UC-01) และไม่สามารถเข้าสู่ระบบด้วยรหัสผ่านเดิมได้ (ไม่จำเป็นต้องล็อกอินอยู่) |
| **ลำดับเหตุการณ์หลัก (Basic Flow)** | 1. ที่หน้าล็อกอิน (UC-31) กด "ลืมรหัสผ่าน"<br>2. ระบบแสดงฟอร์มกรอกอีเมล/ชื่อผู้ใช้ที่ลงทะเบียนไว้<br>3. กรอกอีเมลและกดยืนยัน<br>4. ระบบส่งลิงก์/รหัส OTP รีเซ็ตรหัสผ่านไปยังอีเมลที่ลงทะเบียน (มีอายุจำกัด เช่น 15 นาที)<br>5. เปิดลิงก์/กรอกรหัส OTP แล้วตั้งรหัสผ่านใหม่พร้อมกรอกยืนยันรหัสผ่านใหม่<br>6. ระบบบันทึกรหัสผ่านใหม่ เพิกถอน token เดิมในทุกอุปกรณ์ และพากลับไปหน้าล็อกอิน (UC-31) |
| **ลำดับเหตุการณ์ทางเลือก (Alternative/Exception Flow)** | อีเมลไม่ตรงกับบัญชีใดในระบบ → ระบบแสดงข้อความทั่วไปว่า "หากอีเมลนี้มีอยู่ในระบบ จะได้รับลิงก์รีเซ็ตรหัสผ่าน" โดยไม่ยืนยัน/ปฏิเสธการมีอยู่ของบัญชี (ป้องกัน account enumeration); ลิงก์/OTP หมดอายุหรือถูกใช้ไปแล้ว → แจ้งเตือนให้ขอลิงก์ใหม่; รหัสผ่านใหม่ไม่ตรงกับช่องยืนยัน หรือไม่ผ่านเงื่อนไขความปลอดภัยของรหัสผ่าน → แจ้งเตือน ไม่บันทึก; กดยกเลิกกลางทาง → กลับไปหน้าล็อกอินตามเดิม บัญชียังใช้รหัสผ่านเดิม; บัญชีถูกระงับ (UC-28) → แจ้งเตือนว่าบัญชีถูกระงับ ไม่อนุญาตให้รีเซ็ตรหัสผ่าน |
| **เงื่อนไขหลังทำงานสำเร็จ (Post-condition)** | รหัสผ่านของบัญชีถูกเปลี่ยนเป็นค่าใหม่ อุปกรณ์อื่นที่เคยล็อกอินไว้ถูกบังคับออกจากระบบ (ต้องล็อกอินใหม่ตาม UC-31) ผู้ใช้กลับเข้าสู่ระบบได้ด้วยรหัสผ่านใหม่ |
| **ความสัมพันธ์กับ Use Case อื่น** | เกี่ยวข้องกับ UC-31 (จุดเริ่มต้นและปลายทางของ flow); ผลลัพธ์ปลายทางเดียวกับการเปลี่ยนรหัสผ่านใน UC-33 แต่ actor และ pre-condition ต่างกัน |
| **เกี่ยวข้องกับ Service/Endpoint อื่น** | - |

#### UC-37 · ตั้งค่าระยะเวลาเริ่มต้นของโฟกัส/พัก (Default Duration)

| **ความสัมพันธ์กับ Use Case อื่น** | ไม่มี (ส่งผลต่อ UC-04 เชิงข้อมูล pre-fill เท่านั้น ไม่ใช่ include/extend เชิง flow) |
| **เกี่ยวข้องกับ Service/Endpoint อื่น** | - |

## M2 · เซสชันโฟกัส (Core Loop)

### E1 · วงจรเซสชันโฟกัส

#### UC-04 · เริ่มจับเวลาโฟกัส (ตีบอส)

| หัวข้อ | รายละเอียด |
|-------|-----------|
| **รหัสอ้างอิง** | FN-SES-001 |
| **Actor หลัก** | สมาชิก |
| **Actor รอง** | ระบบจับเวลา, ระบบบอส |
| **คำอธิบายโดยย่อ** | สมาชิกเลือกรายการที่จะโฟกัสและตั้งเวลาเริ่มจับเวลา ระบบจะจำลองเป็นการต่อสู้กับบอส ครอบคลุมไปถึงการพัก การยกเลิก และการแจกรางวัล |
| **เงื่อนไขก่อนเริ่มทำงาน (Pre-condition)** | สมาชิกล็อกอิน มีอวตาร และมีรายการสิ่งที่โฟกัส (UC-03) |
| **ลำดับเหตุการณ์หลัก (Basic Flow)** | 1. เลือกรายการจาก UC-03 และตั้งระยะเวลาโฟกัส/พักส่วนตัว (ระบบ pre-fill ค่าเริ่มต้นจาก UC-37 ถ้าเคยตั้งไว้ สามารถแก้ไขเป็นรายเซสชันได้เสมอ — ยกเว้นกรณีเริ่มเซสชันขณะอยู่ในห้องโฟกัส ให้ใช้ระยะเวลาร่วมของห้องที่เจ้าของห้องตั้งไว้แทนโดยแก้ไขเป็นรายเซสชันไม่ได้ ดู Business Rules ของ UC-18)<br>2. กดเริ่มจับเวลา ระบบจำลองบอส (HP = เวลา × 100)<br>3. แสดงนาฬิกาถอยหลัง เมื่อเวลาครบ เซสชันสำเร็จ คืน HP อวตารเต็ม 100% (ดูกฎ 'การคืน HP อวตารเมื่อเซสชันสำเร็จ' ด้านล่าง) และบันทึกประวัติ<br>4. ได้รับรางวัล EXP ตามจำนวนนาทีที่โฟกัสสำเร็จ (ทุกเซสชันที่จบสำเร็จ ไม่มีขั้นต่ำ — ดูสูตรใน Business Rules) และหากเซสชัน ≥ 20 นาที จะได้รับกล่องสุ่มเพิ่มอีก 1 ใบเก็บเข้ากระเป๋าด้วย (ชนิดกล่องถูกกำหนดตายตัวตามความยาวเซสชัน ณ ตอนนี้ — ดู Business Rules ของ UC-10) **ยังเปิดกล่องไม่ได้ทันที ต้องไปเปิดจากกระเป๋าเท่านั้นผ่าน UC-10**<br>5. ระบบเริ่มนับเวลาพักอัตโนมัติตามเวลาที่ตั้งไว้<br>6. เมื่อจบช่วงพักสามารถเริ่มเซสชันใหม่ได้ หากไม่เริ่ม ระบบจะนับเวลาพักเกินกำหนดเป็นสถิติ |
| **ลำดับเหตุการณ์ทางเลือก (Alternative/Exception Flow)** | **ยกเลิกกลางทาง:** กดยกเลิกเซสชันกลางทาง (กดค้าง) → คืน HP ตามสัดส่วนเวลา บันทึกว่ายกเลิก ไม่ได้รับกล่องสุ่ม<br>**พักเกินกำหนด:** หมดเวลาพักแต่ยังไม่เริ่มใหม่ → ระบบส่ง push notification แจ้งว่าเวลาพักหมดแล้ว (แม้แอปอยู่ background) จากนั้นเริ่มนับเวลาพักเกินไปเรื่อยๆ เพื่อเป็นสถิติ<br>**จบช่วงพักก่อนกำหนด:** กดข้ามช่วงพัก → เริ่มเซสชันใหม่ได้ทันที<br>**อวตารหมดสติ:** HP อวตารเท่ากับ 0 ก่อนเริ่ม → เข้าสถานะ "หมดสติ" ล็อกฟีเจอร์จนกว่าจะชุบชีวิต (UC-30) |
| **เงื่อนไขหลังทำงานสำเร็จ (Post-condition)** | เซสชันถูกบันทึกลงประวัติ สถิติเวลาพัก/เบรกเกิน รางวัล และ HP อัพเดตตามกฎทางธุรกิจ |
| **ความสัมพันธ์กับ Use Case อื่น** | ถูก `<<include>>` โดย UC-30 |
| **กฎทางธุรกิจที่เกี่ยวข้อง (Business Rules)** | **โหมดโฟกัสคนเดียว (Single-player Focus):**<br>- การนับเวลาจับเวลาของอวตารนับถอยหลังแบบเรียลไทม์ (Real-time countdown) สมาชิกเห็นเวลาลดลงวินาทีต่อวินาทีบนหน้าจอ<br>- การคำนวณหักล้างค่าพลังชีวิต (HP) ของบอสจะไม่หักทุกวินาที แต่ระบบจะหักลดเลือดบอสทุกๆ 5 นาทีที่ผู้ใช้โฟกัสสำเร็จ (ส่งดาเมจไปหักเลือดบอสเป็นก้อนทุกๆ 5 นาที)<br>- *ข้อเสนอแนะ UX (ไม่ใช่ business logic):* ฝั่ง client ควรแสดงแถบเลือดบอสไหลลดลงต่อเนื่องแบบ interpolate ระหว่าง 2 จุดที่หักจริง แทนการกระโดดเป็นก้อนทุก 5 นาที เพื่อให้สอดคล้องกับนาฬิกาถอยหลังที่เดินทุกวินาที<br><br>**Max HP และ HP Decay — ยืนยันแล้วโดยทีมออกแบบเกม (แทนที่ค่าตั้งต้นเดิม):**<br>- **Base Max HP:** อวตารเริ่มต้นที่ Max HP = 100 หน่วย ที่เลเวล 1 (ตรงกับ Post-condition ของ UC-02)<br>- **Level Scaling:** ทุกครั้งที่เลเวลอัพ (ตามสูตร EXP ด้านล่าง) Max HP เพิ่มขึ้น +5 หน่วยทันที โดย Current HP ไม่เปลี่ยนแปลง (ไม่มีการฟื้นฟู HP จากการเลเวลอัพ)<br>- **HP Decay Rate:** อวตารเสีย HP 2 หน่วย ต่อทุก 1 ชั่วโมงที่สมาชิกไม่ได้อยู่ในเซสชันโฟกัส นับจากเวลาสิ้นสุดเซสชันล่าสุด (หรือเวลาสร้างอวตารถ้ายังไม่เคยโฟกัสเลย) — ปรับจากค่าตั้งต้นเดิม 5 หน่วย/ชั่วโมง<br>- ไม่มีการหัก HP ระหว่างที่ไอเทมโล่เวลา (UC-13) มีผลอยู่<br>- เมื่อ HP อวตารแตะ 0 → เข้าสถานะ "หมดสติ" ตามลำดับเหตุการณ์ทางเลือกด้านบน (กลไกหมดสติ/ชุบชีวิตยังคงเป็นไปตาม UC-30 แบบเดิมทุกประการ ไม่มีการเปลี่ยนแปลง)<br><br>**สูตร EXP และเลเวลอัพ — ยืนยันแล้ว (ดูหมายเหตุข้อ 16):**<br>- ได้รับ EXP = จำนวนนาทีที่โฟกัสสำเร็จ × 2 ต่อเซสชัน (เฉพาะเซสชันที่จบสำเร็จ ไม่นับเซสชันที่ยกเลิกกลางทาง)<br>- เลเวลอัพเมื่อ EXP สะสมถึงเกณฑ์ของเลเวลถัดไป (ตัวอย่างสูตรขั้นบันได: เลเวล N ต้องใช้ N × 100 EXP สะสม) เอกสารนี้ยังไม่กำหนดเพดานเลเวลสูงสุด — เลเวลอัพส่งผลต่อ Max HP แล้วตามกฎ Level Scaling ด้านบน (ยืนยันแล้ว ไม่ใช่ค่าตั้งต้นอีกต่อไป)<br><br>**การคืน HP อวตารเมื่อเซสชันสำเร็จ (Session Success Heal) — ยืนยันแล้ว (ดูหมายเหตุข้อ 16):**<br>- อวตารได้รับการคืน HP เต็ม (เท่ากับ Max HP ปัจจุบันของอวตาร ซึ่งอาจมากกว่า 100 หากเลเวลอัพแล้ว) ทุกครั้งที่จบเซสชันสำเร็จ ไม่ว่าเซสชันจะสั้นหรือยาวแค่ไหน ตราบใดที่โฟกัสจนครบเวลาที่ตั้งไว้<br>- ไม่มีผลกับเซสชันที่ถูกยกเลิกกลางทาง (ดู alt flow "ยกเลิกกลางทาง") |

| **ความสัมพันธ์กับ Use Case อื่น** | ไม่มี (ใช้ข้อมูลไอเทมร่วมกับ UC-23); ถูกอ้างอิงจาก UC-39 (แสดงไอเทมที่สวมใส่อยู่บนโปรไฟล์เพื่อน) |
| **เกี่ยวข้องกับ Service/Endpoint อื่น** | - |



## M4 · อวตารและความต่อเนื่อง (ระบบชุบชีวิต)

> UC-14 (เลเวลอัพ), UC-15 (HP Decay), UC-17 (หมดสติ) เดิม ถูกยุบรวมเป็นกฎทางธุรกิจของ UC-04 (โมดูล 2) แล้ว ส่วน UC-16 (Streak) เดิม ถูกยุบรวมเป็นกฎทางธุรกิจของ UC-31 (โมดูล 2) โดยเฉพาะ (แก้คำอธิบายให้ตรงจุดในรอบตรวจรีวิวที่ 14 — เดิมเขียนรวมว่า "UC-04/UC-31" ทำให้เข้าใจผิดว่ากฎ Streak มีส่วนอยู่ใน UC-04 ด้วย ทั้งที่จริงอยู่ใน UC-31 เท่านั้น) เพราะทั้งหมดไม่มี actor ริเริ่มเอง — เหลือ UC-30 (เดิมชื่อ UC-17.1 ก่อนเปลี่ยนรหัสเพื่อไม่ให้ชนกับเลข 17 ที่ถูกยุบไปแล้ว) เป็น use case เดียวในโมดูลนี้ที่สมาชิกลงมือทำเองมีเป้าหมายชัดเจน

| **เกี่ยวข้องกับ Service/Endpoint อื่น** | - |

> **หมายเหตุจากการรีวิว (แก้ไขแล้ว):** เดิมข้อเสนอแนะนี้เตือนว่าการรีเซ็ตเลเวล/EXP กลับศูนย์ทั้งหมดเป็น penalty ที่แรงเกินไปในเชิง game design และแนะนำให้ลดโทษเป็นลดเลเวลบางส่วนแทน — ทีมออกแบบเกมยืนยันแล้วให้เปลี่ยนเป็น **ลดเลเวลลง 2 ระดับ** ตามที่ระบุใน Business Rules ด้านบน จึงถือว่าข้อเสนอแนะนี้ถูกรับไปปรับใช้แล้ว

| **ความสัมพันธ์กับ Use Case อื่น** | `<<include>>` UC-22 (เมื่อปิดห้องสำเร็จ); เกี่ยวข้องกับ UC-34 (ออกจากห้อง) และ UC-36 (เตะสมาชิก) ในเชิงผลกระทบต่อจำนวนสมาชิกในห้อง |
| **กฎทางธุรกิจที่เกี่ยวข้อง (Business Rules)** | **โหมดโฟกัสร่วมกันในห้อง (Co-working/Group Focus):**<br>- มีจำนวนสมาชิกสูงสุดได้ไม่เกิน 8 คนต่อห้อง<br>- **สูตรคำนวณเลือดบอสกลุ่ม (Dynamic Boss HP):** ระบบจะไม่ลด HP สูงสุดของบอสย้อนหลัง เพื่อไม่ให้ดาเมจที่ตีสะสมไปแล้วเกิน HP ใหม่จนบอสตายผิดเงื่อนไข<br>- **ระยะเวลาโฟกัส/พักร่วมของห้อง:** ทุกคนในห้องใช้ระยะเวลาโฟกัสและเวลาพักค่าเดียวกันซึ่งเจ้าของห้องเป็นผู้กำหนด/แก้ไขเท่านั้น (สมาชิกคนอื่นตั้งระยะเวลาของตัวเองแยกไม่ได้ขณะเริ่มเซสชันในห้องนี้ ค่าจาก UC-37 ของแต่ละคนถูกแทนที่ด้วยค่าร่วมของห้องชั่วคราว) — แต่สมาชิกแต่ละคนกดเริ่มเซสชันโฟกัสของตัวเองได้อย่างอิสระตามเวลาที่ตนสะดวก (ไม่จำเป็นต้องเริ่มหรือจบพร้อมกัน ใช้ระยะเวลาเดียวกันแค่คนละช่วงเวลาเริ่ม)<br>- **สูตรดาเมจต่อพลังโจมตีส่วนกลาง — เพิ่มในรอบตรวจรีวิวที่ 16:** ดาเมจที่สมาชิกแต่ละคนทำได้จากเซสชันหนึ่งๆ = จำนวนนาทีที่โฟกัสสำเร็จของเซสชันนั้น × 100 (อัตราเดียวกับโหมดเดี่ยวใน UC-04) นำมารวมกันเป็นพลังโจมตีส่วนกลางหักลบ HP บอสกลุ่ม — หน้าจอห้องแสดงความคืบหน้าดาเมจแบบเรียลไทม์ระหว่างที่แต่ละคนกำลังโฟกัสอยู่เพื่อเป็นตัวอย่างความคืบหน้าเท่านั้น แต่ดาเมจจะถูกล็อกเข้ากับ HP บอสกลุ่มจริงก็ต่อเมื่อเซสชันนั้นจบสำเร็จเท่านั้น หากสมาชิกคนใดยกเลิกเซสชันกลางทาง (ตาม alt flow "ยกเลิกกลางทาง" ของ UC-04) ดาเมจของเซสชันนั้นทั้งหมดจะกลายเป็น 0 ทันที (ไม่นับแม้แต่ส่วนที่โฟกัสไปแล้วก่อนยกเลิก) ไม่กระทบดาเมจจากเซสชันก่อนหน้าที่จบสำเร็จไปแล้วของสมาชิกคนนั้นหรือของคนอื่นในห้อง<br>- ในช่วงเวลาที่สมาชิกคนใดอยู่ใน "ช่วงเวลาพักส่วนตัว" หรือ "พักเกินกำหนด" ระบบจะหยุดคำนวณพลังโจมตีของสมาชิกคนนั้นชั่วคราว โดยไม่มีผลกระทบต่อเพื่อนคนอื่นในห้องที่ยังโฟกัสอยู่<br>- พลังโจมตีรวมจากสมาชิกทุกคนจะลด HP บอสกลุ่มได้สูงสุดไม่เกินค่า HP ตั้งต้นของบอสตัวนั้นๆ (ไม่เกินเป้าหมายรวมของห้อง) เมื่อบอสเลือดเหลือ 0 ระบบจะปัดเศษและหยุดหักเลือดทันที |

| **ลำดับเหตุการณ์หลัก (Basic Flow)** | 1. เปิดหน้าประวัติ/สถิติ<br>2. ระบบแสดงตัวกรองช่วงเวลา (รวมตัวเลือก "ทั้งหมด" สำหรับดูสถิติสะสมทุกช่วงเวลาตั้งแต่เริ่มใช้งาน)<br>3. เลือกช่วงเวลาที่ต้องการดู หรือเลือก "ทั้งหมด"<br>4. สรุปเวลาโฟกัส/พัก/เบรกเกินรวมในช่วงที่เลือก (หรือทั้งหมด) แยกออกจากกันชัดเจน<br>5. แสดงรายละเอียดแยกตามแต่ละรายการที่โฟกัส (UC-03) โดยแต่ละรายการก็แสดงเวลาโฟกัส/พัก/เบรกเกินแยกกันเช่นเดียวกับข้อ 4<br>6. แสดงรายการเซสชันย้อนหลังพร้อมสถานะ รวมสรุปห้องที่เคยปิดเป้าหมายแล้ว (UC-18) |
| **ลำดับเหตุการณ์ทางเลือก (Alternative/Exception Flow)** | ยังไม่มีข้อมูลในช่วงเวลาที่เลือก → แจ้งเตือน; แตะรายการใดรายการหนึ่ง → แสดงรายละเอียดเซสชันย่อยของรายการนั้น |
| **เงื่อนไขหลังทำงานสำเร็จ (Post-condition)** | สมาชิกเห็นภาพรวมพฤติกรรมการโฟกัสของตัวเอง ทั้งเชิงเวลาและเชิงรายการ |
| **ความสัมพันธ์กับ Use Case อื่น** | ถูก `<<include>>` จาก UC-18 (โมดูล 5); ถูกอ้างอิงจาก UC-39 (นิยามสถิติโฟกัสที่แสดงบนโปรไฟล์เพื่อน) |
| **กฎทางธุรกิจที่เกี่ยวข้อง (Business Rules)** | **นิยามตัวเลข "เวลาโฟกัสรวม" ในสรุปสถิติ — เพิ่มในรอบตรวจรีวิวที่ 19:** ตัวเลขสรุป "เวลาโฟกัสรวม" ในขั้นตอนที่ 4 และ 5 นับเฉพาะนาทีจากเซสชันที่จบสำเร็จเท่านั้น (เกณฑ์เดียวกับ EXP ใน UC-04, ดาเมจใน UC-18 และ Ranking ใน UC-20) ส่วนรายการเซสชันย้อนหลังในขั้นตอนที่ 6 ยังคงแสดงเซสชันที่ถูกยกเลิกด้วยพร้อมสถานะ "ยกเลิก" กำกับไว้ เพื่อความโปร่งใส เพียงแต่เวลาของเซสชันเหล่านั้นไม่ถูกรวมเข้าตัวเลขสรุป |
| **เกี่ยวข้องกับ Service/Endpoint อื่น** | - |

### E2 · การจัดการบัญชีผู้ใช้

#### UC-28 · ระงับ/เพิกถอนสิทธิ์บัญชีผู้ใช้ (Suspend/Ban Account)

| หัวข้อ | รายละเอียด |
|-------|-----------|
| **รหัสอ้างอิง** | FN-ADM-003 |
| **Actor หลัก** | ผู้ดูแลระบบ |
| **Actor รอง** | สมาชิกที่ถูกระงับ, ระบบยืนยันตัวตน (Auth Service) |
| **คำอธิบายโดยย่อ** | ผู้ดูแลระบบระงับหรือยกเลิกการระงับบัญชีสมาชิกที่ทำผิดกฎ ทำให้บัญชีนั้นเข้าสู่ระบบ (UC-31) หรือรีเซ็ตรหัสผ่าน (UC-38) ไม่ได้จนกว่าจะถูกยกเลิกการระงับ (use case นี้ถูกอ้างอิงจาก UC-31 และ UC-38 มาตั้งแต่ต้น แต่ไม่เคยถูกเขียนไว้จริงเพราะไฟล์ต้นฉบับขาดหายไปตั้งแต่จุดนี้ — เพิ่มเข้ามาในรอบตรวจรีวิวที่ 7) |
| **เงื่อนไขก่อนเริ่มทำงาน (Pre-condition)** | ผู้ดูแลระบบเข้าสู่ระบบด้วยบัญชีที่มีสิทธิ์ผู้ดูแล และมีบัญชีสมาชิกเป้าหมายอยู่ในระบบ |
| **ลำดับเหตุการณ์หลัก (Basic Flow)** | **ระงับบัญชี:** 1. เปิดหน้าจัดการบัญชีผู้ใช้ ค้นหาบัญชีเป้าหมาย<br>2. กด "ระงับบัญชี" ระบุเหตุผลและระยะเวลาระงับ (เลือกจากตัวเลือกมาตรฐาน หรือถาวร — ดู Business Rules)<br>3. กดยืนยัน ระบบเพิกถอน token ที่ใช้งานอยู่ของบัญชีนั้นทันที และบันทึกเหตุผล/ระยะเวลาไว้<br>4. ส่งอีเมลแจ้งเหตุผลและระยะเวลาการระงับไปยังบัญชีที่ถูกระงับ<br><br>**ยกเลิกการระงับ:** 1. เปิดหน้าจัดการบัญชีผู้ใช้ ค้นหาบัญชีที่ถูกระงับอยู่<br>2. กด "ยกเลิกการระงับ"<br>3. ระบบขอคำยืนยัน<br>4. ยืนยันแล้วบัญชีกลับมาเข้าสู่ระบบ (UC-31) และรีเซ็ตรหัสผ่าน (UC-38) ได้ตามปกติ |
| **ลำดับเหตุการณ์ทางเลือก (Alternative/Exception Flow)** | ระงับแบบชั่วคราวครบกำหนดเวลา → ระบบยกเลิกการระงับให้อัตโนมัติโดยไม่ต้องมีผู้ดูแลกดยกเลิก; กดยกเลิกกลางทางระหว่างกรอกเหตุผล/ระยะเวลา → ไม่มีการเปลี่ยนแปลงสถานะบัญชี |
| **เงื่อนไขหลังทำงานสำเร็จ (Post-condition)** | บัญชีเปลี่ยนสถานะเป็น "ถูกระงับ" หรือกลับเป็น "ปกติ" ตามการดำเนินการ ระหว่างถูกระงับ บัญชีจะเข้าสู่ระบบ (UC-31) หรือรีเซ็ตรหัสผ่าน (UC-38) ไม่ได้ ตามที่ระบุไว้ใน alt flow ของทั้งสอง use case นั้น |
| **ความสัมพันธ์กับ Use Case อื่น** | เกี่ยวข้องกับ UC-31 และ UC-38 (เป็นเงื่อนไขปฏิเสธการเข้าถึงในทั้งสอง flow — ถูกอ้างอิงจากทั้งคู่มาตั้งแต่ก่อนเพิ่ม use case นี้) |
| **กฎทางธุรกิจที่เกี่ยวข้อง (Business Rules)** | **ระยะเวลาระงับ — ยืนยันแล้ว (ดูหมายเหตุข้อ 16):** ตัวเลือกมาตรฐาน 1 วัน / 7 วัน / 30 วัน หรือถาวร (ไม่มีกำหนด) — ระงับถาวรต้องยกเลิกด้วยมือโดยผู้ดูแลเท่านั้น ระงับแบบมีกำหนดเวลาจะยกเลิกอัตโนมัติเมื่อครบกำหนด |
| **เกี่ยวข้องกับ Service/Endpoint อื่น** | - |
