# FOCUS · แอปพลิเคชันจับเวลาโฟกัสในรูปแบบเกม (Focus) — Use Case Description

> ตามรูปแบบ [`template_usecase_description.md`](./template_usecase_description.md) — 1 use case ในเอกสารนี้ตรงกับ 1 use case ใน `usecase_module1.puml` ถึง `usecase_module7.puml` (ดู `usecase_overview.puml` สำหรับภาพรวม) จัดกลุ่มเป็น Epic ภายในแต่ละโมดูล

**หมายเหตุทั่วไป (Cross-cutting concerns):**

1. **การตรวจสอบสิทธิ์ (Authentication/Authorization):** ทุก use case ที่ Actor หลักเป็น "สมาชิก", "สมาชิกในห้อง" หรือ "ผู้ดูแลระบบ" ต้องผ่านการตรวจสอบสิทธิ์ (JWT/Keycloak ผ่าน M0) มาก่อนเสมอ จึงไม่ระบุซ้ำในแถว "เกี่ยวข้องกับ Service/Endpoint อื่น" ของแต่ละ use case — ยกเว้น UC-01 และ UC-38 ที่ Actor หลักเป็น Guest (ยังไม่มีสิทธิ์)
2. **Service/Endpoint:** โครงงานนี้ยังไม่ได้ออกแบบ API endpoint แยกรายตัว (อยู่ระหว่างขั้นตอน proposal) จึงระบุ "-" ในช่อง "เกี่ยวข้องกับ Service/Endpoint อื่น" ของทุก use case ไว้ก่อน รอเติมเมื่อเข้าสู่ขั้นตอนออกแบบระบบ (system design)
3. **การยุบรวม use case:** ตามนิยาม "Use Case ต่างจาก Process/Flow" — UC-06, UC-07, UC-09, UC-11, UC-14, UC-15, UC-16, UC-17 (เดิม) ไม่มี actor ที่ริเริ่มด้วยความสมัครใจ เป็นเพียงกระบวนการ/กฎทางธุรกิจอัตโนมัติที่เกิดสืบเนื่องจาก use case อื่น จึงถูกยุบรวมเป็นแถว **"กฎทางธุรกิจที่เกี่ยวข้อง (Business Rules)"** ภายใน UC-04 และ UC-10 แทนการแยกเป็น use case ของตัวเอง รหัส UC ที่ถูกยุบจะไม่ถูกนำกลับมาใช้ซ้ำ (ปล่อยเป็นเลขว่างในลำดับ) — เพื่อไม่ให้ขัดกับกฎนี้เอง use case "ระบบชุบชีวิต" ที่เคยใช้รหัส **UC-17.1** (พาดพิงเลข 17 ที่ถูกยุบไปแล้ว) จึงเปลี่ยนไปใช้รหัสใหม่เป็น **UC-30** ในเอกสารฉบับนี้
4. **รูปแบบรหัสอ้างอิง (FN-XXX-NNN):** ใช้ตัวย่อโมดูลกำกับ — `ACC` = บัญชี/เริ่มต้นใช้งาน (M1), `SES` = เซสชันโฟกัส (M2), `ITM` = ไอเทม/กล่องสุ่ม (M3), `AVT` = อวตาร/ชุบชีวิต (M4), `ROOM` = ห้องโฟกัส/สังคม (M5), `HIST` = ประวัติ/สถิติ (M6), `ADM` = admin (M7) เลขรันตามลำดับภายในโมดูล ไม่ใช่เลข UC เดิม (เพราะมี UC ที่ถูกยุบ)
5. **Use case ที่เพิ่มใหม่ในรอบรีวิวที่ 1:** UC-31 (เข้าสู่ระบบ), UC-32 (ออกจากระบบ), UC-34 (ออกจากห้องโฟกัส), UC-35 (ยกเลิก/ลบเพื่อน) เพื่อปิดช่องโหว่ coverage ที่พบระหว่างรีวิว
6. **แก้ไข/เพิ่มเติมในรอบรีวิวที่ 2 (ตรวจพบจากการ cross-check เอกสาร):**
   - **UC-23 (ดูกระเป๋าไอเทม)** ถูกอ้างอิงจาก UC-10, UC-12, UC-13 มาตั้งแต่ต้น แต่ไม่เคยถูกเขียนเป็น use case จริงในเอกสาร — เพิ่มเข้ามาในรอบนี้
   - **UC-34 (ออกจากห้องโฟกัส)** ถูกประกาศไว้ในหมายเหตุข้อ 5 ว่าเป็น use case ใหม่ แต่เนื้อหาจริงกลับถูกฝังไว้เป็นแค่บรรทัดเดียวใน Basic Flow ของ UC-18 — แก้ให้เป็น use case แยกจริงตามที่ประกาศไว้ พร้อมเพิ่มกรณีเจ้าของห้องออกจากห้อง (โอนสิทธิ์/ปิดห้องอัตโนมัติ) ที่เดิมไม่ได้ระบุ
   - **UC-13** อ้างอิงกฎ "HP Decay/Streak" ว่าอยู่ใน UC-04 ทั้งคู่ ซึ่งผิด — กฎ HP Decay หายไปจากเอกสารทั้งหมด (ตกหล่นตอนยุบ UC-15) ส่วนกฎ Streak อยู่ที่ UC-31 ไม่ใช่ UC-04 — แก้ reference และเติมกฎ HP Decay กับสูตร EXP/เลเวลอัพ (ตกหล่นตอนยุบ UC-14) เข้าไปใน UC-04 ตัวเลขที่เติมเป็น **ค่าตั้งต้นสำหรับให้ทีมออกแบบเกมยืนยัน/ปรับจริงอีกครั้ง ไม่ใช่ค่าสุดท้าย**
   - **UC-24 (สวมใส่/ถอดไอเทมแต่งตัว)**, **UC-33 (แก้ไขโปรไฟล์และเปลี่ยนรหัสผ่าน)**, **UC-36 (เตะสมาชิกออกจากห้อง)** เพิ่มใหม่เพื่อปิดช่องโหว่ coverage ที่พบระหว่างรีวิว
   - เพิ่มข้อกำหนดชัดเจนเรื่องการคำนวณ HP บอสกลุ่มใหม่เมื่อมีสมาชิกออก/ถูกเตะกลางเซสชัน (ใน UC-18) เพราะเดิมมีช่องโหว่ logic ที่ยังไม่ได้ระบุ
   - **UC-30** อ้างอิงถึง "UC-05" ซึ่งไม่มีอยู่จริงในเอกสาร (เอกสารไม่มี use case เลข 05) — แก้เป็นชี้ไปที่ alt flow "ยกเลิกกลางทาง" ของ UC-04 แทน เพราะภารกิจชุบชีวิตใช้กลไกเซสชันเดียวกับ UC-04 อยู่แล้ว
   - **UC-37 (ตั้งค่าระยะเวลาเริ่มต้นของโฟกัส/พัก)** เพิ่มใหม่ เพราะ "ระยะเวลาเริ่มต้น/ค่า default" เป็นเรื่องตั้งค่าที่ทำไม่บ่อย เหมาะอยู่ในหน้าตั้งค่าเดียวกับ UC-33 (คนละเรื่องกับ UC-03 ที่เป็นลิสต์สิ่งที่โฟกัสซึ่งใช้บ่อยทุกเซสชัน จึงไม่ควรอยู่ในตั้งค่า) — UC-04 ขั้นตอนที่ 1 อัปเดตให้ pre-fill ค่าจาก UC-37
   - **หมายเหตุ:** ยังไม่ได้อัปเดตไดอะแกรม `.puml` (`usecase_module1/3/5.puml`, `usecase_overview.puml`) ให้ตรงกับ use case ที่เพิ่ม/แก้ในรอบนี้ ต้องทำก่อนใช้เอกสารชุดนี้ประกอบการยื่นจริง เพราะคู่มือ `generator/guide/usecase_generate_guide.md` กำหนดให้รหัส UC ในไดอะแกรมต้องตรงกับเอกสารข้อความครบ 1:1
   - ส่วนเรื่อง Report/Block ผู้ใช้อื่น, Tutorial/Onboarding, Global Leaderboard/Achievement และการลบบัญชี/ส่งออกข้อมูล (PDPA) ยังไม่เพิ่มในรอบนี้ พิจารณาเพิ่มในรอบถัดไปเมื่อใกล้ขั้น production (Forgot Password/Account Recovery ถูกเพิ่มแล้วในรอบถัดมา — ดูข้อ 7 และ UC-38)
7. **เพิ่มเติมในรอบรีวิวที่ 3 (ลืมรหัสผ่าน):** เพิ่ม **UC-38 (ลืมรหัสผ่าน / Reset Password)** ในโมดูล M1/E3 เพื่อปิดช่องโหว่ coverage "Forgot Password/Account Recovery" ที่ระบุไว้ค้างในหมายเหตุข้อ 6 — Actor หลักเป็น Guest เพราะเกิดขึ้นตอนที่สมาชิกเข้าสู่ระบบไม่ได้ (ยังไม่มี token) จึงถูกเพิ่มเป็นข้อยกเว้นที่สองในหมายเหตุข้อ 1 ต่อจาก UC-01 คนละกรณีกับ UC-33 ซึ่งเป็นการเปลี่ยนรหัสผ่านขณะยังล็อกอินอยู่ — **หมายเหตุ:** ยังไม่ได้อัปเดตไดอะแกรม `usecase_module1.puml`/`usecase_overview.puml` ให้มี UC-38 ต้องทำก่อนใช้เอกสารชุดนี้ประกอบการยื่นจริง
8. **ไล่เช็ค/แก้รหัสอ้างอิง FN-XXX-NNN (หลังเพิ่ม UC-38):** ตรวจพบ 3 จุดที่ไม่สอดคล้องกับกฎในหมายเหตุข้อ 4 จึงแก้ดังนี้ — (ก) `FN-ACC-004` เคยขาดหายจากลำดับ (001,002,003,**005**,006,007,008) จึงไล่เลขใหม่ให้ติดกันเป็น 001-006 (UC-01=001, UC-02=002, UC-03=003, UC-33=004, UC-38=005, UC-37=006); (ข) UC-31 (เข้าสู่ระบบ) และ UC-32 (ออกจากระบบ) เดิมใช้รหัส `ACC` ทั้งที่จัดวางอยู่ใต้โมดูล M2 (E3) จึงเปลี่ยนเป็น `FN-SES-002` และ `FN-SES-003` ตามลำดับให้ตรงกับโมดูลจริงตามกฎข้อ 4; (ค) `FN-ROOM-004` และ `FN-ROOM-005` เคยขาดหายจากลำดับ (001,002,003,**006**,007,008 — ไม่ตรงกับลำดับที่ปรากฏจริงในเอกสารด้วย) จึงไล่เลขใหม่ตามลำดับที่ use case ปรากฏจริงในเอกสารเป็น 001-006 (UC-18=001, UC-20=002, UC-34=003, UC-36=004, UC-21=005, UC-35=006) — ยืนยันแล้วว่ารหัส FN- ไม่ถูกอ้างอิงข้ามที่อื่นในเอกสารเลย (อ้างอิงกันด้วยเลข UC- เท่านั้น) จึงไม่กระทบ cross-reference ภายใน แต่ **ต้องเช็คว่ามีเอกสาร/ระบบอื่นนอกไฟล์นี้ (เช่น ตั๋วงาน, API spec) ที่อ้างอิงรหัสเดิมอยู่หรือไม่ก่อนนำไปใช้จริง**
9. **เพิ่มความชัดเจนให้ Basic Flow ที่คลุมเครือ (รอบตรวจรีวิวที่ 4):**
   - **UC-04:** ขั้นตอนที่ 3 เดิมบอกแค่ "คืน HP อวตาร" โดยไม่มีสูตร — เพิ่มกฎ **"การคืน HP อวตารเมื่อเซสชันสำเร็จ"** ใน Business Rules (ค่าตั้งต้น: คืนเต็ม 100% ทุกครั้งที่จบสำเร็จ ไม่มีผลกับเซสชันที่ยกเลิกกลางทาง)
   - **UC-12:** เพิ่มแถว Business Rules ระบุชัดว่า **ยังไม่มีตารางสูตรผสมไอเทมจริง** ในเอกสาร ต้องรอทีมออกแบบเกมกำหนดก่อนเข้าสู่ขั้น system design
   - **UC-13:** เพิ่มตารางชนิดโล่เวลาใน Business Rules (ค่าตั้งต้น: สามัญ/หายาก = 24 ชม., หายากมาก/ตำนาน = 48 ชม.)
   - **UC-18:** แตก Basic Flow จากประโยคเดียวต่อ sub-flow เป็นขั้นตอนแบบเลขลำดับ (สร้างห้อง/เข้าร่วมห้อง/ปิดห้อง) ให้รายละเอียดเท่ากับ UC อื่น
   - **UC-30:** เพิ่มกฎ **"เงื่อนไขภารกิจชุบชีวิต"** ระบุค่าตั้งต้นว่าเป็นเซสชันคงที่ 20 นาที ไม่อิงกับเซสชันที่ทำให้หมดสติ และไม่ได้รับกล่องสุ่ม/EXP ระหว่างทำภารกิจนี้
   - **UC-24:** แยก Basic Flow เป็นสอง sub-flow ชัดเจน คือ **สวมใส่ (Equip)** และ **ถอด (Unequip)** แทนการรวบไว้ในขั้นตอนเดียวกัน
   - **UC-32:** ระบุจุดเริ่มต้นของปุ่ม "ออกจากระบบ" ว่าอยู่ในหน้าตั้งค่า (ให้สอดคล้องกับ UC-33/UC-37 ที่อยู่หน้าเดียวกัน)
   - ตัวเลขที่เพิ่ม (20 นาทีของ UC-30, 24/48 ชม. ของ UC-13, การคืน HP 100% ของ UC-04) เป็น **ค่าตั้งต้นให้ทีมออกแบบเกมยืนยัน/ปรับจริงอีกครั้ง เช่นเดียวกับตัวเลขอื่นที่ทำเครื่องหมายไว้ในเอกสารนี้** (สูตรผสมของ UC-12 ที่ตอนนั้นยังเป็นช่องว่าง ถูกเติมเนื้อหาจริงแล้วในหมายเหตุข้อ 10)
10. **แก้ไขกลไกกล่องสุ่ม/สูตรผสม/หน้าสถิติ ตามข้อกำหนดที่ผู้ใช้ยืนยันเพิ่มเติม (รอบตรวจรีวิวที่ 5):**
    - **UC-04 → UC-10:** เดิม UC-04 เขียนว่าได้กล่องสุ่มแล้ว "ไปเปิดใน UC-10" ทำให้ดูเหมือนเปิดได้ทันทีจากหน้าสรุปผล — แก้ให้ชัดว่า**เมื่อจบเซสชันจะได้กล่องสุ่มเก็บเข้ากระเป๋าเท่านั้น ยังเปิดไม่ได้ทันที ต้องไปเปิดจากกระเป๋า (UC-23) ผ่าน UC-10 เท่านั้น** ตัด "หรือหน้าสรุปผล" ออกจาก Basic Flow ของ UC-10
    - **UC-10:** กล่องสุ่มที่ได้รับแต่ละใบถูกกำหนด**ชนิดตายตัวตามความยาวเซสชัน ณ ตอนได้รับ** เป็น 3 ชนิด (กล่องระดับต้น 20-30 นาที / ระดับกลาง 30-60 นาที / ระดับสูง 60 นาทีขึ้นไป) แต่ละชนิดแยกกันในกระเป๋าและมีอัตราดรอปของตัวเอง (ใช้ตารางเดิม แค่เปลี่ยน label หัวตารางจาก "ความยาวเซสชัน" เป็น "ชนิดกล่อง") เพิ่ม alt flow กรณีไม่มีกล่องเหลือในกระเป๋าเลย
    - **UC-23:** เพิ่ม "กล่องสุ่มที่ยังไม่เปิด" เป็นหมวดไอเทมที่แสดงในกระเป๋า (แยกย่อยตามชนิด) และระบุว่าเป็นจุดเข้าเปิดกล่องเพียงจุดเดียว
    - **UC-12:** เติม**ตารางสูตรผสมจริง**แทนที่หมายเหตุ "ยังไม่มีสูตร" เดิม — มี 3 สูตร: อัพเกรดไอเทมแต่งตัว (5 ชิ้นระดับเดียวกัน → 1 ชิ้นระดับถัดไป), ผสมโล่เวลา 24 ชม. (วัตถุดิบผสมสามัญ/หายาก 3 ชิ้น), ผสมโล่เวลา 48 ชม. (วัตถุดิบผสมหายากมาก/ตำนาน 3 ชิ้น) — ตัวเลขจำนวนชิ้นเป็นค่าตั้งต้นรอทีมออกแบบเกมยืนยันเช่นเดิม
    - **UC-22:** เพิ่มตัวเลือกช่วงเวลา **"ทั้งหมด" (All-time)** ในตัวกรอง และระบุชัดว่าสถิติรายรายการที่โฟกัส (ขั้นตอนที่ 5) ก็แยกเวลาโฟกัส/พัก/เบรกเกินเหมือนภาพรวมในขั้นตอนที่ 4 ไม่ใช่แค่ยอดรวมเดียว
11. **รีวิวเอกสารทั้งไฟล์หลังรอบที่ 5 แล้วแก้จุดที่ยังไม่เคลียในตัวเอกสาร (รอบตรวจรีวิวที่ 6):** (ไม่รวมประเด็นนอกเอกสารนี้ เช่น ไดอะแกรม .puml ที่ยังไม่อัพเดต และเนื้อหา UC-28 เป็นต้นไปที่หายไปจากไฟล์ต้นฉบับ — ยังเป็น backlog ค้างเหมือนเดิม)
    - **UC-12:** คำอธิบายโดยย่อเดิมพูดถึงแค่สูตร "อัพเกรดเป็นไอเทมที่หายากขึ้น" ไม่ครอบคลุมสูตรผสมโล่เวลาที่เพิ่งเพิ่มในรอบก่อน — แก้ให้ครอบคลุมทั้ง 2 กลุ่มสูตร พร้อมเพิ่มขั้นตอน "เลือกสูตร" ใน Basic Flow ก่อนขั้นตอนเลือกไอเทมต้นทาง (เดิมข้ามขั้นตอนนี้ไปทั้งที่ตอนนี้มีสูตรให้เลือกมากกว่า 1 แบบแล้ว) และเพิ่ม alt flow กรณีเลือกไอเทมไม่ตรงกับสูตรที่เลือกไว้
    - **UC-10:** ตาราง Business Rules เดิมใส่แถว "ต่ำกว่า 20 นาที (ไม่ได้รับกล่องสุ่ม)" ปนอยู่ในตารางเดียวกับ "ชนิดกล่อง" ทำให้ดูเหมือนเป็นชนิดกล่องชนิดที่ 4 — แยกกรณี "ไม่ได้กล่องสุ่ม" ออกมาเป็นข้อความอธิบายก่อนตาราง ให้ตารางเหลือแค่ 3 ชนิดกล่องจริงเท่านั้น
    - **UC-26:** เดิมพูดถึง "ตารางอัตราดรอป" เป็นเอกพจน์ ทั้งที่หลังรอบที่ 5 กล่องสุ่มแยกเป็น 3 ชนิดแล้ว แต่ละชนิดมีตารางอัตราดรอปของตัวเอง — แก้คำอธิบาย/Basic Flow/Alt Flow ให้ระบุชัดว่าเป็น 3 ตารางแยกกัน และการตรวจผลรวม 100% ต้องเช็คแยกต่อชนิดกล่อง
12. **เพิ่ม UC-28 ที่หายไปจากไฟล์ต้นฉบับ (รอบตรวจรีวิวที่ 7):** UC-31 และ UC-38 อ้างอิงถึง "บัญชีถูกระงับ (UC-28)" มาตั้งแต่ต้น แต่ไม่เคยมี use case นี้เขียนไว้จริงในเอกสาร — สาเหตุคือไฟล์ต้นฉบับที่ใช้สร้างเอกสารนี้ตัดจบกลางประโยคหลัง UC-27 พอดี (มีอักขระเสียหาย/ขาดหายในไฟล์จริง ไม่ใช่การละไว้โดยตั้งใจ) ทำให้เนื้อหาตั้งแต่ UC-28 เป็นต้นไปหายไปทั้งหมด — แก้ไขโดย: (ก) เติมประโยคที่ขาดใน Alt Flow ของ UC-27 พร้อมแถว Post-condition/ความสัมพันธ์/Service ที่หายไปให้สมบูรณ์ (ทำเครื่องหมายกำกับไว้ในเนื้อหาว่าเป็นการเติมย้อนหลัง); (ข) เพิ่ม **UC-28 (ระงับ/เพิกถอนสิทธิ์บัญชีผู้ใช้)** เป็น use case ใหม่ รหัส `FN-ADM-003` โดยอนุมานเนื้อหาจากบริบทที่ถูกอ้างอิงถึง (ผลลัพธ์ต้องบล็อกทั้งการเข้าสู่ระบบและการรีเซ็ตรหัสผ่าน) — **ตัวเนื้อหา Basic Flow และระยะเวลาระงับเป็นการอนุมาน/ค่าตั้งต้นทั้งหมด ไม่ใช่เนื้อหาต้นฉบับจริง ต้องให้ทีมงานจริงตรวจทานอีกครั้งก่อนใช้งาน** — ยังไม่ได้อัพเดตไดอะแกรม `.puml` ให้มี UC-28 เช่นเดียวกับ use case อื่นที่เพิ่มก่อนหน้านี้
13. **รีวิวซ้ำหลังเพิ่ม UC-28 (รอบตรวจรีวิวที่ 8):** ตรวจไม่พบรหัส FN-XXX ซ้ำหรือขาดหาย (25 รหัส) และไม่มี UC ใดถูกอ้างอิงโดยไม่มีตัวจริงอีกแล้ว (ยกเว้นเลขที่ถูกยุบตามหมายเหตุข้อ 3 ซึ่งตั้งใจ) — แต่พบว่า UC-28 ถูกวางไว้ใต้ epic "E1 · การจัดการเนื้อหาเกม" ของโมดูล M7 ซึ่งผิดหมวด (UC-28 เป็นเรื่องจัดการบัญชีผู้ใช้ ไม่ใช่เนื้อหาเกม ขัดกับรูปแบบการจัดกลุ่ม epic ที่ใช้ทั้งเอกสาร เช่น M3/M5 ที่แยก E1/E2 ตามหมวดจริง) — แก้โดยเพิ่ม **epic ใหม่ "E2 · การจัดการบัญชีผู้ใช้"** ในโมดูล M7 แล้วย้าย UC-28 มาอยู่ใต้ epic นี้แทน
14. **ปรับ Logic ระบบ HP/HP Decay (UC-04) และ Pity System (UC-10) ตามที่ทีมออกแบบเกมยืนยันมา (รอบตรวจรีวิวที่ 9):**
    - **UC-02/UC-04:** กำหนด **Base Max HP = 100 หน่วยที่เลเวล 1** และ **Level Scaling: Max HP +5 หน่วยทุกครั้งที่เลเวลอัพ** (Current HP ไม่เปลี่ยนตาม ไม่มีการฟื้นฟูจากการเลเวลอัพ) — ปิดช่องว่างเดิมที่เขียนไว้ว่า "เลเวลอัพยังไม่ส่งผลต่อ HP สูงสุด" ตอนนี้มีสูตรแล้วและถือเป็นค่ายืนยัน ไม่ใช่ค่าตั้งต้นอีกต่อไป
    - **UC-04:** ปรับ **HP Decay Rate จาก 5 หน่วย/ชั่วโมง เป็น 2 หน่วย/ชั่วโมง** (ค่ายืนยันแล้ว) — ปรับคำอธิบายการคืน HP เต็มตอนจบเซสชันสำเร็จ ให้หมายถึง "เท่ากับ Max HP ปัจจุบัน" แทนคำว่า "100%" เฉยๆ เพื่อไม่ให้สับสนกับเลข Base Max HP=100 ที่เพิ่งกำหนด
    - **กลไก "หมดสติ" (Fainted):** ทีมงานส่งมาว่าต้องการเปลี่ยนเป็น "ยังโฟกัสได้ปกติ ไม่ล็อกฟีเจอร์ แค่ไม่ได้ EXP/กล่องจนกว่าจะใช้ไอเทมฟื้นฟู HP" ซึ่งขัดกับกลไกเดิมของ **UC-30** ที่ต้องทำภารกิจโฟกัสสำเร็จเพื่อชุบชีวิต — ถามยืนยันแล้วและ **ผู้ใช้เลือกให้คงกลไกหมดสติ/UC-30 ไว้แบบเดิมทั้งหมด** (ล็อกฟีเจอร์จนกว่าจะชุบชีวิตผ่าน UC-30) จึงไม่ได้นำส่วน "ไอเทมฟื้นฟู HP" มาใช้ — ปรับ Post-condition ของ UC-30 เพิ่มเติมแค่ให้สอดคล้องว่าเมื่อรีเซ็ตเลเวลกลับเป็น 1 แล้ว Max HP ก็กลับไปที่ค่าฐาน 100 ตามไปด้วย
    - **UC-10:** ปรับ **Pity System ให้แยกตัวนับตามชนิดกล่องโดยสิ้นเชิง** (เดิมนับรวมทุกชนิด) พร้อมกำหนดเกณฑ์การันตีต่อชนิด (Bronze→หายากขึ้นไป, Silver→หายากมากขึ้นไป, Gold→ตำนานขึ้นไป) — map ชื่อระดับจากที่ทีมส่งมา (Uncommon/Rare/Epic) เป็นภาษาที่ใช้ในเอกสาร (หายาก/หายากมาก/ตำนาน) ตามที่ยืนยัน — และ**ปรับช่วงเวลาของชนิดกล่องจาก 20-30/30-60/60+ นาที เป็น 20-44/45-89/90+ นาที** พร้อมตั้งชื่อภาษาอังกฤษกำกับ (Bronze/Silver/Gold) อัตราดรอป % เดิมไม่เปลี่ยน เปลี่ยนแค่ขอบเขตนาทีที่แบ่งชนิด
15. **แก้บทลงโทษการชุบชีวิตใน UC-30 (รอบตรวจรีวิวที่ 10):** เดิม UC-30 รีเซ็ตเลเวล/EXP กลับเลเวล 1 ทั้งหมดเมื่อชุบชีวิตสำเร็จ ซึ่งมีหมายเหตุจากรีวิวเตือนไว้แล้วว่าโทษแรงเกินไปและแนะนำให้ลดเลเวลบางส่วนแทน — ทีมออกแบบเกมยืนยันให้เปลี่ยนเป็น **ลดเลเวลลง 2 ระดับ** จากเลเวลก่อนหมดสติแทนการรีเซ็ตทั้งหมด (ต่ำสุดคือเลเวล 1 ไม่ติดลบ) EXP ของเลเวลใหม่เริ่มจาก 0 และ Max HP ลดตามส่วนต่าง 2 เลเวล × 5 หน่วย/เลเวล (ตาม Level Scaling ของ UC-04 ที่เพิ่งยืนยันในหมายเหตุข้อ 14) — ปรับ Basic Flow, Post-condition, Business Rules ของ UC-30 ให้ตรงกัน และปิดหมายเหตุจากรีวิวเดิมว่าได้รับการแก้ไขแล้ว

## M2 · เซสชันโฟกัส (Core Loop)

### E1 · วงจรเซสชันโฟกัส

#### UC-04 · เริ่มจับเวลาโฟกัส (ตีบอส)

| หัวข้อ | รายละเอียด |
|-------|-----------|
| **รหัสอ้างอิง** | FN-SES-001 |
| **Actor หลัก** | สมาชิก |
| **Actor รอง** | ระบบจับเวลา, ระบบบอส |
| **คำอธิบายโดยย่อ** | สมาชิกเลือกรายการที่จะโฟกัสและตั้งเวลาเริ่มจับเวลา ระบบจะจำลองเป็นการต่อสู้กับบอส ครอบคลุมไปถึงการพัก การยกเลิก และการแจกรางวัล |
| **เงื่อนไขก่อนเริ่มทำงาน (Pre-condition)** | สมาชิกล็อกอิน มีอวตาร และมีรายการสิ่งที่โฟกัส (UC-03) |
| **ลำดับเหตุการณ์หลัก (Basic Flow)** | 1. เลือกรายการจาก UC-03 และตั้งระยะเวลาโฟกัส/พักส่วนตัว (ระบบ pre-fill ค่าเริ่มต้นจาก UC-37 ถ้าเคยตั้งไว้ สามารถแก้ไขเป็นรายเซสชันได้เสมอ)<br>2. กดเริ่มจับเวลา ระบบจำลองบอส (HP = เวลา × 100)<br>3. แสดงนาฬิกาถอยหลัง เมื่อเวลาครบ เซสชันสำเร็จ คืน HP อวตารเต็ม 100% (ดูกฎ 'การคืน HP อวตารเมื่อเซสชันสำเร็จ' ด้านล่าง) และบันทึกประวัติ<br>4. ได้รับรางวัล EXP และได้รับกล่องสุ่ม 1 ใบเก็บเข้ากระเป๋า หากเซสชัน ≥ 20 นาที (ชนิดกล่องถูกกำหนดตายตัวตามความยาวเซสชัน ณ ตอนนี้ — ดู Business Rules ของ UC-10) **ยังเปิดกล่องไม่ได้ทันที ต้องไปเปิดจากกระเป๋าเท่านั้นผ่าน UC-10**<br>5. ระบบเริ่มนับเวลาพักอัตโนมัติตามเวลาที่ตั้งไว้<br>6. เมื่อจบช่วงพักสามารถเริ่มเซสชันใหม่ได้ หากไม่เริ่ม ระบบจะนับเวลาพักเกินกำหนดเป็นสถิติ |
| **ลำดับเหตุการณ์ทางเลือก (Alternative/Exception Flow)** | **ยกเลิกกลางทาง:** กดยกเลิกเซสชันกลางทาง (กดค้าง) → คืน HP ตามสัดส่วนเวลา บันทึกว่ายกเลิก ไม่ได้รับกล่องสุ่ม<br>**พักเกินกำหนด:** หมดเวลาพักแต่ยังไม่เริ่มใหม่ → ระบบส่ง push notification แจ้งว่าเวลาพักหมดแล้ว (แม้แอปอยู่ background) จากนั้นเริ่มนับเวลาพักเกินไปเรื่อยๆ เพื่อเป็นสถิติ<br>**จบช่วงพักก่อนกำหนด:** กดข้ามช่วงพัก → เริ่มเซสชันใหม่ได้ทันที<br>**อวตารหมดสติ:** HP อวตารเท่ากับ 0 ก่อนเริ่ม → เข้าสถานะ "หมดสติ" ล็อกฟีเจอร์จนกว่าจะชุบชีวิต (UC-30) |
| **เงื่อนไขหลังทำงานสำเร็จ (Post-condition)** | เซสชันถูกบันทึกลงประวัติ สถิติเวลาพัก/เบรกเกิน รางวัล และ HP อัพเดตตามกฎทางธุรกิจ |
| **ความสัมพันธ์กับ Use Case อื่น** | ถูก `<<include>>` โดย UC-30 |
| **กฎทางธุรกิจที่เกี่ยวข้อง (Business Rules)** | **โหมดโฟกัสคนเดียว (Single-player Focus):**<br>- การนับเวลาจับเวลาของอวตารนับถอยหลังแบบเรียลไทม์ (Real-time countdown) สมาชิกเห็นเวลาลดลงวินาทีต่อวินาทีบนหน้าจอ<br>- การคำนวณหักล้างค่าพลังชีวิต (HP) ของบอสจะไม่หักทุกวินาที แต่ระบบจะหักลดเลือดบอสทุกๆ 5 นาทีที่ผู้ใช้โฟกัสสำเร็จ (ส่งดาเมจไปหักเลือดบอสเป็นก้อนทุกๆ 5 นาที)<br>- *ข้อเสนอแนะ UX (ไม่ใช่ business logic):* ฝั่ง client ควรแสดงแถบเลือดบอสไหลลดลงต่อเนื่องแบบ interpolate ระหว่าง 2 จุดที่หักจริง แทนการกระโดดเป็นก้อนทุก 5 นาที เพื่อให้สอดคล้องกับนาฬิกาถอยหลังที่เดินทุกวินาที<br><br>**Max HP และ HP Decay — ยืนยันแล้วโดยทีมออกแบบเกม (แทนที่ค่าตั้งต้นเดิม):**<br>- **Base Max HP:** อวตารเริ่มต้นที่ Max HP = 100 หน่วย ที่เลเวล 1 (ตรงกับ Post-condition ของ UC-02)<br>- **Level Scaling:** ทุกครั้งที่เลเวลอัพ (ตามสูตร EXP ด้านล่าง) Max HP เพิ่มขึ้น +5 หน่วยทันที โดย Current HP ไม่เปลี่ยนแปลง (ไม่มีการฟื้นฟู HP จากการเลเวลอัพ)<br>- **HP Decay Rate:** อวตารเสีย HP 2 หน่วย ต่อทุก 1 ชั่วโมงที่สมาชิกไม่ได้อยู่ในเซสชันโฟกัส นับจากเวลาสิ้นสุดเซสชันล่าสุด (หรือเวลาสร้างอวตารถ้ายังไม่เคยโฟกัสเลย) — ปรับจากค่าตั้งต้นเดิม 5 หน่วย/ชั่วโมง<br>- ไม่มีการหัก HP ระหว่างที่ไอเทมโล่เวลา (UC-13) มีผลอยู่<br>- เมื่อ HP อวตารแตะ 0 → เข้าสถานะ "หมดสติ" ตามลำดับเหตุการณ์ทางเลือกด้านบน (กลไกหมดสติ/ชุบชีวิตยังคงเป็นไปตาม UC-30 แบบเดิมทุกประการ ไม่มีการเปลี่ยนแปลง)<br><br>**สูตร EXP และเลเวลอัพ — ค่าตั้งต้น รอทีมออกแบบเกมยืนยัน (เฉพาะส่วนนี้):**<br>- ได้รับ EXP = จำนวนนาทีที่โฟกัสสำเร็จ × 2 ต่อเซสชัน (เฉพาะเซสชันที่จบสำเร็จ ไม่นับเซสชันที่ยกเลิกกลางทาง)<br>- เลเวลอัพเมื่อ EXP สะสมถึงเกณฑ์ของเลเวลถัดไป (ตัวอย่างสูตรขั้นบันได: เลเวล N ต้องใช้ N × 100 EXP สะสม) เอกสารนี้ยังไม่กำหนดเพดานเลเวลสูงสุด — เลเวลอัพส่งผลต่อ Max HP แล้วตามกฎ Level Scaling ด้านบน (ยืนยันแล้ว ไม่ใช่ค่าตั้งต้นอีกต่อไป)<br><br>**การคืน HP อวตารเมื่อเซสชันสำเร็จ (Session Success Heal) — ค่าตั้งต้น รอทีมออกแบบเกมยืนยัน:**<br>- อวตารได้รับการคืน HP เต็ม (เท่ากับ Max HP ปัจจุบันของอวตาร ซึ่งอาจมากกว่า 100 หากเลเวลอัพแล้ว) ทุกครั้งที่จบเซสชันสำเร็จ ไม่ว่าเซสชันจะสั้นหรือยาวแค่ไหน ตราบใดที่โฟกัสจนครบเวลาที่ตั้งไว้<br>- ไม่มีผลกับเซสชันที่ถูกยกเลิกกลางทาง (ดู alt flow "ยกเลิกกลางทาง") |
| **เกี่ยวข้องกับ Service/Endpoint อื่น** | - |

## M4 · อวตารและความต่อเนื่อง (ระบบชุบชีวิต)

> UC-14 (เลเวลอัพ), UC-15 (HP Decay), UC-16 (Streak), UC-17 (หมดสติ) เดิม ถูกยุบรวมเป็นกฎทางธุรกิจของ UC-04/UC-31 (โมดูล 2) แล้ว เพราะไม่มี actor ริเริ่มเอง — เหลือ UC-30 (เดิมชื่อ UC-17.1 ก่อนเปลี่ยนรหัสเพื่อไม่ให้ชนกับเลข 17 ที่ถูกยุบไปแล้ว) เป็น use case เดียวในโมดูลนี้ที่สมาชิกลงมือทำเองมีเป้าหมายชัดเจน

### E1 · ระบบชุบชีวิต

| หัวข้อ | รายละเอียด |

| **ความสัมพันธ์กับ Use Case อื่น** | `<<include>>` UC-04 (ภารกิจใช้กลไกเซสชันโฟกัสเดียวกัน) |

| **ความสัมพันธ์กับ Use Case อื่น** | `<<include>>` UC-22 (เมื่อปิดห้องสำเร็จ); เกี่ยวข้องกับ UC-34 (ออกจากห้อง) และ UC-36 (เตะสมาชิก) ในเชิงผลกระทบต่อจำนวนสมาชิกในห้อง |
| **กฎทางธุรกิจที่เกี่ยวข้อง (Business Rules)** | **โหมดโฟกัสร่วมกันในห้อง (Co-working/Group Focus):**<br>- มีจำนวนสมาชิกสูงสุดได้ไม่เกิน 8 คนต่อห้อง<br>- **สูตรคำนวณเลือดบอสกลุ่ม (Dynamic Boss HP):** ระบบจะปรับค่าพลังชีวิต (HP) ของบอสกลุ่มแปรผันตามจำนวนคนในห้อง ณ ขณะนั้น เพื่อไม่ให้บอสตายไวเกินไป โดยคำนวณจากสูตร: HP บอสกลุ่ม = เวลาเป้าหมายรวมของห้อง (นาที) × 100 × จำนวนสมาชิกที่อยู่ในห้อง<br>- **กรณีจำนวนสมาชิกเปลี่ยนกลางเซสชัน:** การคำนวณ HP บอสกลุ่มใหม่จะเกิดเฉพาะตอนมีสมาชิกเข้าร่วมเพิ่ม (ปรับ HP สูงสุดเพิ่มขึ้นเท่านั้น) กรณีมีสมาชิกออกจากห้อง (UC-34) หรือถูกเตะ (UC-36) ระหว่างเซสชัน ระบบจะไม่ลด HP สูงสุดของบอสย้อนหลัง เพื่อไม่ให้ดาเมจที่ตีสะสมไปแล้วเกิน HP ใหม่จนบอสตายผิดเงื่อนไข<br>- สมาชิกแต่ละคนในห้องสามารถกำหนดระยะเวลาโฟกัสและเวลาพักของตัวเองได้อย่างอิสระ (ไม่จำเป็นต้องเริ่มหรือจบพร้อมกัน)<br>- เวลาโฟกัสสะสมของสมาชิกแต่ละคนจะแสดงผลบนหน้าจอห้องแบบเรียลไทม์ (Real-time) และจะวิ่งมารวมกันเป็นพลังโจมตีส่วนกลางเพื่อหักลบค่า HP ของบอสกลุ่มแบบเรียลไทม์<br>- ในช่วงเวลาที่สมาชิกคนใดอยู่ใน "ช่วงเวลาพักส่วนตัว" หรือ "พักเกินกำหนด" ระบบจะหยุดคำนวณพลังโจมตีของสมาชิกคนนั้นชั่วคราว โดยไม่มีผลกระทบต่อเพื่อนคนอื่นในห้องที่ยังโฟกัสอยู่<br>- พลังโจมตีรวมจากสมาชิกทุกคนจะลด HP บอสกลุ่มได้สูงสุดไม่เกินค่า HP ตั้งต้นของบอสตัวนั้นๆ (ไม่เกินเป้าหมายรวมของห้อง) เมื่อบอสเลือดเหลือ 0 ระบบจะปัดเศษและหยุดหักเลือดทันที |

### E1 · การจัดการเนื้อหาเกม

#### UC-26 · จัดการข้อมูลไอเทมและอัตราดรอป

| หัวข้อ | รายละเอียด |
|-------|-----------|
| **รหัสอ้างอิง** | FN-ADM-001 |
| **Actor หลัก** | ผู้ดูแลระบบ |
| **Actor รอง** | ระบบฐานข้อมูลไอเทม |
| **คำอธิบายโดยย่อ** | ผู้ดูแลระบบเพิ่ม แก้ไข หรือปิดใช้งานไอเทมในแคตตาล็อกของระบบ รวมถึงปรับตารางอัตราดรอปของกล่องสุ่มแต่ละชนิด (กล่องระดับต้น/กลาง/สูง ตาม UC-10 — คนละตารางแยกกัน 3 ชุด) และสูตรผสมไอเทม (UC-12) เพื่อรักษาสมดุลของเกม |
| **เงื่อนไขก่อนเริ่มทำงาน (Pre-condition)** | ผู้ดูแลระบบเข้าสู่ระบบด้วยบัญชีที่มีสิทธิ์ผู้ดูแล |
| **ลำดับเหตุการณ์หลัก (Basic Flow)** | 1. เปิดหน้าจัดการไอเทม<br>2. แสดงรายการไอเทมทั้งหมด พร้อมตารางอัตราดรอปปัจจุบันแยกตามชนิดกล่องสุ่ม 3 ชุด (ระดับต้น/กลาง/สูง) และตารางสูตรผสม<br>3. เพิ่ม/แก้ไข/ปิดใช้งานไอเทม หรือปรับอัตราดรอปของกล่องชนิดใดชนิดหนึ่ง หรือปรับสูตรผสม<br>4. ตรวจสอบความถูกต้อง (ผลรวมอัตราดรอปของกล่องแต่ละชนิดต้อง = 100% แยกกันต่อชนิด)<br>5. บันทึกการเปลี่ยนแปลงพร้อมประวัติการแก้ไข |
| **ลำดับเหตุการณ์ทางเลือก (Alternative/Exception Flow)** | ผลรวมอัตราดรอปของกล่องชนิดใดชนิดหนึ่งไม่เท่ากับ 100% → แจ้งข้อผิดพลาด ไม่บันทึก (ตรวจแยกต่อชนิดกล่อง); ปิดใช้งานไอเทมที่ถูกอ้างอิงในสูตรผสม → เตือนพร้อมรายการที่ได้รับผลกระทบก่อนยืนยัน |
| **เงื่อนไขหลังทำงานสำเร็จ (Post-condition)** | ข้อมูลไอเทม/อัตราดรอป/สูตรผสมเวอร์ชันใหม่มีผลกับการเปิดกล่องสุ่ม/ผสมครั้งถัดไป ไอเทมที่สมาชิกถือครองอยู่แล้วไม่ได้รับผลกระทบ |
| **ความสัมพันธ์กับ Use Case อื่น** | ไม่มี |
| **เกี่ยวข้องกับ Service/Endpoint อื่น** | - |

#### UC-27 · จัดการรูปแบบอวตาร

| หัวข้อ | รายละเอียด |
|-------|-----------|
| **รหัสอ้างอิง** | FN-ADM-002 |
| **Actor หลัก** | ผู้ดูแลระบบ |
| **Actor รอง** | ระบบฐานข้อมูลอวตาร |
| **คำอธิบายโดยย่อ** | ผู้ดูแลระบบเพิ่ม แก้ไข หรือปิดใช้งานรูปแบบอวตารเริ่มต้นที่แสดงให้สมาชิกใหม่เลือกใน UC-02 |
| **เงื่อนไขก่อนเริ่มทำงาน (Pre-condition)** | ผู้ดูแลระบบเข้าสู่ระบบด้วยบัญชีที่มีสิทธิ์ผู้ดูแล |
| **ลำดับเหตุการณ์หลัก (Basic Flow)** | 1. เปิดหน้าจัดการรูปแบบอวตาร<br>2. แสดงรูปแบบทั้งหมดพร้อมสถานะ<br>3. เพิ่ม/แก้ไข/ปิดใช้งานรูปแบบ<br>4. บันทึกการเปลี่ยนแปลง อัพเดตหน้าเลือกสำหรับสมาชิกใหม่ทันที |
| **ลำดับเหตุการณ์ทางเลือก (Alternative/Exception Flow)** | ปิดใช้งานรูปแบบที่มีสมาชิกใช้งานอยู่ → สมาชิกเดิมใช้ต่อได้ตามปกติ เพียงซ่อนจากบัญชีใหม่; ปิดจนเหลือน้อยกว่า 3 รูปแบบที่ยังเปิดใช้งานอยู่ → แจ้งเตือนว่าตัวเลือกสำหรับสมาชิกใหม่จะเหลือน้อยเกินไป ให้ยืนยันซ้ำก่อนปิดใช้งานจริง _(หมายเหตุ: ไฟล์ต้นฉบับขาดหายไปตั้งแต่ตรงนี้ ประโยคนี้และแถวด้านล่างถูกเติมให้สมบูรณ์ในรอบตรวจรีวิวที่ 7 — ดูหมายเหตุข้อ 12)_ |
| **เงื่อนไขหลังทำงานสำเร็จ (Post-condition)** | รูปแบบอวตารเวอร์ชันใหม่มีผลกับสมาชิกใหม่ที่สมัคร (UC-02) ตั้งแต่ครั้งถัดไป สมาชิกเดิมที่เคยเลือกรูปแบบที่ถูกปิดใช้งานไปแล้วไม่ได้รับผลกระทบ |
| **ความสัมพันธ์กับ Use Case อื่น** | ไม่มี |
| **เกี่ยวข้องกับ Service/Endpoint อื่น** | - |

### E2 · การจัดการบัญชีผู้ใช้

#### UC-28 · ระงับ/เพิกถอนสิทธิ์บัญชีผู้ใช้ (Suspend/Ban Account)

| หัวข้อ | รายละเอียด |
|-------|-----------|
| **รหัสอ้างอิง** | FN-ADM-003 |
| **Actor หลัก** | ผู้ดูแลระบบ |
| **Actor รอง** | สมาชิกที่ถูกระงับ, ระบบยืนยันตัวตน (Auth Service) |
| **คำอธิบายโดยย่อ** | ผู้ดูแลระบบระงับหรือยกเลิกการระงับบัญชีสมาชิกที่ทำผิดกฎ ทำให้บัญชีนั้นเข้าสู่ระบบ (UC-31) หรือรีเซ็ตรหัสผ่าน (UC-38) ไม่ได้จนกว่าจะถูกยกเลิกการระงับ (use case นี้ถูกอ้างอิงจาก UC-31 และ UC-38 มาตั้งแต่ต้น แต่ไม่เคยถูกเขียนไว้จริงเพราะไฟล์ต้นฉบับขาดหายไปตั้งแต่จุดนี้ — เพิ่มเข้ามาในรอบตรวจรีวิวที่ 7) |
| **เงื่อนไขก่อนเริ่มทำงาน (Pre-condition)** | ผู้ดูแลระบบเข้าสู่ระบบด้วยบัญชีที่มีสิทธิ์ผู้ดูแล และมีบัญชีสมาชิกเป้าหมายอยู่ในระบบ |
| **ลำดับเหตุการณ์หลัก (Basic Flow)** | **ระงับบัญชี:** 1. เปิดหน้าจัดการบัญชีผู้ใช้ ค้นหาบัญชีเป้าหมาย<br>2. กด "ระงับบัญชี" ระบุเหตุผลและระยะเวลาระงับ (เลือกจากตัวเลือกมาตรฐาน หรือถาวร — ดู Business Rules)<br>3. กดยืนยัน ระบบเพิกถอน token ที่ใช้งานอยู่ของบัญชีนั้นทันที และบันทึกเหตุผล/ระยะเวลาไว้<br>4. ส่งอีเมลแจ้งเหตุผลและระยะเวลาการระงับไปยังบัญชีที่ถูกระงับ<br><br>**ยกเลิกการระงับ:** 1. เปิดหน้าจัดการบัญชีผู้ใช้ ค้นหาบัญชีที่ถูกระงับอยู่<br>2. กด "ยกเลิกการระงับ"<br>3. ระบบขอคำยืนยัน<br>4. ยืนยันแล้วบัญชีกลับมาเข้าสู่ระบบ (UC-31) และรีเซ็ตรหัสผ่าน (UC-38) ได้ตามปกติ |
| **ลำดับเหตุการณ์ทางเลือก (Alternative/Exception Flow)** | ระงับแบบชั่วคราวครบกำหนดเวลา → ระบบยกเลิกการระงับให้อัตโนมัติโดยไม่ต้องมีผู้ดูแลกดยกเลิก; กดยกเลิกกลางทางระหว่างกรอกเหตุผล/ระยะเวลา → ไม่มีการเปลี่ยนแปลงสถานะบัญชี |
| **เงื่อนไขหลังทำงานสำเร็จ (Post-condition)** | บัญชีเปลี่ยนสถานะเป็น "ถูกระงับ" หรือกลับเป็น "ปกติ" ตามการดำเนินการ ระหว่างถูกระงับ บัญชีจะเข้าสู่ระบบ (UC-31) หรือรีเซ็ตรหัสผ่าน (UC-38) ไม่ได้ ตามที่ระบุไว้ใน alt flow ของทั้งสอง use case นั้น |
| **ความสัมพันธ์กับ Use Case อื่น** | เกี่ยวข้องกับ UC-31 และ UC-38 (เป็นเงื่อนไขปฏิเสธการเข้าถึงในทั้งสอง flow — ถูกอ้างอิงจากทั้งคู่มาตั้งแต่ก่อนเพิ่ม use case นี้) |
| **กฎทางธุรกิจที่เกี่ยวข้อง (Business Rules)** | **ระยะเวลาระงับ — ค่าตั้งต้น รอทีมงาน/ฝ่ายปฏิบัติการยืนยัน:** ตัวเลือกมาตรฐาน 1 วัน / 7 วัน / 30 วัน หรือถาวร (ไม่มีกำหนด) — ระงับถาวรต้องยกเลิกด้วยมือโดยผู้ดูแลเท่านั้น ระงับแบบมีกำหนดเวลาจะยกเลิกอัตโนมัติเมื่อครบกำหนด |
| **เกี่ยวข้องกับ Service/Endpoint อื่น** | - |
