# Pomo RPG · แอปพลิเคชันจับเวลาโฟกัสในรูปแบบเกม (Focus) — Use Case Description

> ตามรูปแบบ [`template_usecase_description.md`](./template_usecase_description.md) — 1 use case ในเอกสารนี้ตรงกับ 1 use case ใน `usecase_module1.puml` ถึง `usecase_module7.puml` (ดู `usecase_overview.puml` สำหรับภาพรวม) จัดกลุ่มเป็น Epic ภายในแต่ละโมดูล

> **หมายเหตุทั่วไป:** ทุก use case ที่ Actor หลักเป็นสมาชิกหรือผู้ดูแลระบบ ถือว่าต้องล็อกอินอยู่ก่อนเสมอ (ดู UC-08) จึงไม่ระบุเงื่อนไขนี้ซ้ำใน Pre-condition ของแต่ละ use case อีก เว้นแต่มีเงื่อนไขเพิ่มเติมนอกเหนือจากการล็อกอินที่จำเป็นต้องระบุ

## M1 · บัญชีและการเริ่มต้นใช้งาน

### E1 · การสมัครสมาชิกและสร้างอวตาร

#### UC-01 · การสมัครสมาชิก

| หัวข้อ | รายละเอียด |
|-------|-----------|
| **รหัสอ้างอิง** | FN-ACC-001 |
| **Actor หลัก** | Guest |
| **Actor รอง** | ระบบฐานข้อมูลผู้ใช้ |
| **คำอธิบายโดยย่อ** | Guest สร้างบัญชีผู้ใช้ใหม่เพื่อเปลี่ยนสถานะเป็นสมาชิก ก่อนจะสามารถเข้าถึงฟีเจอร์อื่นใดของแอปได้ นี่คือ use case เดียวที่ Guest ทำได้ |
| **เงื่อนไขก่อนเริ่มทำงาน (Pre-condition)** | ผู้ใช้ยังไม่มีบัญชีในระบบ (สถานะเป็น Guest) และติดตั้งแอปเรียบร้อยแล้ว |
| **ลำดับเหตุการณ์หลัก (Basic Flow)** | 1. Guest กดปุ่ม "สมัครสมาชิก"<br>2. ระบบแสดงฟอร์มกรอกข้อมูล<br>3. Guest กรอกข้อมูลครบถ้วนและกดยืนยัน<br>4. ระบบตรวจสอบความถูกต้องของข้อมูล<br>5. ระบบสร้างบัญชีใหม่ เปลี่ยนสถานะเป็นสมาชิก และพาไปหน้าสร้างอวตาร |
| **ลำดับเหตุการณ์ทางเลือก (Alternative/Exception Flow)** | ข้อมูลไม่ครบ/ไม่ถูกรูปแบบ → แจ้งเตือนให้แก้ไข; อีเมล/ชื่อผู้ใช้ถูกใช้แล้ว → แจ้งเตือนให้แก้ไข; กดยกเลิกกลางคัน → ล้างข้อมูล กลับหน้าแรก ยังเป็น Guest |
| **เงื่อนไขหลังทำงานสำเร็จ (Post-condition)** | สร้างบัญชีใหม่และบันทึกลงฐานข้อมูล สถานะเปลี่ยนจาก Guest เป็นสมาชิก และพาไปยัง UC-02 |
| **ความสัมพันธ์กับ Use Case อื่น** | `<<include>>` UC-02 (บัญชีใหม่ทุกบัญชีต้องสร้างอวตารเสมอ) |

#### UC-02 · การสร้างอวตาร

| หัวข้อ | รายละเอียด |
|-------|-----------|
| **รหัสอ้างอิง** | FN-ACC-002 |
| **Actor หลัก** | สมาชิก |
| **Actor รอง** | ระบบฐานข้อมูลอวตาร |
| **คำอธิบายโดยย่อ** | สมาชิกที่สมัครใหม่เลือกและสร้างอวตารเริ่มต้นของตัวเอง ก่อนเข้าใช้งานฟีเจอร์หลักของแอป |
| **เงื่อนไขก่อนเริ่มทำงาน (Pre-condition)** | สมัครสมาชิกสำเร็จแล้ว (UC-01) และบัญชียังไม่มีอวตาร |
| **ลำดับเหตุการณ์หลัก (Basic Flow)** | 1. ระบบแสดงหน้าเลือกรูปแบบอวตารเริ่มต้น<br>2. สมาชิกเลือกรูปแบบที่ต้องการ<br>3. สมาชิกตั้งชื่ออวตาร<br>4. สมาชิกกดยืนยันการสร้าง<br>5. ระบบบันทึกอวตาร (ชื่อที่ตั้ง, HP เต็ม, เลเวล 1)<br>6. ระบบพาสมาชิกเข้าสู่หน้าหลัก |
| **ลำดับเหตุการณ์ทางเลือก (Alternative/Exception Flow)** | สมาชิกยังไม่แน่ใจ ต้องการดูตัวเลือกเพิ่มเติม → เลื่อนดูรูปแบบอื่นได้ก่อนยืนยัน; ไม่กรอกชื่ออวตาร → แจ้งเตือนให้กรอก; กรอกชื่ออวตารสั้นกว่า 4 ตัวอักษรหรือเกิน 16 ตัวอักษร → แจ้งเตือน ไม่ให้ยืนยัน; ชื่ออวตารซ้ำกับที่มีอยู่แล้วในระบบ → แจ้งเตือนให้แก้ไข |
| **เงื่อนไขหลังทำงานสำเร็จ (Post-condition)** | อวตารถูกสร้างพร้อมชื่อที่ตั้งไว้ ที่ HP เต็ม (Max HP = 100 หน่วย ตาม Base Max HP ของเลเวล 1 — ดู Business Rules ของ UC-07) เลเวล 1 ช่องสวมใส่ทั้ง 4 ช่อง (หัว, ลำตัว, ขา, อาวุธ) ยังว่างเปล่า (ต้องฟาร์มไอเทมแต่งตัวจากกล่องสุ่มก่อน ดู UC-10, แล้วสวมใส่ผ่าน UC-14) แก้ไขชื่ออวตารภายหลังได้ผ่านหน้าตั้งค่า (UC-04) สมาชิกเข้าสู่หน้าหลักได้ |
| **ความสัมพันธ์กับ Use Case อื่น** | ถูก `<<include>>` จาก UC-01 |
| **กฎทางธุรกิจที่เกี่ยวข้อง (Business Rules)** | **ชื่ออวตาร:** ความยาว 4-16 ตัวอักษร ใช้ได้ทั้งภาษาไทยและภาษาอังกฤษ (หรือผสมกัน) ต้องไม่ซ้ำกับชื่ออวตารของสมาชิกคนอื่นที่มีอยู่แล้วในระบบ (unique ทั้งระบบ) — กฎเดียวกันนี้ใช้กับการแก้ไขชื่ออวตารภายหลังผ่าน UC-04 ด้วย |

|-------|-----------|

| **Actor หลัก** | สมาชิก |
| **Actor รอง** | - |
| **คำอธิบายโดยย่อ** | สมาชิกเพิ่มหรือลบรายการเป้าหมายที่ต้องการโฟกัส เพื่อใช้เลือกตอนเริ่มจับเวลา (UC-07) — ไม่มีฟีเจอร์แก้ไขชื่อรายการที่เพิ่มไปแล้ว หากต้องการเปลี่ยนชื่อต้องลบรายการเดิมแล้วเพิ่มใหม่แทน |

### E3 · การจัดการบัญชีและการตั้งค่า

| **ลำดับเหตุการณ์ทางเลือก (Alternative/Exception Flow)** | ชื่อผู้ใช้ถูกใช้แล้ว → แจ้งเตือนให้แก้ไข; ไม่กรอกชื่ออวตาร (เว้นว่าง) → แจ้งเตือนให้กรอก ไม่บันทึก; กรอกชื่ออวตารสั้นกว่า 4 ตัวอักษรหรือเกิน 16 ตัวอักษร → แจ้งเตือน ไม่บันทึก; ชื่ออวตารใหม่ซ้ำกับที่มีอยู่แล้วในระบบ → แจ้งเตือนให้แก้ไข ไม่บันทึก; รหัสผ่านเดิมไม่ถูกต้อง → แจ้งเตือน ไม่บันทึก; รหัสผ่านใหม่ไม่ตรงกับช่องยืนยัน → แจ้งเตือน |

| **ความสัมพันธ์กับ Use Case อื่น** | - |
| **กฎทางธุรกิจที่เกี่ยวข้อง (Business Rules)** | **ชื่ออวตาร:** ใช้กฎเดียวกับตอนสร้างครั้งแรก (ดู Business Rules ของ UC-02) — ความยาว 4-16 ตัวอักษร ภาษาไทยหรืออังกฤษ (หรือผสมกัน) และต้องไม่ซ้ำกับชื่ออวตารของสมาชิกคนอื่น |

| **ความสัมพันธ์กับ Use Case อื่น** | ไม่มี (ข้อมูลที่บันทึกถูกใช้ต่อใน UC-25 เชิงข้อมูลเท่านั้น ไม่ใช่ include/extend เชิง flow) |

## M2 · เซสชันโฟกัส (Core Loop)

### E1 · วงจรเซสชันโฟกัส

#### UC-07 · เริ่มจับเวลาโฟกัส (ตีบอส)

|-------|-----------|
| **รหัสอ้างอิง** | FN-SES-001 |
| **Actor หลัก** | สมาชิก |
| **Actor รอง** | ระบบจับเวลา, ระบบบอส |
| **คำอธิบายโดยย่อ** | สมาชิกเลือกรายการที่จะโฟกัสและตั้งเวลาเริ่มจับเวลา ระบบจะจำลองเป็นการต่อสู้กับบอส ครอบคลุมไปถึงการพัก การยกเลิก และการแจกรางวัล |
| **เงื่อนไขก่อนเริ่มทำงาน (Pre-condition)** | สมาชิกล็อกอิน มีอวตาร และมีรายการสิ่งที่โฟกัส (UC-03) |
| **ลำดับเหตุการณ์หลัก (Basic Flow)** | 1. เลือกรายการจาก UC-03 และตั้งระยะเวลาโฟกัส/พักส่วนตัว (ระบบ pre-fill ค่าจาก UC-05 ถ้าเคยตั้งไว้ หรือใช้ค่า default Focus 25 นาที Break 5 นาที ถ้าไม่เคยตั้งค่า สามารถแก้ไขเป็นรายเซสชันได้เสมอ ช่วง Focus 1-600 นาที Break 1-60 นาที — ยกเว้นกรณีเริ่มเซสชันขณะอยู่ในห้องโฟกัส ให้ใช้ระยะเวลาร่วมของห้องที่เจ้าของห้องตั้งไว้แทนโดยแก้ไขเป็นรายเซสชันไม่ได้ ดู Business Rules ของ UC-16)<br>2. กดเริ่มจับเวลา ระบบจำลองบอส (HP = เวลา × 100)<br>3. แสดงนาฬิกาถอยหลัง เมื่อเวลาครบ เซสชันสำเร็จ คืน HP อวตารเต็ม 100% (ดูกฎ 'การคืน HP อวตารเมื่อเซสชันสำเร็จ' ด้านล่าง) และบันทึกประวัติ<br>4. ได้รับรางวัล EXP ตามจำนวนนาทีที่โฟกัสสำเร็จ (ทุกเซสชันที่จบสำเร็จ ไม่มีขั้นต่ำ — ดูสูตรใน Business Rules) และหากเซสชัน ≥ 20 นาที จะได้รับกล่องสุ่มเพิ่มอีก 1 ใบเก็บเข้ากระเป๋าด้วย (ชนิดกล่องถูกกำหนดตายตัวตามความยาวเซสชัน ณ ตอนนี้ — ดู Business Rules ของ UC-10) **ยังเปิดกล่องไม่ได้ทันที ต้องไปเปิดจากกระเป๋าเท่านั้นผ่าน UC-10**<br>5. ระบบเริ่มนับเวลาพักอัตโนมัติตามเวลาที่ตั้งไว้<br>6. เมื่อจบช่วงพักสามารถเริ่มเซสชันใหม่ได้ หากไม่เริ่ม ระบบจะนับเวลาพักเกินกำหนดเป็นสถิติ |
| **ลำดับเหตุการณ์ทางเลือก (Alternative/Exception Flow)** | **ยกเลิกกลางทาง:** กดยกเลิกเซสชันกลางทาง (กดค้าง) → HP ลดลง 10 หน่วย (ค่าคงที่ ไม่คำนวณตามสัดส่วนเวลาที่โฟกัสไปแล้ว — ดู Business Rules "บทลงโทษยกเลิกเซสชันกลางทาง") บันทึกว่ายกเลิก ไม่ได้รับกล่องสุ่ม<br>**หยุดชั่วคราวนอกการควบคุมของผู้ใช้ (Interruption):** แอปพลิเคชันถูกปิด, มีสายโทรศัพท์เข้า, หรือแอปหลุดโฟกัส (เช่น สลับไปแอปอื่น) กลางเซสชัน → ระบบหยุดนาฬิกาถอยหลังไว้ชั่วคราว ถ้าผู้ใช้กลับเข้าแอปภายใน 2 นาที (pause) ระบบนับเวลาต่อจากจุดที่ค้างไว้ให้อัตโนมัติ เซสชันดำเนินต่อได้ตามปกติจนครบเวลา **ถ้าเกิน 2 นาที → ถือว่าออกกลางคัน** HP ลดลง 10 หน่วย (ค่าคงที่ เช่นเดียวกับกรณี "ยกเลิกกลางทาง") บันทึกว่ายกเลิก ไม่ได้รับกล่องสุ่ม<br>**พักเกินกำหนด:** หมดเวลาพักแต่ยังไม่เริ่มใหม่ → ระบบส่ง push notification + เสียงแจ้งเตือน (ภาษาไทย) แจ้งว่าเวลาพักหมดแล้ว (แม้แอปอยู่ background) จากนั้นเริ่มนับเวลาพักเกินไปเรื่อยๆ เพื่อเป็นสถิติ<br>**จบช่วงพักก่อนกำหนด:** กดข้ามช่วงพัก → เริ่มเซสชันใหม่ได้ทันที<br>**อวตารหมดสติ:** HP อวตารเท่ากับ 0 ก่อนเริ่ม → เข้าสถานะ "หมดสติ" ล็อกฟีเจอร์จนกว่าจะชุบชีวิต (UC-15) |
| **เงื่อนไขหลังทำงานสำเร็จ (Post-condition)** | เซสชันถูกบันทึกลงประวัติ สถิติเวลาพัก/เบรกเกิน รางวัล และ HP อัพเดตตามกฎทางธุรกิจ |
| **ความสัมพันธ์กับ Use Case อื่น** | ถูก `<<include>>` โดย UC-15 |
| **กฎทางธุรกิจที่เกี่ยวข้อง (Business Rules)** | **โหมดโฟกัสคนเดียว (Single-player Focus):**<br>- การนับเวลาจับเวลาของอวตารนับถอยหลังแบบเรียลไทม์ (Real-time countdown) สมาชิกเห็นเวลาลดลงวินาทีต่อวินาทีบนหน้าจอ<br>- การคำนวณหักล้างค่าพลังชีวิต (HP) ของบอสจะไม่หักทุกวินาที แต่ระบบจะหักลดเลือดบอสทุกๆ 5 นาทีที่ผู้ใช้โฟกัสสำเร็จ (ส่งดาเมจไปหักเลือดบอสเป็นก้อนทุกๆ 5 นาที)<br>- *ข้อเสนอแนะ UX (ไม่ใช่ business logic):* ฝั่ง client ควรแสดงแถบเลือดบอสไหลลดลงต่อเนื่องแบบ interpolate ระหว่าง 2 จุดที่หักจริง แทนการกระโดดเป็นก้อนทุก 5 นาที เพื่อให้สอดคล้องกับนาฬิกาถอยหลังที่เดินทุกวินาที<br><br>**Max HP และ HP Decay — ยืนยันแล้วโดยทีมออกแบบเกม (แทนที่ค่าตั้งต้นเดิม):**<br>- **Base Max HP:** อวตารเริ่มต้นที่ Max HP = 100 หน่วย ที่เลเวล 1 (ตรงกับ Post-condition ของ UC-02)<br>- **Level Scaling:** ทุกครั้งที่เลเวลอัพ (ตามสูตร EXP ด้านล่าง) Max HP เพิ่มขึ้น +5 หน่วยทันที โดย Current HP ไม่เปลี่ยนแปลง (ไม่มีการฟื้นฟู HP จากการเลเวลอัพ)<br>- **HP Decay Rate:** อวตารเสีย HP 2 หน่วย ต่อทุก 1 ชั่วโมงที่สมาชิกไม่ได้อยู่ในเซสชันโฟกัส นับจากเวลาสิ้นสุดเซสชันล่าสุด (หรือเวลาสร้างอวตารถ้ายังไม่เคยโฟกัสเลย) — ปรับจากค่าตั้งต้นเดิม 5 หน่วย/ชั่วโมง<br>- ไม่มีการหัก HP ระหว่างที่ไอเทมโล่เวลา (UC-13) มีผลอยู่<br>- เมื่อ HP อวตารแตะ 0 → เข้าสถานะ "หมดสติ" ตามลำดับเหตุการณ์ทางเลือกด้านบน (กลไกหมดสติ/ชุบชีวิตยังคงเป็นไปตาม UC-15 แบบเดิมทุกประการ ไม่มีการเปลี่ยนแปลง)<br><br>**บทลงโทษยกเลิกเซสชันกลางทาง (Cancel Penalty) — ยืนยันแล้ว:**<br>- เมื่อยกเลิกเซสชันกลางทาง หรือหยุดเกิน 2 นาทีจน "ถือว่าออกกลางคัน" (ดู alt flow ด้านบน) อวตารเสีย HP ทันที 10 หน่วย เป็นค่าคงที่ ไม่คำนวณตามสัดส่วนเวลาที่โฟกัสไปแล้วก่อนยกเลิก<br>- หาก HP ปัจจุบันน้อยกว่า 10 ให้ HP เหลือ 0 (ไม่ติดลบ) และเข้าสถานะ "หมดสติ" ทันทีตามกฎด้านบน<br><br>**สูตร EXP และเลเวลอัพ — ยืนยันแล้ว (ดูหมายเหตุข้อ 16):**<br>- ได้รับ EXP = จำนวนนาทีที่โฟกัสสำเร็จ × 2 ต่อเซสชัน (เฉพาะเซสชันที่จบสำเร็จ ไม่นับเซสชันที่ยกเลิกกลางทาง)<br>- เลเวลอัพเมื่อ EXP สะสมถึงเกณฑ์ของเลเวลถัดไป (สูตรขั้นบันไดที่ยืนยันแล้ว: เลเวล N ต้องใช้ N × 100 EXP สะสม) เพดานเลเวลสูงสุด = 80 — เลเวลอัพส่งผลต่อ Max HP แล้วตามกฎ Level Scaling ด้านบน (ยืนยันแล้ว ไม่ใช่ค่าตั้งต้นอีกต่อไป)<br>- **EXP ที่เลเวลสูงสุด (Level Cap):** เมื่ออวตารถึงเลเวล 80 แล้ว หยุดสะสม EXP ทันที เซสชันที่จบสำเร็จหลังจากนั้นยังคงได้รับรางวัลอื่น (กล่องสุ่มถ้าเข้าเงื่อนไข) ตามปกติ แต่ไม่มี EXP เพิ่มเข้าระบบอีก<br><br>**การคืน HP อวตารเมื่อเซสชันสำเร็จ (Session Success Heal) — ยืนยันแล้ว (ดูหมายเหตุข้อ 16):**<br>- อวตารได้รับการคืน HP เต็ม (เท่ากับ Max HP ปัจจุบันของอวตาร ซึ่งอาจมากกว่า 100 หากเลเวลอัพแล้ว) ทุกครั้งที่จบเซสชันสำเร็จ ไม่ว่าเซสชันจะสั้นหรือยาวแค่ไหน ตราบใดที่โฟกัสจนครบเวลาที่ตั้งไว้<br>- ไม่มีผลกับเซสชันที่ถูกยกเลิกกลางทาง (ดู alt flow "ยกเลิกกลางทาง") |

| **ลำดับเหตุการณ์หลัก (Basic Flow)** | 1. เปิดกระเป๋าไอเทม (UC-12) เลือกกล่องสุ่มที่มีอยู่จริงในกระเป๋า (แสดงเฉพาะชนิดที่มี)<br>2. กดเปิดกล่องสุ่มที่เลือก<br>3. ระบบสุ่มไอเทม 1 ชิ้นตามอัตราดรอปของชนิดกล่องนั้น (ตามความยาวเซสชันตอนที่ได้รับกล่อง ไม่ใช่ตอนเปิด)<br>4. ตรวจสอบเงื่อนไข Pity System ก่อนสรุปผล (ดู Business Rules)<br>5. นำกล่องสุ่มที่เปิดออกจากกระเป๋า และเพิ่มไอเทม 1 ชิ้นที่ได้เข้ากระเป๋า<br>6. แสดงผลไอเทมที่ได้รับ พร้อมความคืบหน้าตัวนับ Pity ปัจจุบัน (เช่น "อีก 3 กล่องจนการันตีของหายากมาก/ตำนาน") เพื่อความโปร่งใส |
| **ลำดับเหตุการณ์ทางเลือก (Alternative/Exception Flow)** | เงื่อนไข Pity ถูกกระตุ้น → การันตีดรอปของหายากมาก/ตำนานแทนผลสุ่มปกติ; ไม่มีกล่องสุ่มเหลือในกระเป๋าเลย → ไม่แสดงตัวเลือกเปิดกล่อง พร้อมข้อความแนะนำให้ไปโฟกัส (ลิงก์ไป UC-07) |
| **เงื่อนไขหลังทำงานสำเร็จ (Post-condition)** | ไอเทมที่สุ่มได้ถูกเพิ่มเข้ากระเป๋าของสมาชิก (UC-12) |

| **คำอธิบายโดยย่อ** | สมาชิกสวมใส่หรือถอดไอเทมแต่งตัวให้กับอวตารของตัวเอง แยกตามช่องสวมใส่ 4 ช่อง (หัว, ลำตัว, ขา, อาวุธ) โดยไอเทมได้มาจากกล่องสุ่ม (UC-10) หรือการผสม (UC-11) |
| **เงื่อนไขก่อนเริ่มทำงาน (Pre-condition)** | สมาชิกมีไอเทมประเภทแต่งตัวอย่างน้อย 1 ชิ้นในกระเป๋า (UC-12) |
| **ลำดับเหตุการณ์หลัก (Basic Flow)** | **สวมใส่ (Equip):** 1. เปิดหน้าแต่งตัวอวตาร ระบบแสดงช่องสวมใส่ 4 ช่อง ได้แก่ หัว, ลำตัว, ขา, และอาวุธ<br>2. เลือกช่องที่ต้องการแต่ง แล้วเลือกไอเทมแต่งตัวจากกระเป๋าที่ตรงกับประเภทของช่องนั้น (ทำได้ทีละช่องหรือหลายช่องก่อนยืนยันรวมก็ได้ ระบบแสดงตัวอย่างอวตารทันทีทุกครั้งที่เลือก)<br>3. กดยืนยัน ไอเทมที่เลือกถูกสวมใส่จริง<br><br>**ถอด (Unequip):** 1. เปิดหน้าแต่งตัวอวตาร เลือกช่องที่มีไอเทมสวมอยู่<br>2. กดถอด<br>3. ไอเทมกลับเข้ากระเป๋าทันที ช่องนั้นว่างลง (ไม่ต้องกดยืนยันซ้ำ) |
| **ลำดับเหตุการณ์ทางเลือก (Alternative/Exception Flow)** | ไม่มีไอเทมแต่งตัวประเภทที่ตรงกับช่องนั้นในกระเป๋า → ช่องนั้นแสดงเป็นว่าง กดเลือกไม่ได้; ไอเทมที่เลือกไม่ตรงประเภทกับช่อง (เช่น นำไอเทมประเภทอาวุธไปใส่ช่องหัว) → ระบบปฏิเสธ ไม่ให้สวมใส่ |
| **เงื่อนไขหลังทำงานสำเร็จ (Post-condition)** | อวตารแสดงผลไอเทมที่สวมใส่ล่าสุดในทุกหน้าจอที่แสดงอวตาร (หน้าหลัก, ห้องโฟกัส, โปรไฟล์) |
| **ความสัมพันธ์กับ Use Case อื่น** | ไม่มี (ใช้ข้อมูลไอเทมร่วมกับ UC-12); ถูกอ้างอิงจาก UC-22 (แสดงไอเทมที่สวมใส่อยู่บนโปรไฟล์เพื่อน) |

## M4 · อวตารและความต่อเนื่อง (ระบบชุบชีวิต)

> เดิมโมดูลนี้เคยมี use case แยกสำหรับกลไกเลเวลอัพ, HP Decay, และสถานะหมดสติ — ทั้งหมดถูกยุบรวมเป็นกฎทางธุรกิจของ UC-07 (โมดูล 2) แล้ว ส่วนกลไก Streak เดิมถูกยุบรวมเป็นกฎทางธุรกิจของ UC-08 (โมดูล 2) โดยเฉพาะ (ไม่ใช่ส่วนหนึ่งของ UC-07) เพราะทั้งหมดไม่มี actor ริเริ่มเอง — เหลือ UC-15 (ระบบชุบชีวิต) เป็น use case เดียวในโมดูลนี้ที่สมาชิกลงมือทำเองมีเป้าหมายชัดเจน

### E1 · ระบบชุบชีวิต

#### UC-15 · ระบบชุบชีวิต (Revival System)

| หัวข้อ | รายละเอียด |
|-------|-----------|
| **รหัสอ้างอิง** | FN-AVT-001 |
| **Actor หลัก** | สมาชิก |
| **Actor รอง** | - |
| **คำอธิบายโดยย่อ** | สมาชิกชุบชีวิตอวตารที่หมดสติกลับมาได้ โดยต้องทำภารกิจโฟกัสตามที่ระบบกำหนดให้สำเร็จในเซสชันเดียว |
| **เงื่อนไขก่อนเริ่มทำงาน (Pre-condition)** | อวตารอยู่ในสถานะ "หมดสติ" (ตามกฎทางธุรกิจ HP=0 ที่ระบุใน UC-07) |
| **ลำดับเหตุการณ์หลัก (Basic Flow)** | 1. เปิดแอปเห็นหน้าจอชุบชีวิตพร้อมภารกิจ (ระยะเวลาภารกิจตามที่ระบบกำหนด — ดู Business Rules)<br>2. เริ่มเซสชันโฟกัสภารกิจชุบชีวิต (เชื่อมกับ UC-07)<br>3. ตรวจสอบว่าเซสชันจบสำเร็จครบตามเงื่อนไข<br>4. ชุบชีวิตอวตาร ปลดล็อกฟีเจอร์<br>5. ลดเลเวลลง 2 ระดับจากเลเวลปัจจุบัน (ต่ำสุดคือเลเวล 1 ห้ามติดลบ) รีเซ็ต EXP ของเลเวลใหม่เป็น 0 ปรับ Max HP ตาม Level Scaling ของเลเวลใหม่ (ดู UC-07) แล้วคืน HP เต็มเท่ากับ Max HP ใหม่นั้น พร้อมแจ้งยืนยันความสำเร็จ |
| **ลำดับเหตุการณ์ทางเลือก (Alternative/Exception Flow)** | ยกเลิกเซสชันภารกิจกลางทาง (ตาม alt flow "ยกเลิกกลางทาง" ของ UC-07) → ภารกิจไม่สำเร็จ อวตารยังหมดสติ ลองใหม่ได้ไม่จำกัดจำนวนครั้ง |
| **เงื่อนไขหลังทำงานสำเร็จ (Post-condition)** | อวตารกลับมามีชีวิต เลเวลลดลง 2 ระดับจากก่อนหมดสติ (ต่ำสุดคือเลเวล 1) EXP ของเลเวลใหม่เริ่มจาก 0 Max HP ปรับลดตาม Level Scaling ของเลเวลใหม่ (ดู Business Rules ของ UC-07) และ HP ปัจจุบันถูกคืนเต็มเท่ากับ Max HP ใหม่นั้น ฟีเจอร์ที่ถูกล็อกถูกปลดล็อก (ไอเทมในกระเป๋ายังคงอยู่) |
| **ความสัมพันธ์กับ Use Case อื่น** | `<<include>>` UC-07 (ภารกิจใช้กลไกเซสชันโฟกัสเดียวกัน) |
| **กฎทางธุรกิจที่เกี่ยวข้อง (Business Rules)** | **เงื่อนไขภารกิจชุบชีวิต — ยืนยันแล้ว (ดูหมายเหตุข้อ 16):** ภารกิจคือเซสชันโฟกัสความยาวคงที่ 20 นาที (ไม่อิงกับความยาวเซสชันที่ทำให้อวตารหมดสติ) นับเป็นเซสชันปกติผ่านกลไก UC-07 แต่ระหว่างสถานะ "หมดสติ" จะไม่ได้รับกล่องสุ่ม/EXP จากเซสชันนี้ (รางวัลปกติกลับมาใช้งานได้หลังชุบชีวิตสำเร็จเท่านั้น)<br><br>**บทลงโทษการชุบชีวิต (Revival Penalty) — ยืนยันแล้วโดยทีมออกแบบเกม (แทนที่การรีเซ็ตกลับเลเวล 1 ทั้งหมดแบบเดิม):** ลดเลเวลลง 2 ระดับจากเลเวลก่อนหมดสติ หากเลเวลปัจจุบันคือ 1 หรือ 2 (ลดแล้วจะติดลบหรือเป็น 0) ให้หยุดที่เลเวล 1 เป็นอย่างต่ำ ไม่ติดลบ (กรณีนี้เลเวลจริงจะลดลงแค่ 0 หรือ 1 ระดับเท่านั้น ไม่ใช่ 2 ระดับเต็ม) — EXP ของเลเวลใหม่เริ่มนับจาก 0 ใหม่ (ไม่ใช่ EXP สะสมเดิม) Max HP คำนวณใหม่ทั้งหมดจาก Base Max HP + (เลเวลใหม่หลังปรับลด − 1) × 5 หน่วย/ระดับ ตาม Level Scaling ของ UC-07 (ไม่ใช่การลดคงที่ 10 หน่วยเสมอไป — กรณีเลเวลถูก clip ที่เลเวล 1 ผลต่าง Max HP ที่แท้จริงจะน้อยกว่า 10 ตามจำนวนระดับที่ลดจริง) |

> **หมายเหตุจากการรีวิว (แก้ไขแล้ว):** เดิมข้อเสนอแนะนี้เตือนว่าการรีเซ็ตเลเวล/EXP กลับศูนย์ทั้งหมดเป็น penalty ที่แรงเกินไปในเชิง game design และแนะนำให้ลดโทษเป็นลดเลเวลบางส่วนแทน — ทีมออกแบบเกมยืนยันแล้วให้เปลี่ยนเป็น **ลดเลเวลลง 2 ระดับ** ตามที่ระบุใน Business Rules ด้านบน จึงถือว่าข้อเสนอแนะนี้ถูกรับไปปรับใช้แล้ว

## M5 · ห้องโฟกัสและสังคม

### E1 · ห้องโฟกัส

#### UC-16 · จัดการห้องโฟกัส

| หัวข้อ | รายละเอียด |
|-------|-----------|
| **รหัสอ้างอิง** | FN-ROOM-001 |
| **Actor หลัก** | สมาชิก (Member) |
| **Actor รอง** | ระบบสร้างลิงก์เชิญ, ระบบนับเวลาถอยหลังห้อง, ระบบสรุปสถิติ |
| **คำอธิบายโดยย่อ** | การดำเนินการแบบครบวงจรของห้องโฟกัสแบบแข่งขัน ครอบคลุมการสร้างห้องหรือเข้าร่วมห้องด้วยลิงก์/รหัสเชิญ การเริ่มห้องเพื่อเปิดกรอบเวลาแข่งขัน และการปิดห้อง (ไม่ว่าจะด้วยเวลาห้องหมดอัตโนมัติ หรือเจ้าของห้องกดปิดเอง) เพื่อสรุปสถิติว่าใครโฟกัสสะสมได้มากที่สุดภายในเวลาห้อง (การออกจากห้องและการเตะสมาชิกแยกเป็น UC-18 และ UC-19) |
| **เงื่อนไขก่อนเริ่มทำงาน (Pre-condition)** | เฉพาะ flow "สร้างห้อง" และ "เข้าร่วมห้อง" เพิ่มเติมว่าต้องไม่ได้อยู่ในห้องโฟกัสอื่นค้างอยู่ (ต้องออกจากห้องเดิมก่อนผ่าน UC-18 หากมี); ส่วน flow "เริ่มห้อง" "จัดการระยะเวลาร่วมของห้อง" และ "ปิดห้อง" ผู้ทำต้องเป็นเจ้าของห้องที่อยู่ในห้องนั้นอยู่แล้ว |
| **ลำดับเหตุการณ์หลัก (Basic Flow)** | **สร้างห้อง:** 1. กด "สร้างห้องโฟกัส"<br>2. ตั้งชื่อห้อง กำหนดเวลาห้อง (Room Duration — อายุของห้องทั้งหมดเป็นนาที เช่น 300 นาที) และกำหนดระยะเวลาโฟกัส/พักร่วมที่สมาชิกทุกคนในห้องจะใช้เหมือนกัน (ค่านี้แทนที่ค่าเริ่มต้นส่วนตัวจาก UC-05 ขณะเริ่มเซสชันภายในห้องนี้)<br>3. กดยืนยันสร้างห้อง ระบบสร้างลิงก์/รหัสเชิญให้ และแสดงรหัสเชิญ (เลขโค้ด) พร้อมลิงก์เชิญนั้นบนหน้าจอให้เจ้าของห้องเห็นทันที (ดูซ้ำได้ภายหลังจากหน้าห้องตลอดเวลาที่ห้องยังไม่ปิด)<br>4. แชร์ลิงก์/รหัสเชิญให้เพื่อนที่ต้องการชวน<br><br>**เข้าร่วมห้อง:** 1. กด "เข้าร่วมห้อง"<br>2. กรอกรหัสเชิญ หรือกดลิงก์เชิญโดยตรง<br>3. ระบบตรวจสอบความถูกต้องของรหัสและจำนวนสมาชิกในห้อง และตรวจว่าห้องยังไม่ถูกเริ่ม (ยังไม่กด "เริ่มห้อง")<br>4. เข้าร่วมห้องสำเร็จ เห็นหน้าจอห้องพร้อมรายชื่อสมาชิกคนอื่นและระยะเวลาโฟกัส/พักร่วมของห้องที่เจ้าของห้องตั้งไว้ — เข้าร่วมได้เฉพาะห้องที่ยังไม่เริ่มนับเวลาถอยหลังเท่านั้น (ห้องที่เริ่มไปแล้วเข้าร่วมเพิ่มไม่ได้)<br><br>**เริ่มห้อง:** 1. เจ้าของห้องกด "เริ่มห้อง" เมื่อพร้อม<br>2. ระบบเริ่มนับเวลาถอยหลังของห้องตามเวลาห้องที่ตั้งไว้ (เช่น 300 นาที) และแสดงเวลาที่เหลือให้สมาชิกทุกคนเห็น (อัพเดตเมื่อเปิด/รีเฟรชหน้าห้อง ไม่ใช่การนับถอยหลังแบบเรียลไทม์ต่อเนื่อง — ดูรายละเอียดการแสดงผลใน UC-17)<br>3. สมาชิกแต่ละคนกดเริ่มเซสชันโฟกัสของตัวเองได้อย่างอิสระตามเวลาที่ตนสะดวก ไม่จำเป็นต้องเริ่มหรือจบพร้อมกัน ตราบใดที่ยังอยู่ภายในเวลาที่เหลือของห้อง<br>4. เมื่อเวลาห้องนับถอยหลังครบกำหนด ระบบปิดห้องให้อัตโนมัติและสรุปสถิติทันที (ขั้นตอนเดียวกับ "ปิดห้อง" ด้านล่าง)<br><br>**จัดการระยะเวลาร่วมของห้อง:** 1. เจ้าของห้องเปิดหน้าห้อง กดแก้ไขระยะเวลาโฟกัส/พักร่วม<br>2. ตั้งค่าระยะเวลาใหม่แล้วกดยืนยัน<br>3. ระบบอัพเดตค่าระยะเวลาร่วมของห้อง มีผลกับเซสชันที่สมาชิกคนใดก็ตามกดเริ่มใหม่หลังจากนี้เท่านั้น (ไม่กระทบเซสชันที่กำลังทำงานอยู่ก่อนหน้านี้)<br><br>**ปิดห้อง:** 1. เจ้าของห้องกด "ปิดห้อง" ได้ทุกเมื่อ (ไม่ต้องรอเวลาห้องหมด)<br>2. ระบบขอคำยืนยัน<br>3. ยืนยันแล้วระบบสรุปสถิติทันทีโดยเรียงลำดับ (Ranking): เวลาโฟกัส (Focus), เวลาพัก (Rest), และเวลาพักเกินกำหนด (Over-rest) ของแต่ละคน — ผู้ที่สะสมเวลาโฟกัสสำเร็จได้มากที่สุดภายในเวลาห้องคือผู้ชนะการแข่งขัน<br>4. บันทึกประวัติร่วม (UC-23) และแสดงหน้าสรุปผลให้สมาชิกทุกคนในห้องเห็น |
| **ลำดับเหตุการณ์ทางเลือก (Alternative/Exception Flow)** | เจ้าของห้องกด "เริ่มห้อง" ขณะมีสมาชิกในห้องไม่ถึง 2 คน (มีแค่เจ้าของห้องคนเดียว) → ปุ่ม "เริ่มห้อง" ถูกปิดใช้งาน (กดไม่ได้) พร้อมข้อความแจ้งว่าต้องมีสมาชิกอย่างน้อย 2 คนก่อนเริ่มห้องได้; รหัสเชิญไม่ถูกต้อง/ห้องปิดไปแล้ว หรือสมาชิกในห้องเต็ม 8 คนแล้ว → แจ้งเตือน; ห้องเริ่มนับเวลาถอยหลังไปแล้ว (เจ้าของห้องกด "เริ่มห้อง" แล้ว) → ไม่อนุญาตให้เข้าร่วม แจ้งเตือนว่าห้องเริ่มไปแล้ว; สมาชิกที่ไม่ใช่เจ้าของห้องพยายามแก้ไขระยะเวลาร่วมหรือกด "เริ่มห้อง"/"ปิดห้อง" → ไม่แสดงตัวเลือกนี้ให้ (ทำได้เฉพาะเจ้าของห้องเท่านั้น); สมาชิกที่อยู่ในห้องอื่นอยู่แล้วพยายามเข้าร่วมห้องใหม่ → ไม่อนุญาต แจ้งเตือนให้ออกจากห้องเดิมก่อน (UC-18); เวลาห้องนับถอยหลังครบกำหนดพอดีขณะที่สมาชิกบางคนยังมีเซสชันโฟกัสกำลังทำงานอยู่ → ปล่อยให้เซสชันที่กำลังทำงานอยู่ ณ ขณะนั้นโฟกัสจนจบตามปกติก่อน (ไม่ตัดสัญญาณกลางคัน) แล้วห้องจึงปิดและสรุปสถิติจริงหลังเซสชันเหล่านั้นจบครบทุกคน (ไม่อนุญาตให้เริ่มเซสชันใหม่อีกหลังเวลาห้องครบกำหนดแล้ว) |
| **เงื่อนไขหลังทำงานสำเร็จ (Post-condition)** | ข้อมูลการสร้าง การเข้าร่วม การเริ่มห้อง หรือการปิดห้อง ถูกอัพเดต เมื่อปิดห้องสำเร็จ (ไม่ว่าจะกดปิดเองหรือเวลาห้องหมดอัตโนมัติ): Ranking แสดงผล สถิติถูกสรุปลงประวัติ ประวัติห้องเก็บไว้ 30 วัน |
| **ความสัมพันธ์กับ Use Case อื่น** | `<<include>>` UC-23 (เมื่อปิดห้องสำเร็จ); เกี่ยวข้องกับ UC-18 (ออกจากห้อง) และ UC-19 (เตะสมาชิก) ในเชิงผลกระทบต่อจำนวนสมาชิกในห้อง |
| **กฎทางธุรกิจที่เกี่ยวข้อง (Business Rules)** | **โหมดโฟกัสร่วมกันในห้อง (Co-working / Group Focus) — ปรับปรุงใหม่: เปลี่ยนจากระบบตีบอสกลุ่มร่วมกัน (Cooperative Boss HP) เป็นการแข่งขันสะสมเวลาโฟกัสภายในเวลาห้องที่จำกัด (Time-boxed Competition):**<br>- **จำนวนสมาชิก:** ขั้นต่ำ 2 คน สูงสุด 8 คนต่อห้อง — ตรวจสอบขั้นต่ำเฉพาะตอนกด "เริ่มห้อง" เท่านั้น (กดไม่ได้ถ้ามีสมาชิกไม่ถึง 2 คน) หลังห้องเริ่มแล้ว หากสมาชิกทยอยออกจนเหลือต่ำกว่า 2 คน ไม่ทำให้ห้องปิดอัตโนมัติทันที (ดู UC-18 กรณีเจ้าของห้องเป็นสมาชิกคนสุดท้าย ห้องจึงจะปิด)<br>- **เวลาห้อง (Room Duration):** เจ้าของห้องกำหนดอายุของห้องเป็นนาทีตอนสร้างห้อง (ช่วง 5-1000 นาที เช่น 300 นาที) เมื่อกด "เริ่มห้อง" ระบบจะเริ่มนับถอยหลังจากค่านี้ เมื่อครบกำหนดห้องจะปิดอัตโนมัติและสรุปสถิติทันที (ไม่ต้องรอเจ้าของห้องกดปิด) เจ้าของห้องยังปิดห้องเองก่อนครบกำหนดได้เสมอ<br>- **ห้องล็อกสมาชิกเมื่อเริ่มแล้ว:** เข้าร่วมห้องได้เฉพาะช่วงที่ห้องยังไม่กด "เริ่มห้อง" เท่านั้น เมื่อเจ้าของห้องกด "เริ่มห้อง" ไปแล้ว จะไม่มีสมาชิกใหม่เข้าร่วมเพิ่มได้อีกตลอดอายุห้องนั้น (จำนวนสมาชิกคงที่หลังเริ่ม ยกเว้นออกเอง/ถูกเตะซึ่งไม่กระทบการล็อกนี้)<br>- **ไม่มีบอสกลุ่ม/HP ส่วนกลางอีกต่อไป:** ตัดระบบ Dynamic Boss HP และสูตรดาเมจส่วนกลางแบบเดิมออกทั้งหมด แต่ละเซสชันของแต่ละคนยังคงจำลองเป็นการตีบอสส่วนตัวตามกลไกเดิมของ UC-07 (HP บอส = เวลาเซสชัน × 100) เป็นภาพเชิงเกมของตัวเองเท่านั้น ไม่มีการรวม HP หรือดาเมจข้ามคนอีกต่อไป<br>- **กติกาการแข่งขัน:** สมาชิกแต่ละคนแข่งกันสะสม "เวลาโฟกัสสำเร็จ" ของตัวเองให้ได้มากที่สุดภายในกรอบเวลาห้อง (นิยามเดียวกับเวลาโฟกัสสะสมใน UC-17) ผู้ที่สะสมได้มากที่สุดเมื่อห้องปิดคือผู้ชนะ — **Tie-break:** หากเวลาโฟกัสสะสมเท่ากันพอดีตั้งแต่ 2 คนขึ้นไปเมื่อห้องปิด ผู้ที่มีเวลาพัก (Rest) สะสมน้อยกว่าเป็นผู้ชนะ (ไม่นับเวลาพักเกินกำหนด/Over-rest ในการตัดสิน) หากเวลาพักยังเท่ากันอีก ถือว่าชนะร่วมกัน (Tie)<br>- **ระยะเวลาโฟกัส/พักร่วมของห้อง:** ทุกคนในห้องใช้ระยะเวลาโฟกัสและเวลาพักค่าเดียวกันซึ่งเจ้าของห้องเป็นผู้กำหนด/แก้ไขเท่านั้น (สมาชิกคนอื่นตั้งระยะเวลาของตัวเองแยกไม่ได้ขณะเริ่มเซสชันในห้องนี้ ค่าจาก UC-05 ของแต่ละคนถูกแทนที่ด้วยค่าร่วมของห้องชั่วคราว) ช่วงค่าที่อนุญาต: Focus time 1-600 นาที, Break time 1-60 นาที, ถ้าไม่ตั้ง default คือ Focus 25 นาที Break 5 นาที — แต่สมาชิกแต่ละคนกดเริ่มเซสชันโฟกัสของตัวเองได้อย่างอิสระตามเวลาที่ตนสะดวก (ไม่จำเป็นต้องเริ่มหรือจบพร้อมกัน ใช้ระยะเวลาเดียวกันแค่คนละช่วงเวลาเริ่ม) ตราบใดที่ยังอยู่ภายในเวลาที่เหลือของห้อง<br>- **สมาชิก 1 คน อยู่ได้ห้องเดียวในเวลาเดียวกัน:** ต้องกดออกจากห้องปัจจุบัน (UC-18) ก่อนเสมอ จึงจะเข้าร่วมห้องอื่นได้ ระบบไม่อนุญาตให้อยู่ 2 ห้องพร้อมกัน<br>- ในช่วงเวลาที่สมาชิกคนใดอยู่ใน "ช่วงเวลาพักส่วนตัว" หรือ "พักเกินกำหนด" เวลาช่วงนั้นไม่ถูกนับเป็นเวลาโฟกัสสะสมของคนนั้น โดยไม่มีผลกระทบต่อเพื่อนคนอื่นในห้องที่ยังโฟกัสอยู่ |

#### UC-17 · ดูเวลาโฟกัสของสมาชิกในห้อง (Ranking)

| หัวข้อ | รายละเอียด |
|-------|-----------|
| **รหัสอ้างอิง** | FN-ROOM-002 |
| **Actor หลัก** | สมาชิก (Member) |
| **Actor รอง** | - |
| **คำอธิบายโดยย่อ** | สมาชิกในห้องดูสถิติเวลาโฟกัส เวลาพัก และเวลาพักเกินกำหนดของทุกคนในห้อง พร้อมเวลาห้องที่เหลืออยู่ อัพเดตทุกครั้งที่เปิดหรือรีเฟรชหน้าห้อง (ไม่ใช่การอัพเดตสดต่อเนื่องแบบเรียลไทม์) และสามารถดูสรุป Ranking สุดท้ายเมื่อห้องปิด (ไม่ว่าจะปิดเองหรือเวลาห้องหมดอัตโนมัติ — ดู UC-16) |
| **เงื่อนไขก่อนเริ่มทำงาน (Pre-condition)** | สมาชิกอยู่ในห้องโฟกัสที่ยังไม่ปิด |
| **ลำดับเหตุการณ์หลัก (Basic Flow)** | 1. เปิดหน้าห้อง (หรือรีเฟรชหน้าห้องที่เปิดค้างไว้)<br>2. ระบบดึงสถิติสะสม ณ ขณะนั้นของทุกคนในห้อง (เฉพาะเวลาโฟกัสจากเซสชันที่จบสำเร็จแล้วเท่านั้น — ดูนิยามใน Business Rules)<br>3. แสดงตาราง Ranking เรียงตามเวลาโฟกัสสะสม (มากไปน้อย) พร้อมคอลัมน์เวลาพัก และเวลาพักเกินกำหนด (Over-rest) ควบคู่กัน — ผู้ที่อยู่อันดับ 1 ณ ขณะนั้นคือผู้นำการแข่งขัน<br>4. แสดงเวลาห้องที่เหลือ (Room Countdown) ณ ขณะที่เปิด/รีเฟรชหน้านี้ เพื่อให้เห็นว่าเหลือเวลาแข่งขันอีกเท่าไรก่อนห้องจะปิดอัตโนมัติ |
| **ลำดับเหตุการณ์ทางเลือก (Alternative/Exception Flow)** | ยังไม่มีผู้ใดเริ่มเซสชันเลย → แสดงเวลาโฟกัสทุกคนเป็น 0; ห้องยังไม่ถูกเริ่ม (ยังไม่กด "เริ่มห้อง" ตาม UC-16) → แสดงเวลาห้องเต็มจำนวนตามที่ตั้งไว้ ยังไม่นับถอยหลัง |
| **เงื่อนไขหลังทำงานสำเร็จ (Post-condition)** | สมาชิกเห็นสถิติและเวลาห้องที่เหลือ ณ เวลาที่เปิด/รีเฟรชหน้าล่าสุดของทุกคน (ไม่บันทึกแยก—สรุปเมื่อปิดห้องผ่าน UC-16) ต้องรีเฟรชหน้าเองเพื่อดูความคืบหน้าล่าสุดของคนอื่นที่เกิดขึ้นหลังจากนั้น |
| **ความสัมพันธ์กับ Use Case อื่น** | ไม่มี |
| **กฎทางธุรกิจที่เกี่ยวข้อง (Business Rules)** | **นิยาม "เวลาโฟกัสสะสม" ใน Ranking — เพิ่มในรอบตรวจรีวิวที่ 17:** นับเฉพาะนาทีที่โฟกัสจากเซสชันที่จบสำเร็จเท่านั้น (เกณฑ์เดียวกับสูตร EXP ใน UC-07) เซสชันที่ถูกยกเลิกกลางทางไม่นับรวมแม้แต่ส่วนที่โฟกัสไปแล้วก่อนยกเลิก เพื่อให้ตัวเลข Ranking สะท้อนเวลาที่โฟกัสสำเร็จจริงเสมอ ไม่ใช่แค่เวลาที่พยายามโฟกัส — ตัวเลขนี้คือเกณฑ์ตัดสินผู้ชนะการแข่งขันของห้อง (UC-16) โดยตรง |

| **ลำดับเหตุการณ์หลัก (Basic Flow)** | 1. เปิดหน้าจัดการไอเทม<br>2. แสดงรายการไอเทมทั้งหมด พร้อมตารางอัตราดรอปปัจจุบันแยกตามชนิดกล่องสุ่ม 3 ชุด (ระดับต้น/กลาง/สูง) และตารางสูตรผสม<br>3. เพิ่ม/แก้ไข/ลบ/ปิดใช้งานไอเทม หรือปรับอัตราดรอปของกล่องชนิดใดชนิดหนึ่ง หรือปรับสูตรผสม<br>4. ตรวจสอบความถูกต้อง (ผลรวมอัตราดรอปของกล่องแต่ละชนิดต้อง = 100% แยกกันต่อชนิด)<br>5. บันทึกการเปลี่ยนแปลงพร้อมประวัติการแก้ไข |
| **ลำดับเหตุการณ์ทางเลือก (Alternative/Exception Flow)** | ผลรวมอัตราดรอปของกล่องชนิดใดชนิดหนึ่งไม่เท่ากับ 100% → แจ้งข้อผิดพลาด ไม่บันทึก (ตรวจแยกต่อชนิดกล่อง); ปิดใช้งานไอเทมที่ถูกอ้างอิงในสูตรผสม → เตือนพร้อมรายการที่ได้รับผลกระทบก่อนยืนยัน; ลบไอเทมที่สมาชิกบางคนถือครองอยู่แล้ว หรือถูกอ้างอิงในสูตรผสม → ไม่อนุญาตให้ลบถาวร ระบบแนะนำให้ปิดใช้งานแทนเพื่อไม่ให้ข้อมูลอ้างอิงเสีย (ลบถาวรได้เฉพาะไอเทมที่ไม่มีสมาชิกถือครองและไม่ถูกอ้างอิงในสูตรผสมใดเท่านั้น) |
| **เงื่อนไขหลังทำงานสำเร็จ (Post-condition)** | ข้อมูลไอเทม/อัตราดรอป/สูตรผสมเวอร์ชันใหม่มีผลกับการเปิดกล่องสุ่ม/ผสมครั้งถัดไป ไอเทมที่ถูกลบถาวรหายไปจากแคตตาล็อก ไอเทมที่สมาชิกถือครองอยู่แล้ว (ก่อนถูกลบ/ปิดใช้งาน) ไม่ได้รับผลกระทบ |
| **ความสัมพันธ์กับ Use Case อื่น** | ไม่มี |

### E2 · การจัดการ Feedback ผู้ใช้

#### UC-25 · ดูและตอบกลับ Feedback

| หัวข้อ | รายละเอียด |
|-------|-----------|
| **รหัสอ้างอิง** | FN-ADM-002 |
| **Actor หลัก** | ผู้ดูแลระบบ |
| **Actor รอง** | สมาชิกผู้ส่ง Feedback |
| **คำอธิบายโดยย่อ** | ผู้ดูแลระบบตรวจดูรายการ Feedback ที่สมาชิกส่งเข้ามาผ่าน UC-06 และตอบกลับหรือปรับสถานะ เพื่อแจ้งความคืบหน้าให้สมาชิกเจ้าของ Feedback ทราบ |
| **เงื่อนไขก่อนเริ่มทำงาน (Pre-condition)** | ผู้ดูแลระบบเข้าสู่ระบบด้วยบัญชีที่มีสิทธิ์ผู้ดูแล และมี Feedback อย่างน้อย 1 รายการที่สมาชิกส่งมา (UC-06) |
| **ลำดับเหตุการณ์หลัก (Basic Flow)** | 1. เปิดหน้าจัดการ Feedback ระบบแสดงรายการทั้งหมดเรียงตามวันที่ส่งล่าสุด พร้อมสถานะ ("รอตรวจสอบ" / "ตอบกลับแล้ว")<br>2. กรอง/เรียงรายการตามหมวดหมู่หรือสถานะได้<br>3. เลือกดูรายละเอียด Feedback ที่ต้องการ<br>4. พิมพ์คำตอบหรือหมายเหตุ แล้วกดบันทึก<br>5. ระบบเปลี่ยนสถานะเป็น "ตอบกลับแล้ว" และแจ้งเตือนสมาชิกเจ้าของ Feedback |
| **ลำดับเหตุการณ์ทางเลือก (Alternative/Exception Flow)** | ยังไม่มี Feedback ในระบบเลย → แสดงข้อความว่ายังไม่มีรายการ; ไม่กรอกคำตอบก่อนกดบันทึก → แจ้งเตือนให้กรอก |
| **เงื่อนไขหลังทำงานสำเร็จ (Post-condition)** | Feedback ที่ตอบแล้วเปลี่ยนสถานะเป็น "ตอบกลับแล้ว" พร้อมคำตอบที่บันทึกไว้ สมาชิกเจ้าของเห็นคำตอบผ่าน UC-06 |
| **ความสัมพันธ์กับ Use Case อื่น** | เกี่ยวข้องกับ UC-06 (ใช้ข้อมูล Feedback ร่วมกัน เชิงข้อมูลเท่านั้น ไม่ใช่ include/extend เชิง flow) |

## ส่วนเพิ่มเติม (Addendum) · การตั้งค่าการแจ้งเตือน

> UC-26 เพิ่มเข้ามาภายหลังตามคำขอผู้ใช้ เนื้อหาจัดอยู่ในหมวดเดียวกับ M1 · E3 (การจัดการบัญชีและการตั้งค่า) แต่วางเลขต่อท้ายสุดของเอกสารโดยตั้งใจ เพื่อไม่กระทบเลข UC-01 ถึง UC-25 เดิมทั้งหมด (ดู `usecase_module1.puml` สำหรับตำแหน่งใน use case diagram)

#### UC-26 · ตั้งค่าการแจ้งเตือน (เสียง/การสั่น)

| หัวข้อ | รายละเอียด |
|-------|-----------|
| **รหัสอ้างอิง** | FN-ACC-007 |
| **Actor หลัก** | สมาชิก |
| **Actor รอง** | - |
| **คำอธิบายโดยย่อ** | สมาชิกเปิด/ปิดเสียงแจ้งเตือนและการสั่นของแอปแยกจากกันได้อิสระในหน้าตั้งค่า เพื่อควบคุมการแจ้งเตือนทุกประเภทของระบบ (เช่น แจ้งเตือนพักเกินกำหนดใน UC-07, แจ้งเตือนคำขอเป็นเพื่อนใน UC-20) |
| **เงื่อนไขก่อนเริ่มทำงาน (Pre-condition)** | - |
| **ลำดับเหตุการณ์หลัก (Basic Flow)** | 1. เปิดหน้าตั้งค่า เลือกเมนู "การแจ้งเตือน"<br>2. ระบบแสดงสวิตช์ "เสียงแจ้งเตือน" และ "การสั่น" แยกกัน พร้อมสถานะปัจจุบัน<br>3. สลับสวิตช์เปิด/ปิดอย่างใดอย่างหนึ่งหรือทั้งคู่ตามต้องการ<br>4. ระบบบันทึกค่าและมีผลทันทีโดยไม่ต้องกดบันทึกแยก |
| **ลำดับเหตุการณ์ทางเลือก (Alternative/Exception Flow)** | ปิดทั้งเสียงและการสั่น → ระบบยังคงส่ง push notification (ข้อความ/ไอคอน) ตามปกติทุกกรณี เพียงไม่มีเสียงหรือการสั่นประกอบ (ไม่ใช่การปิดการแจ้งเตือนทั้งหมด) |

| **กฎทางธุรกิจที่เกี่ยวข้อง (Business Rules)** | **ค่า Default:** เปิดทั้งเสียงแจ้งเตือนและการสั่น หากสมาชิกไม่เคยตั้งค่าเอง<br>**ความเป็นอิสระของสวิตช์:** เสียงแจ้งเตือนและการสั่นตั้งค่าแยกจากกันได้อิสระ ปิดอย่างใดอย่างหนึ่งหรือทั้งคู่พร้อมกันก็ได้ ไม่ผูกกัน |
