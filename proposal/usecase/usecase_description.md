# FOCUS · แอปพลิเคชันจับเวลาโฟกัสในรูปแบบเกม (Focus) — Use Case Description

> ตามรูปแบบ [`template_usecase_description.md`](./template_usecase_description.md) — 1 use case ในเอกสารนี้ตรงกับ 1 use case ใน `usecase_module1.puml` ถึง `usecase_module7.puml` (ดู `usecase_overview.puml` สำหรับภาพรวม) จัดกลุ่มเป็น Epic ภายในแต่ละโมดูล

**หมายเหตุทั่วไป (Cross-cutting concerns):**

1. **การตรวจสอบสิทธิ์ (Authentication/Authorization):** ทุก use case ที่ Actor หลักเป็น "สมาชิก", "สมาชิกในห้อง" หรือ "ผู้ดูแลระบบ" ต้องผ่านการตรวจสอบสิทธิ์ (JWT/Keycloak ผ่าน M0) มาก่อนเสมอ จึงไม่ระบุซ้ำในแถว "เกี่ยวข้องกับ Service/Endpoint อื่น" ของแต่ละ use case — ยกเว้น UC-01 ที่ Actor หลักเป็น Guest (ยังไม่มีสิทธิ์)
2. **Service/Endpoint:** โครงงานนี้ยังไม่ได้ออกแบบ API endpoint แยกรายตัว (อยู่ระหว่างขั้นตอน proposal) จึงระบุ "-" ในช่อง "เกี่ยวข้องกับ Service/Endpoint อื่น" ของทุก use case ไว้ก่อน รอเติมเมื่อเข้าสู่ขั้นตอนออกแบบระบบ (system design)
3. **การยุบรวม use case:** ตามนิยาม "Use Case ต่างจาก Process/Flow" — UC-06, UC-07, UC-09, UC-11, UC-14, UC-15, UC-16, UC-17 (เดิม) ไม่มี actor ที่ริเริ่มด้วยความสมัครใจ เป็นเพียงกระบวนการ/กฎทางธุรกิจอัตโนมัติที่เกิดสืบเนื่องจาก use case อื่น จึงถูกยุบรวมเป็นแถว **"กฎทางธุรกิจที่เกี่ยวข้อง (Business Rules)"** ภายใน UC-04 และ UC-10 แทนการแยกเป็น use case ของตัวเอง รหัส UC ที่ถูกยุบจะไม่ถูกนำกลับมาใช้ซ้ำ (ปล่อยเป็นเลขว่างในลำดับ) — เพื่อไม่ให้ขัดกับกฎนี้เอง use case "ระบบชุบชีวิต" ที่เคยใช้รหัส **UC-17.1** (พาดพิงเลข 17 ที่ถูกยุบไปแล้ว) จึงเปลี่ยนไปใช้รหัสใหม่เป็น **UC-30** ในเอกสารฉบับนี้
4. **รูปแบบรหัสอ้างอิง (FN-XXX-NNN):** ใช้ตัวย่อโมดูลกำกับ — `ACC` = บัญชี/เริ่มต้นใช้งาน (M1), `SES` = เซสชันโฟกัส (M2), `ITM` = ไอเทม/กล่องสุ่ม (M3), `AVT` = อวตาร/ชุบชีวิต (M4), `ROOM` = ห้องโฟกัส/สังคม (M5), `HIST` = ประวัติ/สถิติ (M6), `ADM` = admin (M7) เลขรันตามลำดับภายในโมดูล ไม่ใช่เลข UC เดิม (เพราะมี UC ที่ถูกยุบ)
5. **Use case ที่เพิ่มใหม่ในรอบรีวิวนี้:** UC-31 (เข้าสู่ระบบ), UC-32 (ออกจากระบบ), UC-34 (ออกจากห้องโฟกัส), UC-35 (ยกเลิก/ลบเพื่อน) เพื่อปิดช่องโหว่ coverage ที่พบระหว่างรีวิว ส่วนเรื่อง Report/Block ผู้ใช้อื่น และ Forgot Password/Account Recovery ยังไม่เพิ่มในรอบนี้ พิจารณาเพิ่มในรอบถัดไปเมื่อใกล้ขั้น production

#### UC-01 · การสมัครสมาชิก

| หัวข้อ | รายละเอียด |
|-------|-----------|
| **รหัสอ้างอิง** | FN-ACC-001 |
| **Actor หลัก** | Guest |
| **Actor รอง** | ระบบฐานข้อมูลผู้ใช้ |
| **คำอธิบายโดยย่อ** | Guest สร้างบัญชีผู้ใช้ใหม่เพื่อเปลี่ยนสถานะเป็นสมาชิก ก่อนจะสามารถเข้าถึงฟีเจอร์อื่นใดของแอปได้ นี่คือ use case เดียวที่ Guest ทำได้ |

| **ลำดับเหตุการณ์หลัก (Basic Flow)** | 1. Guest กดปุ่ม "สมัครสมาชิก"<br>2. ระบบแสดงฟอร์มกรอกข้อมูล<br>3. Guest กรอกข้อมูลครบถ้วนและกดยืนยัน<br>4. ระบบตรวจสอบความถูกต้องของข้อมูล<br>5. ระบบสร้างบัญชีใหม่ เปลี่ยนสถานะเป็นสมาชิก และพาไปหน้าสร้างอวตาร |
| **ลำดับเหตุการณ์ทางเลือก (Alternative/Exception Flow)** | ข้อมูลไม่ครบ/ไม่ถูกรูปแบบ → แจ้งเตือนให้แก้ไข; อีเมล/ชื่อผู้ใช้ถูกใช้แล้ว → แจ้งเตือนให้แก้ไข; กดยกเลิกกลางคัน → ล้างข้อมูล กลับหน้าแรก ยังเป็น Guest |
| **เงื่อนไขหลังทำงานสำเร็จ (Post-condition)** | สร้างบัญชีใหม่และบันทึกลงฐานข้อมูล สถานะเปลี่ยนจาก Guest เป็นสมาชิก และพาไปยัง UC-02 |

## M2 · เซสชันโฟกัส (Core Loop)

### E1 · วงจรเซสชันโฟกัส

#### UC-04 · เริ่มจับเวลาโฟกัส (ตีบอส)

| หัวข้อ | รายละเอียด |
|-------|-----------|
| **รหัสอ้างอิง** | FN-SES-001 |
| **Actor หลัก** | สมาชิก |
| **Actor รอง** | ระบบจับเวลา, ระบบบอส |
| **คำอธิบายโดยย่อ** | สมาชิกเลือกรายการที่จะโฟกัสและตั้งเวลาเริ่มจับเวลา ระบบจะจำลองเป็นการต่อสู้กับบอส ครอบคลุมไปถึงการพัก การยกเลิก และการแจกรางวัล |
| **เงื่อนไขก่อนเริ่มทำงาน (Pre-condition)** | สมาชิกล็อกอิน มีอวตาร และมีรายการสิ่งที่โฟกัส (UC-03) |
| **ลำดับเหตุการณ์หลัก (Basic Flow)** | 1. เลือกรายการจาก UC-03 และตั้งระยะเวลาโฟกัส/พักส่วนตัว<br>2. กดเริ่มจับเวลา ระบบจำลองบอส (HP = เวลา × 100)<br>3. แสดงนาฬิกาถอยหลัง เมื่อเวลาครบ เซสชันสำเร็จ คืน HP อวตาร และบันทึกประวัติ<br>4. ได้รับรางวัล EXP และแจกกล่องสุ่มตามอัตราที่กำหนด (ไปเปิดใน UC-10) หากเซสชัน ≥ 20 นาที<br>5. ระบบเริ่มนับเวลาพักอัตโนมัติตามเวลาที่ตั้งไว้<br>6. เมื่อจบช่วงพักสามารถเริ่มเซสชันใหม่ได้ หากไม่เริ่ม ระบบจะนับเวลาพักเกินกำหนดเป็นสถิติ |
| **ลำดับเหตุการณ์ทางเลือก (Alternative/Exception Flow)** | **ยกเลิกกลางทาง:** กดยกเลิกเซสชันกลางทาง (กดค้าง) → คืน HP ตามสัดส่วนเวลา บันทึกว่ายกเลิก ไม่ได้รับกล่องสุ่ม<br>**พักเกินกำหนด:** หมดเวลาพักแต่ยังไม่เริ่มใหม่ → ระบบนับเวลาพักเกินไปเรื่อยๆ เพื่อเป็นสถิติ<br>**จบช่วงพักก่อนกำหนด:** กดข้ามช่วงพัก → เริ่มเซสชันใหม่ได้ทันที<br>**อวตารหมดสติ:** HP อวตารเท่ากับ 0 ก่อนเริ่ม → เข้าสถานะ "หมดสติ" ล็อกฟีเจอร์จนกว่าจะชุบชีวิต (UC-30) |
| **เงื่อนไขหลังทำงานสำเร็จ (Post-condition)** | เซสชันถูกบันทึกลงประวัติ สถิติเวลาพัก/เบรกเกิน รางวัล และ HP อัพเดตตามกฎทางธุรกิจ |
| **ความสัมพันธ์กับ Use Case อื่น** | ถูก `<<include>>` โดย UC-30 |
| **กฎทางธุรกิจที่เกี่ยวข้อง (Business Rules)** | **โหมดโฟกัสคนเดียว (Single-player Focus):**<br>- การนับเวลาจับเวลาของอวตารนับถอยหลังแบบเรียลไทม์ (Real-time countdown) สมาชิกเห็นเวลาลดลงวินาทีต่อวินาทีบนหน้าจอ<br>- การคำนวณหักล้างค่าพลังชีวิต (HP) ของบอสจะไม่หักทุกวินาที แต่ระบบจะหักลดเลือดบอสทุกๆ 5 นาทีที่ผู้ใช้โฟกัสสำเร็จ (ส่งดาเมจไปหักเลือดบอสเป็นก้อนทุกๆ 5 นาที) |
| **เกี่ยวข้องกับ Service/Endpoint อื่น** | - |



### E3 · การเข้าสู่ระบบและออกจากระบบ

#### UC-31 · เข้าสู่ระบบ

| หัวข้อ | รายละเอียด |
|-------|-----------|
| **รหัสอ้างอิง** | FN-ACC-005 |
| **Actor หลัก** | สมาชิก, ผู้ดูแลระบบ |
| **Actor รอง** | ระบบยืนยันตัวตน (Auth Service) |
| **คำอธิบายโดยย่อ** | สมาชิกที่มีบัญชีอยู่แล้วยืนยันตัวตนเพื่อเข้าใช้งานแอปบนอุปกรณ์ที่ยังไม่ได้ล็อกอิน เป็นเงื่อนไขก่อนเข้าถึง use case อื่นๆ ที่ต้องใช้สิทธิ์สมาชิก |
| **เงื่อนไขก่อนเริ่มทำงาน (Pre-condition)** | สมัครสมาชิกสำเร็จแล้ว (UC-01) และยังไม่ได้ล็อกอินในอุปกรณ์/เซสชันนี้ |
| **ลำดับเหตุการณ์หลัก (Basic Flow)** | 1. เปิดแอปเห็นหน้าล็อกอิน<br>2. กรอกอีเมล/ชื่อผู้ใช้และรหัสผ่าน<br>3. กดยืนยันเข้าสู่ระบบ<br>4. ระบบตรวจสอบข้อมูลกับฐานข้อมูลผู้ใช้<br>5. ออก token ยืนยันตัวตน พาเข้าหน้าหลัก |
| **ลำดับเหตุการณ์ทางเลือก (Alternative/Exception Flow)** | อีเมล/รหัสผ่านไม่ถูกต้อง → แจ้งเตือน ไม่ออก token; บัญชีถูกระงับ (UC-28) → แจ้งเตือนว่าบัญชีถูกระงับ พร้อมเหตุผล ไม่อนุญาตให้เข้าสู่ระบบ |
| **เงื่อนไขหลังทำงานสำเร็จ (Post-condition)** | สมาชิกได้รับสิทธิ์เข้าถึงฟีเจอร์ที่ต้อง login ทั้งหมด (token ยืนยันตัวตนถูกออกและมีผล) และหากเป็นการล็อกอินครั้งแรกของวัน Streak ไฟจะเพิ่มขึ้น 1 ระดับ |
| **ความสัมพันธ์กับ Use Case อื่น** | ไม่มี (เป็นเงื่อนไข pre-condition ของทุก use case ที่ Actor หลักเป็นสมาชิก แต่ไม่ใช่ include/extend เชิง flow) |
| **กฎทางธุรกิจที่เกี่ยวข้อง (Business Rules)** | **อัพเดต Streak ไฟ (Daily Login):** Streak ไฟจะเพิ่มขึ้น 1 ระดับ เมื่อมีการเข้าสู่ระบบ (Login) เป็นครั้งแรกของวันเท่านั้น (ไม่เพิ่มหลังจบเซสชันโฟกัส) |
| **เกี่ยวข้องกับ Service/Endpoint อื่น** | - |
